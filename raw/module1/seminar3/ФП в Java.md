## 1. Что такое функциональное программирование и зачем оно в Java

Функциональное программирование (ФП) — парадигма, в которой вычисление строится как применение и композиция функций, а не как последовательность команд, изменяющих состояние. Java — не функциональный язык в чистом виде (как, например, Haskell), а объектно-ориентированный язык, начиная с 8 версии заимствовавший инструменты ФП: лямбда-выражения, функциональные интерфейсы, `Stream API`, `Optional`.

Смысл заимствования — не смена парадигмы, а расширение словаря. ООП остаётся основным способом моделирования домена (см. семинар 1) и организации кода (SOLID, паттерны GoF — семинар 2); функциональные инструменты решают конкретный класс задач — обработку данных, конфигурируемое поведение, работу с последовательностями — заметно компактнее, чем эквивалентный код в чисто объектном стиле.

Практический пример разницы в стиле — фильтрация списка, знакомая по семинару 2 и 3:

```java
// императивный стиль: как именно перебирать и проверять
List<Instrument> result = new ArrayList<>();
for (Instrument instrument : repository) {
    if (instrument.getPrice().compareTo(minPrice) >= 0) {
        result.add(instrument);
    }
}

// функциональный стиль: что нужно получить, а не как перебирать
Predicate<Instrument> filter = instrument -> instrument.getPrice().compareTo(minPrice) >= 0;
```

Второй вариант не описывает цикл — он описывает *условие*, а применение этого условия к коллекции делегируется вызывающему коду (или, позже, `Stream API`).

## 2. Функции как значения: функциональные интерфейсы

В языках с полноценной поддержкой ФП функции — такие же значения, как числа или строки: их можно сохранить в переменную, передать как аргумент, вернуть из другой функции. У Java нет типа «функция» на уровне языка — вместо этого функция представляется как объект, реализующий интерфейс с одним абстрактным методом.

**Функциональный интерфейс** (SAM — *single abstract method*) — интерфейс ровно с одним нереализованным методом. `default`- и `static`-методы не в счёт, их может быть любое количество:

```java
@FunctionalInterface
public interface InstrumentFilter {
    boolean matches(Instrument instrument);
}
```

`@FunctionalInterface` не меняет поведение интерфейса — это указание компилятору проверить, что абстрактный метод действительно один, и сообщить об ошибке, если кто-то добавит второй. Интерфейс остаётся функциональным и без аннотации — она лишь фиксирует контракт явно.

**Лямбда-выражение** — способ создать реализацию функционального интерфейса без явного класса:

```java
InstrumentFilter cheap = instrument -> instrument.getPrice().compareTo(BigDecimal.TEN) < 0;
```

Синтаксически лямбда — это `(параметры) -> тело`. Тело может быть выражением (значение которого станет результатом метода) или блоком в фигурных скобках с явным `return`. Типы параметров в большинстве случаев выводятся компилятором из сигнатуры функционального интерфейса и не пишутся явно.

## 3. Чистые функции и побочные эффекты

**Чистая функция** — функция, результат которой зависит только от аргументов и которая не производит наблюдаемых побочных эффектов (не меняет внешнее состояние, не пишет в файл, не обращается к сети). Одинаковые аргументы всегда дают одинаковый результат.

```java
// чистая функция: результат зависит только от аргумента
Function<BigDecimal, BigDecimal> withTax = price -> price.multiply(new BigDecimal("1.13"));

// нечистая функция: скрытая зависимость от изменяемого поля counter,
// побочный эффект — изменение counter при каждом вызове
Function<BigDecimal, BigDecimal> withTaxAndLog = price -> {
    counter++; // побочный эффект
    return price.multiply(new BigDecimal("1.13"));
};
```

Java не проверяет и не гарантирует чистоту функций на уровне языка — лямбда может свободно читать и изменять внешнее состояние. Чистота — соглашение, которого разработчик придерживается сознательно, а не свойство, навязанное компилятором.

**Почему это важно на практике.** Чистые функции проще тестировать (не нужно готовить внешнее состояние и проверять его после вызова), проще понимать в отрыве от остального кода и безопаснее использовать в параллельном исполнении — им не нужна синхронизация, потому что нет разделяемого изменяемого состояния. Стремление писать лямбды как чистые функции, где это возможно, — практическая рекомендация, а не требование языка.

## 4. Неизменяемость (immutability)

Тесно связанное со стилем ФП свойство — предпочтение неизменяемых (immutable) объектов изменяемым. Неизменяемый объект нельзя модифицировать после создания — любое «изменение» создаёт новый объект.

В проекте Market Pulse неизменяемость уже встречалась на семинаре 1: поля `Instrument` — `private final`, класс не предоставляет сеттеров, единственный способ задать состояние — конструктор. `BigDecimal`, использованный для денежных величин, сам по себе неизменяем: любая операция (`add`, `multiply`) возвращает новый объект, не модифицируя исходный.

```java
BigDecimal price = new BigDecimal("100");
BigDecimal withTax = price.multiply(new BigDecimal("1.13"));
// price по-прежнему равен 100 — multiply не изменил исходный объект, а создал новый
```

**Почему это сочетается с ФП.** Захват переменных в лямбде работает только с *effectively final* значениями (см. раздел 2 сопроводительного материала «Лямбда-выражения: устройство») — если бы захваченные объекты было принято свободно изменять, композиция функций стала бы источником трудноуловимых ошибок: одна и та же лямбда могла бы возвращать разные результаты в зависимости от того, что произошло с захваченным объектом между созданием лямбды и её вызовом. Неизменяемость устраняет этот источник ошибок целиком.

## 5. Обзор `java.util.function`

Прежде чем объявлять собственный функциональный интерфейс, стоит проверить, нет ли уже подходящего в пакете `java.util.function`. Он покрывает подавляющее большинство практических случаев:

| **Интерфейс**         | **Абстрактный метод** | **Смысл**                                   | **Пример**                                         |
| --------------------- | --------------------- | ------------------------------------------- | -------------------------------------------------- |
| `Predicate<T>`        | `boolean test(T t)`   | Проверка условия                            | `instrument -> instrument.getPrice().signum() > 0` |
| `Function<T, R>`      | `R apply(T t)`        | Преобразование `T` в `R`                    | `Instrument::getTicker`                            |
| `Supplier<T>`         | `T get()`             | Поставщик значения без входных данных       | `() -> new ArrayList<>()`                          |
| `Consumer<T>`         | `void accept(T t)`    | Действие над значением, без результата      | `instrument -> System.out.println(instrument)`     |
| `BiFunction<T, U, R>` | `R apply(T t, U u)`   | Преобразование двух аргументов в результат  | `(a, b) -> a.getPrice().add(b.getPrice())`         |
| `UnaryOperator<T>`    | `T apply(T t)`        | `Function<T, T>` — вход и выход одного типа | `price -> price.multiply(BigDecimal.TEN)`          |

Использование стандартных интерфейсов вместо собственных — не только меньше кода, но и совместимость: методы `Stream API` (семинар 6) принимают именно `Predicate`, `Function`, `Consumer` — свой интерфейс типа `InstrumentFilter` пришлось бы адаптировать вручную.

## 6. Функции высшего порядка и композиция

**Функция высшего порядка** — функция, которая принимает другую функцию как аргумент или возвращает функцию как результат. В Java эта роль реализуется через параметры и возвращаемые значения типа функционального интерфейса:

```java
// принимает функцию как аргумент
public static List<Instrument> filter(List<Instrument> items, Predicate<Instrument> condition) {
    List<Instrument> result = new ArrayList<>();
    for (Instrument item : items) {
        if (condition.test(item)) {
            result.add(item);
        }
    }
    return result;
}

// возвращает функцию как результат
public static Predicate<Instrument> minPriceFilter(BigDecimal minPrice) {
    return instrument -> instrument.getPrice().compareTo(minPrice) >= 0;
}
```

**Композиция** — построение новой функции из существующих без изменения исходных. У ключевых интерфейсов `java.util.function` для этого есть готовые `default`-методы:

```java
Predicate<Instrument> isStock = instrument -> instrument instanceof Stock;
Predicate<Instrument> isExpensive = instrument -> instrument.getPrice().compareTo(BigDecimal.valueOf(1000)) > 0;

Predicate<Instrument> expensiveStock = isStock.and(isExpensive);
Predicate<Instrument> notStock = isStock.negate();

Function<Instrument, String> ticker = Instrument::getTicker;
Function<String, String> upperCase = String::toUpperCase;
Function<Instrument, String> upperTicker = ticker.andThen(upperCase);
```

`andThen` на `Function` выполняет исходную функцию, а затем передаёт её результат в следующую — композиция в математическом смысле: `upperTicker(x) = upperCase(ticker(x))`.

## 7. Метод-ссылки

Метод-ссылка (`::`) — синтаксический сахар для лямбды, тело которой сводится к вызову одного уже существующего метода:

```java
// лямбда
Function<Instrument, String> ticker1 = instrument -> instrument.getTicker();
// метод-ссылка на метод экземпляра
Function<Instrument, String> ticker2 = Instrument::getTicker;

// лямбда
Supplier<ArrayList<Instrument>> supplier1 = () -> new ArrayList<>();
// метод-ссылка на конструктор
Supplier<ArrayList<Instrument>> supplier2 = ArrayList::new;

// лямбда
Consumer<String> printer1 = s -> System.out.println(s);
// метод-ссылка на метод конкретного объекта
Consumer<String> printer2 = System.out::println;

// лямбда
BiFunction<String, String, Boolean> eq1 = (a, b) -> a.equalsIgnoreCase(b);
// метод-ссылка на метод произвольного объекта первого аргумента
BiFunction<String, String, Boolean> eq2 = String::equalsIgnoreCase;
```

Правило простое: если единственное, что делает лямбда, — это вызывает существующий метод с теми же аргументами, в том же порядке, — метод-ссылка предпочтительнее: короче и явно указывает на переиспользуемый метод, а не на новую логику.

## 8. Optional как функциональный подход к отсутствию значения

`null` — не типизированное отсутствие значения: любая ссылочная переменная в Java может быть `null`, и компилятор не различает «ссылка, которая точно указывает на объект» и «ссылка, которая может быть пустой». `Optional<T>` делает это различие явным на уровне типов: метод, возвращающий `Optional<Instrument>`, сигнализирует вызывающему коду о возможном отсутствии результата прямо в сигнатуре — в отличие от метода, возвращающего `Instrument`, где на возможность `null` можно узнать только из документации или исходного кода.

```java
public Optional<Instrument> findByTicker(String ticker) {
    for (Instrument instrument : instruments) {
        if (instrument.getTicker().equalsIgnoreCase(ticker)) {
            return Optional.of(instrument);
        }
    }
    return Optional.empty();
}
```

`Optional` вписывается в функциональный стиль естественно — у него есть `map`, `filter`, `ifPresent`, принимающие функциональные интерфейсы:

```java
repository.findByTicker(ticker)
    .map(Instrument::getDescription)
    .ifPresentOrElse(
        System.out::println,
        () -> System.out.println("Инструмент не найден: " + ticker)
    );
```

Подробный разбор API `Optional` — на самом семинаре (этап 6 базового и продвинутого трека).

## 9. Куда это ведёт: Stream API и Vavr

Сегодняшние инструменты — лямбды, функциональные интерфейсы, `Optional` — фундамент для `Stream API`, который появится на семинаре 6 и заменит ручные циклы обработки коллекций декларативными цепочками вызовов:

```java
// то, что пишем сегодня
List<Instrument> result = new ArrayList<>();
for (Instrument instrument : repository) {
    if (filter.test(instrument)) {
        result.add(instrument);
    }
}
result.sort(Comparator.comparing(Instrument::getPrice));

// то, во что это свернётся на семинаре 6
List<Instrument> result = repository.stream()
    .filter(filter)
    .sorted(Comparator.comparing(Instrument::getPrice))
    .toList();
```

Без сегодняшнего материала — лямбд, `Predicate`, `Comparator`, метод-ссылок — код справа был бы набором незнакомых конструкций; с сегодняшним материалом это тот же самый инструментарий, применённый к новому API.

На продвинутом треке также затрагивается [Vavr](https://vavr.io/) — библиотека, расширяющая функциональный инструментарий Java типами вроде `Try` и `Either`, которых нет в стандартной библиотеке. Подробнее — в материалах продвинутого трека семинара 3.
