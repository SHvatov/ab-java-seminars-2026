## 1. Почему это не «просто анонимный класс»

Частое упрощение — «лямбда — это анонимный класс, только короче записывается». На уровне синтаксиса разница действительно сводится к этому:

```java
// анонимный класс
Predicate<Instrument> a = new Predicate<Instrument>() {
    @Override
    public boolean test(Instrument instrument) {
        return instrument.getPrice().signum() > 0;
    }
};

// лямбда
Predicate<Instrument> b = instrument -> instrument.getPrice().signum() > 0;
```

Но на уровне байт-кода и рантайма это два принципиально разных механизма: анонимный класс — сущность, полностью определённая на этапе компиляции, лямбда — сущность, порождаемая JVM во время выполнения по описанию, оставленному компилятором. Понимание разницы объясняет два практических следствия: почему лямбды дешевле анонимных классов при частом создании и почему effectively final — не произвольное синтаксическое ограничение.

## 2. Как компилируется анонимный класс

Для каждого анонимного класса компилятор генерирует отдельный именованный `.class`-файл — `ИмяВнешнегоКласса$N.class`, где `N` — порядковый номер анонимного класса в пределах внешнего класса. Реализация `Iterator` из `InstrumentRepository` (семинар 2) превращается в файл `InstrumentRepository$1.class`, физически существующий на диске после компиляции, наравне с `InstrumentRepository.class`.

Если анонимный класс захватывает локальные переменные, компилятор добавляет в сгенерированный класс синтетические (сгенерированные, не написанные разработчиком) `private final`-поля — по одному на каждую захваченную переменную — и заполняет их через конструктор в момент создания экземпляра:

```java
// то, что пишет разработчик
String type = "STOCK";
InstrumentFilter filter = new InstrumentFilter() {
    @Override
    public boolean matches(Instrument instrument) {
        return instrument.getClass().getSimpleName().equalsIgnoreCase(type);
    }
};

// приблизительный эквивалент того, что генерирует компилятор
final class Outer$1 implements InstrumentFilter {
    private final String val$type; // синтетическое поле под захваченную переменную

    Outer$1(String type) {
        this.val$type = type;
    }

    @Override
    public boolean matches(Instrument instrument) {
        return instrument.getClass().getSimpleName().equalsIgnoreCase(val$type);
    }
}
```

Каждое создание анонимного класса — это `new Outer$1(type)`: выделение памяти под объект с копией захваченного значения в отдельном поле. Требование effectively final для захватываемых переменных существует и здесь — по той же причине, что и для лямбд (раздел 5), просто менее очевидной, поскольку копирование скрыто в конструкторе, а не в параметрах.

## 3. Как компилируется лямбда: тело метода и invokedynamic

Компилятор не создаёт отдельный `.class`-файл под тело лямбды. Вместо этого тело лямбды становится приватным (обычно статическим, если лямбда ничего не захватывает — либо экземплярным, если захватывает `this`) методом того же класса, в котором лямбда написана, с сгенерированным именем вида `lambda$имяМетода$N`:

```java
// то, что пишет разработчик, внутри ListCommand.call()
Predicate<Instrument> filter = instrument -> instrument.getClass().getSimpleName().equalsIgnoreCase(type);

// синтетический метод, который увидит javap -p -c ListCommand.class
private static boolean lambda$call$0(String type, Instrument instrument) {
    return instrument.getClass().getSimpleName().equalsIgnoreCase(type);
}
```

Захваченная переменная `type` стала обычным параметром метода — никакого отдельного объекта-обёртки для неё не потребовалось.

В месте, где лямбда *создаётся* (не вызывается — именно создаётся, то есть присваивается переменной типа `Predicate`), байт-код содержит инструкцию `invokedynamic`:

```
0: invokedynamic #7,  0    // InvokeDynamic #0:test:(Ljava/lang/String;)Ljava/util/function/Predicate;
5: astore_2
```

`invokedynamic` — инструкция JVM, введённая в Java 7 изначально для динамических языков на JVM (Groovy, JRuby), а с Java 8 используемая компилятором и для лямбд. В отличие от `invokestatic`/`invokevirtual`, она не связана с конкретным методом напрямую — вместо этого при **первом** выполнении вызывает так называемый *bootstrap-метод*, который сам определяет, что именно нужно вызвать, и «запоминает» это решение (linkage) для всех последующих выполнений той же инструкции.

## 4. LambdaMetafactory и скрытые классы

Bootstrap-метод, на который ссылается `invokedynamic` для лямбд, — `java.lang.invoke.LambdaMetafactory.metafactory` (или его вариант `altMetafactory` для более сложных случаев — сериализуемых лямбд, лямбд с мостовыми методами). При первом достижении инструкции `invokedynamic` происходит следующее:

1. JVM вызывает `LambdaMetafactory.metafactory` с информацией о целевом функциональном интерфейсе (`Predicate`), сигнатуре его абстрактного метода (`test`) и ссылкой (`MethodHandle`) на синтетический метод `lambda$call$0`, реализующий тело лямбды.
2. `metafactory` генерирует новый класс **во время выполнения** — реализацию `Predicate`, у которой метод `test` делегирует вызов в `lambda$call$0`. Этот класс не существует как файл на диске ни до, ни после запуска — он существует только в памяти запущенной JVM.
3. Такой класс называется *скрытым классом* (hidden class, `java.lang.invoke.MethodHandles.Lookup.defineHiddenClass` — тот же механизм, на котором реализован `LambdaMetafactory`, начиная с Java 15, где API скрытых классов стал публичным; до этого использовался аналогичный внутренний механизм). Скрытый класс не появляется в результатах `Class.forName`, не виден стандартному class loader'у как обычный класс, и его невозможно найти в файловой системе — он присутствует только в памяти процесса JVM.
4. JVM связывает (linkage) конкретную инструкцию `invokedynamic` с фабрикой, создающей экземпляры этого скрытого класса, и вызывает её, чтобы получить объект `Predicate` для данного конкретного вызова — с переданным `type` в качестве захваченного аргумента конструктора скрытого класса.
5. При всех последующих выполнениях той же инструкции `invokedynamic` (например, при повторных вызовах `ListCommand.call()`) шаги 1–3 не повторяются — связывание закешировано, выполняется сразу шаг 4.

Итог: `javap -c` покажет `invokedynamic` и синтетический метод `lambda$call$0`, но не покажет класс-реализацию `Predicate` — потому что на этапе компиляции такого класса ещё не существует, компилятор оставил лишь «инструкцию», как его сгенерировать при необходимости.

> **Практическое следствие:** количество лямбд, порождаемых одним и тем же лямбда-выражением в исходном коде (например, внутри цикла), не приводит к росту числа классов — каждое лямбда-*выражение* соответствует одному bootstrap-вызову за время жизни JVM, вне зависимости от того, сколько раз оно фактически выполняется.

## 5. Почему захват работает только с effectively final

`lambda$call$0` из раздела 3 получает захваченную переменную `type` как обычный параметр метода. Параметры метода в JVM передаются по значению — на момент вызова `invokedynamic` формируется набор аргументов для будущего вызова скрытого класса, и этот набор фиксируется в момент создания лямбды.

Если бы исходная переменная `type` могла быть переприсвоена после создания лямбды, но до её вызова, возник бы неразрешимый вопрос: какое значение должен получить `lambda$call$0` — то, что было на момент создания лямбды, или то, что стало на момент её вызова? Простое копирование значения в параметр решает эту неоднозначность, но только если значение не менялось — иначе копия и переменная разошлись бы, и поведение лямбды стало бы зависеть от момента её вызова непредсказуемым для разработчика образом.

Java выбрала простое и предсказуемое правило: переменная, захватываемая лямбдой (или анонимным классом — по той же причине, раздел 2), обязана быть *effectively final* — не переприсваиваться после инициализации. Компилятор проверяет это статически и отказывается компилировать код, где условие нарушено, вместо того чтобы допустить состояние гонки между «значением на момент захвата» и «значением на момент вызова».

Важная деталь: effectively final требование касается самой переменной (ссылки), а не объекта, на который она указывает. Изменять состояние захваченного изменяемого объекта — можно:

```java
List<String> collected = new ArrayList<>(); // ссылка collected не меняется
Consumer<Instrument> collector = instrument -> collected.add(instrument.getTicker()); // объект, на который ссылается collected, меняется — это разрешено
```

## 6. Стоимость: первый вызов vs последующие

Первое выполнение `invokedynamic` для конкретного лямбда-выражения дороже обычного вызова метода — оно включает вызов `LambdaMetafactory.metafactory`, генерацию скрытого класса и связывание `MethodHandle`. Это разовые издержки, происходящие один раз за время жизни JVM для каждого лямбда-выражения в исходном коде (не за каждое *выполнение* этого выражения).

После первого связывания повторные выполнения той же инструкции `invokedynamic` по стоимости сопоставимы с обычным вызовом метода — JVM больше не обращается к `LambdaMetafactory`, а использует закешированный `MethodHandle`. Для кода, который выполняется многократно (циклы, обработчики, часто вызываемые команды CLI), эта разовая стоимость амортизируется и становится незначительной. Для кода, который выполняется считаное число раз за время жизни программы, разница между лямбдой и анонимным классом на практике не имеет значения — в таких случаях выбор между ними стоит делать исходя из читаемости, а не производительности.

## 7. Лямбды и сериализация

Лямбды, в отличие от анонимных классов, по умолчанию **не** реализуют `Serializable`, даже если целевой функциональный интерфейс сам его не требует. Присвоить лямбду переменной типа `Serializable & Predicate<Instrument>` без явного указания в самом функциональном интерфейсе, что он `extends Serializable`, не получится без приведения типов, а десериализация лямбд между разными версиями JVM или даже разными компиляциями одного и того же исходного кода не гарантирована — сериализованная форма лямбды жёстко привязана к деталям её генерации конкретным компилятором.

> **Практическая рекомендация:** не сериализовать лямбды напрямую (как поля `Serializable`-классов, в кэшах между перезапусками приложения и подобное). Если объект, реализующий функциональный интерфейс, должен переживать сериализацию, для этого по-прежнему подходит именованный класс — та самая ситуация, где анонимный класс или отдельный класс остаётся уместнее лямбды.
