### Этап 1. Теория: принципы SOLID (~15–20 мин)

Изложение принципов SOLID в формате мини-лекции, отталкиваясь от нарушения LSP, уже присутствующего в доменной модели после первого семинара (`getDividends(BigDecimal)` в `Instrument`). Порядок изложения: LSP → ISP (как проектная причина нарушения LSP) → SRP → OCP (со ссылкой на переход от `instanceof`-цепочки к полиморфизму на семинаре 1) → DIP — верхнеуровнево, с явной пометкой, что принцип будет разобран подробно при знакомстве со Spring. Отдельным блоком — KISS и DRY как менее формальные, но практические ориентиры.

По итогам этапа фиксируется исправление нарушения LSP: `getDividends` выносится из `Instrument` в интерфейс `DividendPaying`, реализуемый только `Stock`.

### Этап 2. Подключение picocli, знакомство с `pom.xml` (~10 мин)

Проект переходит от точки входа `demo.Main` к полноценному CLI. В качестве библиотеки для разбора аргументов командной строки используется [picocli](https://picocli.info/) — позволяет декларативно описывать команды и опции через аннотации, не реализуя разбор аргументов вручную.

Перед добавлением зависимости — краткое объяснение назначения `pom.xml`: манифест проекта для Maven, описывающий координаты проекта, зависимости и их версии, плагины сборки. Указание версии зависимости в одном месте гарантирует, что все участники проекта используют одну и ту же версию библиотеки.

```xml
<dependency>
    <groupId>info.picocli</groupId>
    <artifactId>picocli</artifactId>
    <version>4.7.7</version>
</dependency>
```

После подключения зависимости — краткий обзор основных классов picocli, без разбора отдельного демонстрационного примера:

```java
/**
 * Основные типы picocli, используемые в проекте:
 *
 * @Command       — аннотация класса-команды (имя, описание, вложенные subcommands)
 * @Option         — аннотация именованного параметра (например, --type)
 * @Parameters     — аннотация позиционного параметра (например, тикер)
 * Callable<Integer> — интерфейс, реализуемый командой; call() возвращает код завершения
 * CommandLine     — точка запуска: разбирает args и вызывает call() нужной команды
 */
```

### Этап 3. Обсуждение: какие паттерны GoF применимы к CLI (~10 мин)

Формат — открытое обсуждение с группой, а не лекция. Вопросы для обсуждения:

- Какие паттерны проектирования уже знакомы студентам (из прошлого опыта, других курсов, самостоятельного изучения)?
- Как устроена обработка команды в CLI-приложении: что должно происходить, когда пользователь вводит `search AAPL`?
- Как устроено создание объектов `Stock`, `Bond`, `Etf`, когда параметры инструмента приходят не из кода, а из строки, введённой пользователем?
- Как отфильтровать список инструментов по типу так, чтобы добавление нового типа инструмента не требовало правки кода фильтрации?
- Как перебрать все инструменты в хранилище, не раскрывая наружу, как именно они там хранятся?

Ожидаемое направление ответов и разбор кандидатов:

| **Задача**                                                        | **Паттерн-кандидат** |
| ----------------------------------------------------------------- | -------------------- |
| Обработка пользовательской команды                                | Command              |
| Создание объекта инструмента по строковому типу                   | Factory Method       |
| Фильтрация списка инструментов без хрупкого сравнения по типу     | Strategy             |
| Обход инструментов в хранилище без раскрытия внутренней структуры | Iterator             |

Обсуждается, что picocli уже частично реализует Command на уровне библиотеки — каждая аннотированная классом `@Command` команда, реализующая `Callable`, фактически является объектом-командой с методом `call()` в роли `execute()`. Это наблюдение используется как переход к совместной реализации: паттерн не придумывается с нуля, а узнаётся в уже используемом инструменте.

Iterator фиксируется в списке кандидатов, но не реализуется совместно — он отдаётся студентам на этапе 5.

### Этап 4. Совместная реализация: Factory Method, Strategy (~25–30 мин)

**Factory Method: создание инструмента по типу.** Пользователь вводит тип инструмента строкой (`STOCK`, `BOND`, `ETF`); код, создающий объект, не должен разрастаться в `if/else` при добавлении нового типа:

```java
public interface InstrumentFactory {
    Instrument create(String ticker, String name, BigDecimal price);
}

public final class InstrumentFactories {
    private static final Map<String, InstrumentFactory> REGISTRY = new HashMap<>();

    private InstrumentFactories() {
    }

    public static void register(String type, InstrumentFactory factory) {
        REGISTRY.put(type.toUpperCase(), factory);
    }

    public static Instrument create(String type, String ticker, String name, BigDecimal price) {
        InstrumentFactory factory = REGISTRY.get(type.toUpperCase());
        if (factory == null) {
            throw new IllegalArgumentException("Unknown instrument type: " + type);
        }
        return factory.create(ticker, name, price);
    }
}

public class StockFactory implements InstrumentFactory {

    static {
        InstrumentFactories.register("STOCK", new StockFactory());
    }

    @Override
    public Instrument create(String ticker, String name, BigDecimal price) {
        return new Stock(ticker, name, price);
    }
}
```

Подход аналогичен регистрации драйверов в JDBC (`DriverManager`): реестр не содержит статического перечисления всех фабрик, а каждая фабрика регистрирует себя сама при загрузке своего класса. Добавление нового типа инструмента сводится к реализации нового `InstrumentFactory` со статическим блоком регистрации — ни `InstrumentFactories`, ни существующие фабрики не изменяются, что напрямую иллюстрирует OCP, рассмотренный на этапе 1.

Отдельно стоит обсудить со студентами оговорку, знакомую по JDBC: статический блок выполняется только при загрузке класса JVM, поэтому `StockFactory` должен быть каким-то образом затронут до первого вызова `InstrumentFactories.create(...)` — например, явным обращением к классу (по аналогии с историческим `Class.forName("com.mysql.cj.jdbc.Driver")`) либо через `ServiceLoader`. Для целей семинара достаточно зафиксировать этот нюанс на словах, не усложняя пример механизмом `ServiceLoader`.

**Strategy: фильтрация инструментов по типу.** Наивный способ реализовать фильтр `--type` в будущей команде `list` — сравнение строки с именем класса напрямую в коде команды:

```java
if (type == null || instrument.getClass().getSimpleName().equalsIgnoreCase(type)) {
    // вывести инструмент
}
```

Такое условие само по себе не нарушает OCP только пока правило фильтрации одно; как только потребуется, например, фильтр по диапазону цены или по сочетанию признаков, код команды придётся дополнять новыми ветками `if`, вперемешку с уже существующей логикой. Strategy инкапсулирует условие отбора в отдельный, взаимозаменяемый объект — команда работает с абстракцией, не зная, по какому именно правилу происходит отбор:

```java
public interface InstrumentFilter {
    boolean matches(Instrument instrument);
}

public class TypeFilter implements InstrumentFilter {
    private final String type;

    public TypeFilter(String type) {
        this.type = type;
    }

    @Override
    public boolean matches(Instrument instrument) {
        return instrument.getClass().getSimpleName().equalsIgnoreCase(type);
    }
}

public class NoOpFilter implements InstrumentFilter {
    @Override
    public boolean matches(Instrument instrument) {
        return true;
    }
}
```

Использование в коде, работающем со списком инструментов:

```java
InstrumentFilter filter = (type == null) ? new NoOpFilter() : new TypeFilter(type);

for (Instrument instrument : repository) {
    if (filter.matches(instrument)) {
        System.out.println(instrument.getDescription());
    }
}
```

Добавление нового способа отбора (например, `PriceAboveFilter`) сводится к реализации нового `InstrumentFilter` — код, перебирающий инструменты, не изменяется. Применение паттерна здесь напрямую опирается на этап 1: инкапсуляция алгоритма отбора в подменяемый объект — это то же соблюдение OCP, что и в Factory Method, но применительно не к созданию объекта, а к варьируемому поведению.

### Этап 5. Самостоятельная работа: Iterator над `InstrumentRepository` (~15 мин)

Задание: `InstrumentRepository` хранит инструменты во внутреннем массиве. Требуется реализовать возможность перебора всех инструментов репозитория, не раскрывая наружу сам массив (то есть без метода вида `getAll(): Instrument[]`). Если метод добавления записей (`add(Instrument instrument)`) в репозитории ещё отсутствует, его необходимо добавить — без него нечем будет наполнить репозиторий для проверки перебора.

Условия:

- Реализовать собственный тип, предоставляющий методы `hasNext()` и `next()` (пока без реализации стандартного интерфейса `java.util.Iterator` — это будет показано на этапе 6).
- Проверить работу реализации классическим циклом:

```java
InstrumentIterator iterator = repository.iterator();
while (iterator.hasNext()) {
    Instrument instrument = iterator.next();
    System.out.println(instrument.getDescription());
}
```

- `for-each` на этом этапе не использовать — синтаксис ещё не разобран и будет объяснён на этапе 6.

### Этап 6. Разбор решений, эталонная реализация Iterator (~10–15 мин)

Разбор одного-двух студенческих решений: как авторы оформили собственный интерфейс `InstrumentIterator` (отдельный интерфейс, вложенный класс, анонимный класс), как обработана ситуация вызова `next()` после исчерпания элементов.

Эталонная реализация — переход от собственного интерфейса к `java.util.Iterator<Instrument>` и `Iterable<Instrument>`:

```java
public class InstrumentRepository implements Iterable<Instrument> {
    private Instrument[] instruments = new Instrument[100];
    private int size = 0;

    public void add(Instrument instrument) {
        instruments[size++] = instrument;
    }

    @Override
    public Iterator<Instrument> iterator() {
        return new Iterator<>() {
            private int cursor = 0;

            @Override
            public boolean hasNext() {
                return cursor < size;
            }

            @Override
            public Instrument next() {
                if (!hasNext()) {
                    throw new NoSuchElementException();
                }
                return instruments[cursor++];
            }
        };
    }
}
```

Далее — явная демонстрация, что `for-each` является синтаксическим сахаром над `Iterator`, на паре эквивалентных фрагментов:

```java
// то, что пишет программист
for (Instrument instrument : repository) {
    System.out.println(instrument.getDescription());
}

// во что это разворачивается компилятором
Iterator<Instrument> iterator = repository.iterator();
while (iterator.hasNext()) {
    Instrument instrument = iterator.next();
    System.out.println(instrument.getDescription());
}
```

Реализация `Iterable<Instrument>` — условие, необходимое и достаточное для использования `for-each`; без него компилятор не примет объект в конструкции `for (... : repository)`.

### Этап 7. Совместная реализация: Command поверх picocli, команда `list --type` (~15–20 мин)

Первые доменные команды приложения — `SearchCommand` и `AddCommand`, каждая инкапсулирует один сценарий использования и делегирует получение данных доменному слою, не смешивая парсинг аргументов с бизнес-логикой (связь с SRP, этап 1):

```java
@Command(name = "search", description = "Поиск инструмента по тикеру")
public class SearchCommand implements Callable<Integer> {

    @Parameters(index = "0", description = "Тикер инструмента")
    private String ticker;

    private final InstrumentRepository repository;

    public SearchCommand(InstrumentRepository repository) {
        this.repository = repository;
    }

    @Override
    public Integer call() {
        Instrument instrument = repository.findByTicker(ticker);
        if (instrument == null) {
            System.out.println("Инструмент не найден: " + ticker);
            return 1;
        }
        System.out.println(instrument.getDescription());
        return 0;
    }
}
```

Команда `add`, использующая `InstrumentFactories` из этапа 4 — связывает Factory Method и Command: создание объекта остаётся ответственностью фабрики, команда лишь разбирает аргументы и передаёт их дальше:

```java
@Command(name = "add", description = "Добавление инструмента в репозиторий")
public class AddCommand implements Callable<Integer> {

    @Parameters(index = "0", description = "Тип инструмента (STOCK, BOND, ETF)")
    private String type;

    @Parameters(index = "1", description = "Тикер инструмента")
    private String ticker;

    @Parameters(index = "2", description = "Название инструмента")
    private String name;

    @Parameters(index = "3", description = "Цена инструмента")
    private BigDecimal price;

    private final InstrumentRepository repository;

    public AddCommand(InstrumentRepository repository) {
        this.repository = repository;
    }

    @Override
    public Integer call() {
        Instrument instrument = InstrumentFactories.create(type, ticker, name, price);
        repository.add(instrument);
        System.out.println("Добавлено: " + instrument.getDescription());
        return 0;
    }
}
```

Команда `list`, использующая итератор из этапа 6 (уже в форме `for-each`, поскольку сахар разобран) и `InstrumentFilter` из этапа 4 вместо прямого сравнения строк — добавление нового способа фильтрации не потребует изменения кода команды:

```java
@Command(name = "list", description = "Список инструментов")
public class ListCommand implements Callable<Integer> {

    @Option(names = "--type", description = "Фильтр по типу инструмента")
    private String type;

    private final InstrumentRepository repository;

    public ListCommand(InstrumentRepository repository) {
        this.repository = repository;
    }

    @Override
    public Integer call() {
        InstrumentFilter filter = (type == null) ? new NoOpFilter() : new TypeFilter(type);

        for (Instrument instrument : repository) {
            if (filter.matches(instrument)) {
                System.out.println(instrument.getDescription());
            }
        }
        return 0;
    }
}
```

Точка входа приложения регистрирует команды в корневом объекте picocli:

```java
@Command(name = "market-pulse", subcommands = {SearchCommand.class, AddCommand.class, ListCommand.class})
public class MarketPulseCli implements Runnable {
    @Override
    public void run() {
        System.out.println("Используйте --help для списка команд");
    }
}

public class Main {
    public static void main(String[] args) {
        int exitCode = new CommandLine(new MarketPulseCli()).execute(args);
        System.exit(exitCode);
    }
}
```

`InstrumentRepository` на данном этапе — заглушка с захардкоженным набором инструментов; подключение реального источника данных запланировано на семинар 5.

Стоит отдельно отметить со студентами, что `TypeFilter` внутри по-прежнему сравнивает `getClass().getSimpleName()` — Strategy убрал разрастание условий в `ListCommand`, но не саму хрупкость сравнения по имени класса. Более надёжная реализация (например, сверка через `InstrumentFactories` или явный признак типа на `Instrument`) может быть предложена студентам как самостоятельное улучшение за рамками занятия.

## 1. От ООП к принципам проектирования

Механизмы ООП — наследование, интерфейсы, композиция — допускают как удачное, так и неудачное применение; сам по себе язык не проверяет качество дизайна. Принципы SOLID формулируют пять правил, следование которым делает код устойчивым к изменению и расширению:

- **S** — Single Responsibility Principle;
- **O** — Open/Closed Principle;
- **L** — Liskov Substitution Principle;
- **I** — Interface Segregation Principle;
- **D** — Dependency Inversion Principle.

Изложение начинается с LSP, поскольку его нарушение уже присутствует в доменной модели после первого семинара.

---

## 2. Принцип подстановки Лисков (LSP)

Объекты подтипа должны быть заменяемы объектами базового типа без нарушения корректности программы. Метод базового класса задаёт контракт (допустимые аргументы, возвращаемое значение, возможные исключения), и подкласс не вправе этот контракт сужать.

Метод `getDividends(BigDecimal)`, объявленный в `Instrument`, не имеет осмысленной реализации для `Bond` — облигация приносит купонный доход, а не дивиденды:

```java
// Нарушение LSP — оба варианта искажают контракт
public BigDecimal getDividends(BigDecimal shareCount) {
    return BigDecimal.ZERO;                                    // молчаливая подмена
}
public BigDecimal getDividends(BigDecimal shareCount) {
    throw new UnsupportedOperationException("Bonds do not pay dividends");  // отказ
}
```

Корректное решение — вынести операцию из базового класса в отдельную абстракцию, реализуемую только теми типами, для которых она осмысленна:

```java
public interface DividendPaying {
    BigDecimal getDividends(BigDecimal shareCount);
}

public class Stock extends Instrument implements DividendPaying {
    @Override
    public BigDecimal getDividends(BigDecimal shareCount) {
        return dividendPerShare.multiply(shareCount);
    }
}
```

`Bond` интерфейс `DividendPaying` не реализует; код, работающий с дивидендами, оперирует типом `DividendPaying`, и некорректный вызов становится невозможен на уровне компиляции.

---

## 3. Принцип разделения интерфейса (ISP)

Клиенты не должны зависеть от методов, которые они не используют. Единый интерфейс `Instrument` со всеми возможными операциями (дивиденды, купонный доход, дата погашения, состав активов) заставил бы каждый подкласс реализовывать чужие обязанности — то есть привёл бы к тому же нарушению LSP, что и в разделе 2. ISP и LSP взаимосвязаны: типичная причина нарушения LSP — «толстый» интерфейс, а устранение — разложение на узкие роли:

```java
public interface DividendPaying { BigDecimal getDividends(BigDecimal shareCount); }
public interface CouponBearing { BigDecimal getCouponYield(); LocalDate getMaturityDate(); }
public interface Diversified { List<Instrument> getUnderlyingAssets(); }
```

Класс реализует только те интерфейсы, которые соответствуют его природе: `Stock implements DividendPaying`, `Bond implements CouponBearing`, `Etf implements Diversified`.

---

## 4. Принцип единственной ответственности (SRP)

У класса должна быть одна причина для изменения. Если в `Portfolio` добавить наряду с расчётом стоимости ещё и вывод в консоль:

```java
public class Portfolio {
    public BigDecimal totalValue(Map<String, BigDecimal> prices) { /* доменная логика */ }
    public void printReport() { /* форматирование и вывод */ }
}
```

`printReport` вводит вторую причину для изменения класса — переход на формат JSON или вывод в файл потребует править `Portfolio` по причине, не связанной с учётом позиций. Форматирование выносится в отдельный компонент. Тот же вопрос применим к CLI-командам: команда, которая одновременно парсит аргументы, обращается к данным и форматирует вывод, нарушает SRP так же, как `Portfolio` с `printReport`.

---

## 5. Принцип открытости/закрытости (OCP)

Код должен быть открыт для расширения, но закрыт для изменения. Ранняя версия определения типа инструмента через `instanceof`-цепочку требовала правки при каждом новом типе:

```java
if (type == InstrumentType.STOCK) return "Акция";
else if (type == InstrumentType.BOND) return "Облигация";
else if (type == InstrumentType.ETF) return "Фонд";
else throw new IllegalStateException("Unknown type");
```

Переход к полиморфному `getDescription()`, выполненный на первом семинаре, устраняет проблему: добавление нового подкласса `Instrument` не требует изменения ни одного существующего класса. Тот же принцип будет проверен сегодня при реализации создания объектов через фабрику и фильтрации инструментов через Strategy: добавление нового типа или нового правила отбора должно сводиться к добавлению нового класса, а не к правке существующего.

---

## 6. Принцип инверсии зависимостей (DIP) — обзорно

Модули верхнего уровня не должны зависеть от модулей нижнего уровня — оба должны зависеть от абстракций. На практике это означает, что бизнес-логика не должна напрямую зависеть от конкретной реализации внешнего клиента или библиотеки — она должна зависеть от интерфейса, скрывающего детали.

Полноценное применение DIP становится наглядным при работе с фреймворками, управляющими внедрением зависимостей (dependency injection) — тема будет подробно разобрана при знакомстве со Spring на одном из следующих семинаров.

---

## 7. KISS и DRY

**KISS** (Keep It Simple, Stupid) — предпочтение простого решения сложному при прочих равных: паттерн проектирования оправдан, когда решает реальную проблему, а не потому, что был недавно изучен.

**DRY** (Don't Repeat Yourself) — устранение дублирования *знания*, а не дублирования текста. Два похожих, но независимо изменяющихся фрагмента кода не обязательно нарушают DRY; преждевременное обобщение ради устранения текстового сходства может дать абстракцию хуже дублирования.

---

Принципы SOLID задают критерии качества дизайна, но не дают готовых конструкций для их достижения — такие конструкции представляют собой паттерны проектирования (GoF), рассматриваемые в практической части семинара.
