### Этап 2. Подключение picocli (~5 мин)

Проект переходит от точки входа `demo.Main` к полноценному CLI. В качестве библиотеки для разбора аргументов командной строки используется [picocli](https://picocli.info/) — позволяет декларативно описывать команды и опции через аннотации, не реализуя разбор аргументов вручную.

```xml
<dependency>
    <groupId>info.picocli</groupId>
    <artifactId>picocli</artifactId>
    <version>4.7.7</version>
</dependency>
```

Краткий обзор основных классов picocli, без разбора отдельного демонстрационного примера:

```java
/**
 * Основные типы picocli, используемые в проекте:
 *
 * @Command       — аннотация класса-команды (имя, описание, вложенные subcommands)
 * @Option         — аннотация именованного параметра (например, --type)
 * @Parameters     — аннотация позиционного параметра (например, тикер)
 * Callable<Integer> — интерфейс, реализуемый командой; call() возвращает код завершения
 * CommandLine     — точка запуска: разбирает args и вызывает call() нужной команды
 */
```

### Этап 3. Обсуждение: какие паттерны GoF применимы к CLI (~10 мин)

Формат — открытое обсуждение с группой, а не лекция. Вопросы для обсуждения:

- Какие паттерны проектирования уже знакомы студентам (из прошлого опыта, других курсов, самостоятельного изучения)?
- Как устроена обработка команды в CLI-приложении: что должно происходить, когда пользователь вводит `search AAPL`?
- Как устроено создание объектов `Stock`, `Bond`, `Etf`, когда параметры инструмента приходят не из кода, а из строки, введённой пользователем?
- Как отфильтровать список инструментов по типу так, чтобы добавление нового типа инструмента не требовало правки кода фильтрации?
- Как перебрать все инструменты в хранилище, не раскрывая наружу, как именно они там хранятся?
- Как добавить к любому методу репозитория побочное поведение (замер времени выполнения, логирование), не трогая код самого репозитория?

Ожидаемое направление ответов и разбор кандидатов:

| **Задача**                                                        | **Паттерн-кандидат** |
| ----------------------------------------------------------------- | -------------------- |
| Обработка пользовательской команды                                | Command              |
| Создание объекта инструмента по строковому типу                   | Factory Method       |
| Фильтрация списка инструментов без хрупкого сравнения по типу     | Strategy             |
| Обход инструментов в хранилище без раскрытия внутренней структуры | Iterator             |
| Побочное поведение вокруг вызовов метода без изменения его кода   | Proxy                |

Обсуждается, что picocli уже частично реализует Command на уровне библиотеки — каждая аннотированная классом `@Command` команда, реализующая `Callable`, фактически является объектом-командой с методом `call()` в роли `execute()`. Это наблюдение используется как переход к совместной реализации: паттерн не придумывается с нуля, а узнаётся в уже используемом инструменте.

Iterator фиксируется в списке кандидатов, но не реализуется совместно — он отдаётся студентам на этапе 5. Proxy фиксируется, но подробно разбирается только на этапе 8, после того как остальные паттерны уже реализованы и есть на чём его продемонстрировать.

### Этап 4. Совместная реализация: Factory Method, Strategy (~25–30 мин)

**Factory Method: создание инструмента по типу.** Пользователь вводит тип инструмента строкой (`STOCK`, `BOND`, `ETF`); код, создающий объект, не должен разрастаться в `if/else` при добавлении нового типа:

```java
public interface InstrumentFactory {
    Instrument create(String ticker, String name, BigDecimal price);
}

public final class InstrumentFactories {
    private static final Map<String, InstrumentFactory> REGISTRY = new HashMap<>();

    private InstrumentFactories() {
    }

    public static void register(String type, InstrumentFactory factory) {
        REGISTRY.put(type.toUpperCase(), factory);
    }

    public static Instrument create(String type, String ticker, String name, BigDecimal price) {
        InstrumentFactory factory = REGISTRY.get(type.toUpperCase());
        if (factory == null) {
            throw new IllegalArgumentException("Unknown instrument type: " + type);
        }
        return factory.create(ticker, name, price);
    }
}

public class StockFactory implements InstrumentFactory {

    static {
        InstrumentFactories.register("STOCK", new StockFactory());
    }

    @Override
    public Instrument create(String ticker, String name, BigDecimal price) {
        return new Stock(ticker, name, price);
    }
}
```

Подход аналогичен регистрации драйверов в JDBC (`DriverManager`): реестр не содержит статического перечисления всех фабрик, а каждая фабрика регистрирует себя сама при загрузке своего класса. Добавление нового типа инструмента сводится к реализации нового `InstrumentFactory` со статическим блоком регистрации — ни `InstrumentFactories`, ни существующие фабрики не изменяются, что напрямую иллюстрирует OCP.

**Strategy: фильтрация инструментов по типу.** Наивный способ реализовать фильтр `--type` в команде `list` — сравнение строки с именем класса напрямую в коде команды. Такое условие разрастётся при появлении второго правила отбора. Strategy инкапсулирует условие отбора в отдельный, взаимозаменяемый объект:

```java
public interface InstrumentFilter {
    boolean matches(Instrument instrument);
}

public class TypeFilter implements InstrumentFilter {
    private final String type;

    public TypeFilter(String type) {
        this.type = type;
    }

    @Override
    public boolean matches(Instrument instrument) {
        return instrument.getClass().getSimpleName().equalsIgnoreCase(type);
    }
}

public class NoOpFilter implements InstrumentFilter {
    @Override
    public boolean matches(Instrument instrument) {
        return true;
    }
}
```

Добавление нового способа отбора (например, `PriceAboveFilter`) сводится к реализации нового `InstrumentFilter` — код, перебирающий инструменты, не изменяется.

### Этап 5. Самостоятельная работа: Iterator над `InstrumentRepository` (~15 мин)

Задание: `InstrumentRepository` хранит инструменты во внутреннем массиве. Требуется реализовать возможность перебора всех инструментов репозитория, не раскрывая наружу сам массив (то есть без метода вида `getAll(): Instrument[]`). Если метод добавления записей (`add(Instrument instrument)`) в репозитории ещё отсутствует, его необходимо добавить — без него нечем будет наполнить репозиторий для проверки перебора.

Условия:

- Реализовать собственный тип, предоставляющий методы `hasNext()` и `next()` (пока без реализации стандартного интерфейса `java.util.Iterator` — это будет показано на этапе 6).
- Проверить работу реализации классическим циклом:

```java
InstrumentIterator iterator = repository.iterator();
while (iterator.hasNext()) {
    Instrument instrument = iterator.next();
    System.out.println(instrument.getDescription());
}
```

- `for-each` на этом этапе не использовать — синтаксис ещё не разобран и будет объяснён на этапе 6.

### Этап 6. Разбор решений, эталонная реализация Iterator (~10–15 мин)

Разбор одного-двух студенческих решений: как авторы оформили собственный интерфейс `InstrumentIterator` (отдельный интерфейс, вложенный класс, анонимный класс), как обработана ситуация вызова `next()` после исчерпания элементов.

Эталонная реализация — переход от собственного интерфейса к `java.util.Iterator<Instrument>` и `Iterable<Instrument>`. Одновременно `InstrumentRepository` выделяется в интерфейс, а текущая реализация переименовывается в `InMemoryInstrumentRepository` — это понадобится на этапе 8 для JDK Dynamic Proxy, которому для работы требуется интерфейс:

```java
public interface InstrumentRepository extends Iterable<Instrument> {
    void add(Instrument instrument);
    Instrument findByTicker(String ticker);
}

public class InMemoryInstrumentRepository implements InstrumentRepository {
    private Instrument[] instruments = new Instrument[100];
    private int size = 0;

    @Override
    public void add(Instrument instrument) {
        instruments[size++] = instrument;
    }

    @Override
    public Instrument findByTicker(String ticker) {
        for (Instrument instrument : this) {
            if (instrument.getTicker().equalsIgnoreCase(ticker)) {
                return instrument;
            }
        }
        return null;
    }

    @Override
    public Iterator<Instrument> iterator() {
        return new Iterator<>() {
            private int cursor = 0;

            @Override
            public boolean hasNext() {
                return cursor < size;
            }

            @Override
            public Instrument next() {
                if (!hasNext()) {
                    throw new NoSuchElementException();
                }
                return instruments[cursor++];
            }
        };
    }
}
```

Далее — явная демонстрация, что `for-each` является синтаксическим сахаром над `Iterator`, на паре эквивалентных фрагментов:

```java
// то, что пишет программист
for (Instrument instrument : repository) {
    System.out.println(instrument.getDescription());
}

// во что это разворачивается компилятором
Iterator<Instrument> iterator = repository.iterator();
while (iterator.hasNext()) {
    Instrument instrument = iterator.next();
    System.out.println(instrument.getDescription());
}
```

Реализация `Iterable<Instrument>` — условие, необходимое и достаточное для использования `for-each`; без него компилятор не примет объект в конструкции `for (... : repository)`.

### Этап 7. Совместная реализация: Command поверх picocli, команда `list --type` (~15–20 мин)

Первые доменные команды приложения — `SearchCommand` и `AddCommand`, каждая инкапсулирует один сценарий использования и делегирует получение данных доменному слою, не смешивая парсинг аргументов с бизнес-логикой (связь с SRP):

```java
@Command(name = "search", description = "Поиск инструмента по тикеру")
public class SearchCommand implements Callable<Integer> {

    @Parameters(index = "0", description = "Тикер инструмента")
    private String ticker;

    private final InstrumentRepository repository;

    public SearchCommand(InstrumentRepository repository) {
        this.repository = repository;
    }

    @Override
    public Integer call() {
        Instrument instrument = repository.findByTicker(ticker);
        if (instrument == null) {
            System.out.println("Инструмент не найден: " + ticker);
            return 1;
        }
        System.out.println(instrument.getDescription());
        return 0;
    }
}
```

Команда `add`, использующая `InstrumentFactories` из этапа 4 — связывает Factory Method и Command: создание объекта остаётся ответственностью фабрики, команда лишь разбирает аргументы и передаёт их дальше:

```java
@Command(name = "add", description = "Добавление инструмента в репозиторий")
public class AddCommand implements Callable<Integer> {

    @Parameters(index = "0", description = "Тип инструмента (STOCK, BOND, ETF)")
    private String type;

    @Parameters(index = "1", description = "Тикер инструмента")
    private String ticker;

    @Parameters(index = "2", description = "Название инструмента")
    private String name;

    @Parameters(index = "3", description = "Цена инструмента")
    private BigDecimal price;

    private final InstrumentRepository repository;

    public AddCommand(InstrumentRepository repository) {
        this.repository = repository;
    }

    @Override
    public Integer call() {
        Instrument instrument = InstrumentFactories.create(type, ticker, name, price);
        repository.add(instrument);
        System.out.println("Добавлено: " + instrument.getDescription());
        return 0;
    }
}
```

Команда `list`, использующая итератор из этапа 6 (уже в форме `for-each`, поскольку сахар разобран) и `InstrumentFilter` из этапа 4 вместо прямого сравнения строк:

```java
@Command(name = "list", description = "Список инструментов")
public class ListCommand implements Callable<Integer> {

    @Option(names = "--type", description = "Фильтр по типу инструмента")
    private String type;

    private final InstrumentRepository repository;

    public ListCommand(InstrumentRepository repository) {
        this.repository = repository;
    }

    @Override
    public Integer call() {
        InstrumentFilter filter = (type == null) ? new NoOpFilter() : new TypeFilter(type);

        for (Instrument instrument : repository) {
            if (filter.matches(instrument)) {
                System.out.println(instrument.getDescription());
            }
        }
        return 0;
    }
}
```

Точка входа приложения регистрирует команды в корневом объекте picocli:

```java
@Command(name = "market-pulse", subcommands = {SearchCommand.class, AddCommand.class, ListCommand.class})
public class MarketPulseCli implements Runnable {
    @Override
    public void run() {
        System.out.println("Используйте --help для списка команд");
    }
}

public class Main {
    public static void main(String[] args) {
        InstrumentRepository repository = new InMemoryInstrumentRepository();
        int exitCode = new CommandLine(new MarketPulseCli()).execute(args);
        System.exit(exitCode);
    }
}
```

### Этап 8. Proxy: самописный, JDK Dynamic Proxy, CGLIB (~25–30 мин)

Представим ситуацию: приложение обращается к смежной системе — например, к T-Invest API за котировками — и в последнее время такие обращения стали выполняться заметно дольше обычного. Чтобы разобраться в причине, нужно замерять время выполнения каждого метода `InstrumentRepository` и логировать его вместе с параметрами вызова.

Самый прямолинейный вариант — просто прописать замер времени и логирование в начале и в конце каждого метода, который делает такое обращение. Задачу это решает, но нарушает сразу два принципа: SRP (метод отвечает и за собственную логику, и за диагностику) и DRY (пара «замерить время → залогировать» дублируется в каждом методе). Естественный следующий шаг — вынести эту сквозную логику в отдельный объект-обёртку, которому делегируется вызов; рассмотренные ниже варианты расположены по нарастанию автоматизации, от полностью ручной работы до генерации кода библиотекой. Подробное сопоставление вынесено в материал `java_proxying.md`, здесь — реализация каждого варианта на примере проекта.

**Вариант 1 — самописный прокси.** Класс, реализующий тот же интерфейс, что и целевой объект, оборачивает каждый вызов вручную:

```java
public class TimingInstrumentRepositoryProxy implements InstrumentRepository {
    private final InstrumentRepository delegate;

    public TimingInstrumentRepositoryProxy(InstrumentRepository delegate) {
        this.delegate = delegate;
    }

    @Override
    public void add(Instrument instrument) {
        long start = System.nanoTime();
        delegate.add(instrument);
        System.out.printf("add() выполнен за %d нс%n", System.nanoTime() - start);
    }

    @Override
    public Instrument findByTicker(String ticker) {
        long start = System.nanoTime();
        Instrument result = delegate.findByTicker(ticker);
        System.out.printf("findByTicker() выполнен за %d нс%n", System.nanoTime() - start);
        return result;
    }

    @Override
    public Iterator<Instrument> iterator() {
        return delegate.iterator();
    }
}
```

Обсуждается со студентами итог: дублирование никуда не делось, оно просто переехало из целевого класса в прокси — на каждый метод `InstrumentRepository` по-прежнему приходится вручную писать одинаковую пару «замерить → делегировать → залогировать», а добавление нового метода в интерфейс требует не забыть дописать его и здесь. Профита относительно наивного варианта немного, а кода в проекте стало больше.

Следующие два варианта решают именно эту проблему — прокси генерируется библиотекой во время выполнения, и сквозная логика описывается один раз для всех методов сразу.

**Вариант 2 — JDK Dynamic Proxy.** `java.lang.reflect.Proxy` генерирует класс, реализующий заданные интерфейсы, во время выполнения; вызовы перехватываются единственным `InvocationHandler`, без ручного дублирования сигнатур методов:

```java
InstrumentRepository target = new InMemoryInstrumentRepository();

InstrumentRepository proxy = (InstrumentRepository) Proxy.newProxyInstance(
        InstrumentRepository.class.getClassLoader(),
        new Class<?>[] { InstrumentRepository.class },
        (Object p, Method method, Object[] args) -> {
            long start = System.nanoTime();
            Object result = method.invoke(target, args);
            System.out.printf("%s() выполнен за %d нс%n", method.getName(), System.nanoTime() - start);
            return result;
        });
```

Обсуждается ограничение: сгенерированный класс наследует `java.lang.reflect.Proxy`, поэтому может реализовывать только интерфейсы — прокси для класса без интерфейса этим способом не построить.

**Вариант 3 — CGLIB.** Работает подстановкой подкласса и не требует интерфейса — годится для случаев, когда прокси нужен непосредственно для `InMemoryInstrumentRepository`, минуя интерфейс:

```xml
<dependency>
    <groupId>cglib</groupId>
    <artifactId>cglib</artifactId>
    <version>3.3.0</version>
</dependency>
```

```java
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(InMemoryInstrumentRepository.class);
enhancer.setCallback((MethodInterceptor) (obj, method, args, methodProxy) -> {
    long start = System.nanoTime();
    Object result = methodProxy.invokeSuper(obj, args);
    System.out.printf("%s() выполнен за %d нс%n", method.getName(), System.nanoTime() - start);
    return result;
});

InstrumentRepository repository = (InstrumentRepository) enhancer.create();
```

Обсуждается ограничение: CGLIB генерирует подкласс целевого класса, поэтому не может проксировать `final`-классы и `final`-методы — переопределить их в сгенерированном подклассе невозможно.

Итог этапа — сопоставление трёх вариантов по критериям, которое совпадает с итоговой таблицей материала `java_proxying.md`: необходимость интерфейса, работа с `final`, дополнительные зависимости, накладные расходы на вызов.

## 1. От ООП к принципам проектирования

Механизмы ООП допускают как удачное, так и неудачное применение; сам по себе язык не проверяет качество дизайна. Принципы SOLID формулируют пять правил, следование которым делает код устойчивым к изменению и расширению:

- **S** — Single Responsibility Principle;
- **O** — Open/Closed Principle;
- **L** — Liskov Substitution Principle;
- **I** — Interface Segregation Principle;
- **D** — Dependency Inversion Principle.

---

## 2. Принцип подстановки Лисков (LSP)

Объекты подтипа должны быть заменяемы объектами базового типа без нарушения корректности программы. Метод базового класса задаёт контракт (допустимые аргументы, возвращаемое значение, возможные исключения), и подкласс не вправе этот контракт сужать.

Метод `getDividends(BigDecimal)`, объявленный в `Instrument`, не имеет осмысленной реализации для `Bond` — облигация приносит купонный доход, а не дивиденды:

```java
// Нарушение LSP — оба варианта искажают контракт
public BigDecimal getDividends(BigDecimal shareCount) {
    return BigDecimal.ZERO;                                    // молчаливая подмена
}
public BigDecimal getDividends(BigDecimal shareCount) {
    throw new UnsupportedOperationException("Bonds do not pay dividends");  // отказ
}
```

Корректное решение — вынести операцию из базового класса в отдельную абстракцию, реализуемую только теми типами, для которых она осмысленна:

```java
public interface DividendPaying {
    BigDecimal getDividends(BigDecimal shareCount);
}

public class Stock extends Instrument implements DividendPaying {
    @Override
    public BigDecimal getDividends(BigDecimal shareCount) {
        return dividendPerShare.multiply(shareCount);
    }
}
```

`Bond` интерфейс `DividendPaying` не реализует; код, работающий с дивидендами, оперирует типом `DividendPaying`, и некорректный вызов становится невозможен на уровне компиляции.

---

## 3. Принцип разделения интерфейса (ISP)

Клиенты не должны зависеть от методов, которые они не используют. Единый интерфейс `Instrument` со всеми возможными операциями (дивиденды, купонный доход, дата погашения, состав активов) заставил бы каждый подкласс реализовывать чужие обязанности — то есть привёл бы к тому же нарушению LSP, что и в разделе 2. ISP и LSP взаимосвязаны: типичная причина нарушения LSP — «толстый» интерфейс, а устранение — разложение на узкие роли:

```java
public interface DividendPaying { BigDecimal getDividends(BigDecimal shareCount); }
public interface CouponBearing { BigDecimal getCouponYield(); LocalDate getMaturityDate(); }
public interface Diversified { List<Instrument> getUnderlyingAssets(); }
```

Класс реализует только те интерфейсы, которые соответствуют его природе: `Stock implements DividendPaying`, `Bond implements CouponBearing`, `Etf implements Diversified`.

> `List` — часть Java Collection Framework, подробно рассматривается на семинаре 5. `LocalDate` — класс из пакета `java.time`, работа со временем в Java разбирается точечно, по мере необходимости.

---

## 4. Принцип единственной ответственности (SRP)

У класса должна быть одна причина для изменения. Если в `Portfolio` добавить наряду с расчётом стоимости ещё и вывод в консоль:

```java
public class Portfolio {
    public BigDecimal totalValue(Map<String, BigDecimal> prices) { /* доменная логика */ }
    public void printReport() { /* форматирование и вывод */ }
}
```

`printReport` вводит вторую причину для изменения класса — переход на формат JSON или вывод в файл потребует править `Portfolio` по причине, не связанной с учётом позиций. Форматирование выносится в отдельный компонент. Тот же вопрос применим к CLI-командам: команда, которая одновременно парсит аргументы, обращается к данным и форматирует вывод, нарушает SRP так же, как `Portfolio` с `printReport`.

> `Map` — часть Java Collection Framework, подробно рассматривается на семинаре 5.

---

## 5. Принцип открытости/закрытости (OCP)

Код должен быть открыт для расширения, но закрыт для изменения. Ранняя версия определения типа инструмента через `instanceof`-цепочку требовала правки при каждом новом типе:

```java
if (type == InstrumentType.STOCK) return "Акция";
else if (type == InstrumentType.BOND) return "Облигация";
else if (type == InstrumentType.ETF) return "Фонд";
else throw new IllegalStateException("Unknown type");
```

Переход к полиморфному `getDescription()`, выполненный на первом семинаре, устраняет проблему: добавление нового подкласса `Instrument` не требует изменения ни одного существующего класса. Тот же принцип применим к созданию объектов через фабрику и фильтрации инструментов через Strategy: добавление нового типа или нового правила отбора должно сводиться к добавлению нового класса, а не к правке существующего.

---

## 6. Принцип инверсии зависимостей (DIP) — обзорно

Модули верхнего уровня не должны зависеть от модулей нижнего уровня — оба должны зависеть от абстракций. На практике это означает, что бизнес-логика не должна напрямую зависеть от конкретной реализации внешнего клиента или библиотеки — она должна зависеть от интерфейса, скрывающего детали.

Полноценное применение DIP становится наглядным при работе с фреймворками, управляющими внедрением зависимостей (dependency injection) — тема будет подробно разобрана при знакомстве со Spring на одном из следующих семинаров.

---

## 7. Взаимосвязь принципов друг с другом и с ООП

Принципы SOLID не независимы друг от друга — каждый опирается на конкретный механизм ООП и одновременно создаёт условия, при которых соблюдение соседних принципов становится естественным или, напротив, необходимым.

**Связь с механизмами ООП.** Каждый принцип регламентирует использование одного из столпов ООП:

| **Механизм ООП**        | **Принцип, который его регламентирует**                                                                                              |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| Наследование            | LSP — подтип обязан вести себя как заменитель базового типа                                                                          |
| Инкапсуляция            | SRP — единица инкапсуляции должна отвечать за одну область                                                                           |
| Полиморфизм             | OCP — расширение поведения через новые реализации, а не через изменение существующих                                                 |
| Абстракция (интерфейсы) | ISP — абстракция должна быть узкой, под конкретного потребителя; DIP — зависимость строится на абстракции, а не на конкретном классе |

**Связи между принципами.** ISP и LSP взаимно обусловлены: «толстый» интерфейс (нарушение ISP) почти всегда вынуждает часть подклассов реализовывать чужие операции заглушками или исключениями (нарушение LSP) — это было продемонстрировано в разделах 2–3 на примере `Instrument`. SRP можно рассматривать как ISP, применённый не к интерфейсу, а к классу: у класса, как и у интерфейса, должна быть одна причина для изменения, единая ось ответственности.

OCP оказывается следствием, а не независимым правилом: расширяемость без модификации существующего кода становится возможной только тогда, когда система уже полагается на LSP (новый подкласс действительно подставим на место базового), на DIP (клиентский код зависит от абстракции, а не от конкретной реализации) и на ISP (абстракция достаточно узкая, чтобы новая реализация не тянула за собой лишних обязательств). В этом смысле OCP — не столько отдельная техника, сколько наблюдаемый эффект от корректного применения остальных четырёх принципов.

DIP связывает эту картину с полиморфизмом на уровне архитектуры: если модуль зависит от абстракции, а не от конкретного класса, подмена реализации (в том числе ради соблюдения OCP) не требует изменения зависящего кода — ровно тот же полиморфный механизм, что используется в LSP, применяется здесь на уровне модулей, а не отдельных объектов.

---

## 8. KISS и DRY

**KISS** (Keep It Simple, Stupid) — предпочтение простого решения сложному при прочих равных: паттерн проектирования оправдан, когда решает реальную проблему, а не потому, что был недавно изучен.

**DRY** (Don't Repeat Yourself) — устранение дублирования *знания*, а не дублирования текста. Два похожих, но независимо изменяющихся фрагмента кода не обязательно нарушают DRY; преждевременное обобщение ради устранения текстового сходства может дать абстракцию хуже дублирования.

---

Принципы SOLID задают критерии качества дизайна, но не дают готовых конструкций для их достижения — такие конструкции представляют собой паттерны проектирования (GoF).

## 1. Зачем нужен объект-заместитель

Proxy — структурный паттерн, предоставляющий объект-заместитель, который перехватывает обращения к другому объекту и добавляет собственное поведение вокруг вызова (замер времени, логирование, кэширование, отложенная инициализация, проверка прав доступа), не изменяя код целевого объекта.

Клиентский код взаимодействует с прокси так же, как с самим целевым объектом — оба реализуют один и тот же контракт, и клиент не обязан знать, с чем именно он работает напрямую.

В Java эта идея реализуется тремя принципиально разными способами, различающимися по тому, на каком этапе создаётся класс-заместитель: во время написания кода, во время выполнения на основе интерфейса или во время выполнения на основе подкласса.

---

## 2. Самописный (статический) прокси

Класс-заместитель пишется вручную, реализует тот же интерфейс, что и целевой объект, хранит ссылку на него и делегирует каждый вызов, добавляя собственную логику до или после делегирования:

```java
public interface Repository {
    void save(Object entity);
    Object findById(long id);
}

public class TimingRepositoryProxy implements Repository {
    private final Repository delegate;

    public TimingRepositoryProxy(Repository delegate) {
        this.delegate = delegate;
    }

    @Override
    public void save(Object entity) {
        long start = System.nanoTime();
        delegate.save(entity);
        System.out.printf("save() выполнен за %d нс%n", System.nanoTime() - start);
    }

    @Override
    public Object findById(long id) {
        long start = System.nanoTime();
        Object result = delegate.findById(id);
        System.out.printf("findById() выполнен за %d нс%n", System.nanoTime() - start);
        return result;
    }
}
```

**Преимущества.** Код прозрачен и отлаживается как обычный класс — никакой генерации байт-кода или рефлексии, минимальные накладные расходы на вызов, отсутствие зависимости от сторонних библиотек.

**Ограничения.** Число классов-прокси растёт пропорционально числу проксируемых интерфейсов; каждый новый метод интерфейса требует ручного добавления реализации в прокси, а сквозная логика (например, замер времени) дублируется в каждом методе каждого прокси, что нарушает DRY при росте числа проксируемых типов.

---

## 3. JDK Dynamic Proxy

Пакет `java.lang.reflect` предоставляет механизм генерации класса-заместителя во время выполнения — без необходимости писать класс прокси вручную. Генерация опирается на интерфейс: `Proxy.newProxyInstance` создаёт объект, реализующий заданный набор интерфейсов, все вызовы методов которого перенаправляются в единственный обработчик `InvocationHandler`:

```java
Repository target = new InMemoryRepository();

Repository proxy = (Repository) Proxy.newProxyInstance(
        Repository.class.getClassLoader(),
        new Class<?>[] { Repository.class },
        (Object p, Method method, Object[] args) -> {
            long start = System.nanoTime();
            Object result = method.invoke(target, args);
            System.out.printf("%s() выполнен за %d нс%n", method.getName(), System.nanoTime() - start);
            return result;
        });
```

Один `InvocationHandler` покрывает все методы интерфейса — при добавлении нового метода в интерфейс прокси автоматически начинает его перехватывать, без правки кода обработчика.

**Преимущества.** Не требует сторонних библиотек — механизм входит в стандартную библиотеку Java; один обработчик покрывает произвольное число методов и интерфейсов; логика перехвата отделена от контракта в явном виде.

**Ограничения.** Работает только с интерфейсами — прокси не может быть создан для класса без интерфейса, а сгенерированный класс не может дополнительно наследовать другой класс (он уже наследует `java.lang.reflect.Proxy`). Каждый вызов метода проходит через `Method.invoke`, что дороже прямого вызова за счёт рефлексии — на большинстве задач разница пренебрежимо мала, но заметна в горячих участках кода с интенсивными вызовами.

---

## 4. CGLIB

CGLIB (Code Generation Library) — сторонняя библиотека, генерирующая прокси иначе: не через реализацию интерфейса, а через создание подкласса целевого класса во время выполнения. Это снимает главное ограничение JDK Dynamic Proxy — интерфейс становится не обязателен:

```java
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(InMemoryRepository.class);
enhancer.setCallback((MethodInterceptor) (obj, method, args, methodProxy) -> {
    long start = System.nanoTime();
    Object result = methodProxy.invokeSuper(obj, args);
    System.out.printf("%s() выполнен за %d нс%n", method.getName(), System.nanoTime() - start);
    return result;
});

Repository proxy = (Repository) enhancer.create();
```

`Enhancer` генерирует новый класс байт-кодом непосредственно во время выполнения (в отличие от JDK Dynamic Proxy, использующего для этого более узкий встроенный механизм), переопределяя методы целевого класса и вызывая исходную реализацию через `methodProxy.invokeSuper`.

**Преимущества.** Не требует интерфейса — можно проксировать конкретный класс напрямую; переопределение методов подклассом обычно быстрее вызова через рефлексию, поскольку сгенерированный байт-код вызывает метод напрямую, а не через `Method.invoke`.

**Ограничения.** Генерация подкласса означает, что переопределить `final`-класс или `final`-метод невозможно — переопределение запрещено языком независимо от способа генерации кода. Требует дополнительной зависимости (`cglib`, либо получаемой транзитивно через другую библиотеку). Целевой класс должен иметь доступный конструктор, вызываемый при создании прокси (в готовых фреймворках, например в Spring, для обхода этого ограничения дополнительно используется библиотека Objenesis, позволяющая создавать экземпляр класса без вызова конструктора).

## 5. Сравнение подходов

| **Критерий**                              | **Самописный прокси**                                                  | **JDK Dynamic Proxy**                                          | **CGLIB**                                              |
| ----------------------------------------- | ---------------------------------------------------------------------- | -------------------------------------------------------------- | ------------------------------------------------------ |
| Требует интерфейс                         | Да                                                                     | Да                                                             | Нет                                                    |
| Работает с `final`-классами/методами      | — (ограничение не применимо)                                           | — (ограничение не применимо)                                   | Нет                                                    |
| Дополнительная зависимость                | Нет                                                                    | Нет (входит в JDK)                                             | Да                                                     |
| Поддержка новых методов без правки прокси | Нет — нужно дописывать вручную                                         | Да — автоматически                                             | Да — автоматически                                     |
| Относительная скорость вызова             | Без косвенности                                                        | Медленнее — вызов через рефлексию                              | Быстрее JDK Dynamic Proxy, но медленнее прямого вызова |
| Когда уместен                             | Небольшое, стабильное число методов; максимальная прозрачность отладки | Есть интерфейс; число проксируемых типов и методов может расти | Интерфейса нет или проксировать нужно конкретный класс |

## 6. Проксирование в промышленных фреймворках

Оба динамических механизма — не учебная абстракция, а основа механизма Dependency Injection в Spring: если бин реализует хотя бы один интерфейс, Spring по умолчанию создаёт JDK Dynamic Proxy; если интерфейса нет — CGLIB. Тем же способом реализованы декларативные транзакции (`@Transactional`), кэширование (`@Cacheable`) и аспектно-ориентированное программирование (Spring AOP в целом): аннотация на методе фактически превращает вызов в обращение к прокси, добавляющему поведение вокруг оригинального метода — ту же задачу, что решают примеры этого материала, только для более общего круга сквозных задач.
