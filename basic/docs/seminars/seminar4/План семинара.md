### Этап 1. Мотивация: два способа сломаться (~15 мин)

**Самостоятельное исследование (~5 мин).** Прежде чем показывать заготовленные примеры, даём студентам открыть проект
в состоянии семинара 3 и самим попробовать его сломать — задание группе:

> Поэкспериментируйте с командами `add`, `search`, `list`. Попробуйте ввести то, чего программа не ожидает:
> несуществующий тип инструмента, уже занятый тикер, что угодно ещё. Что удалось сломать? Как именно это выглядит —
> программа падает с ошибкой, зависает, или просто делает что-то не то и молчит?

Через 5 минут — короткое обсуждение находок: кто что сломал и чем один способ сломать отличался от другого. Дальше
разбираем два характерных случая — скорее всего, оба уже прозвучали в находках группы.

**Поломка первая — громкая, но никем не услышанная.**

```bash
market-pulse add CRYPTO BTC Bitcoin RUB
```

С семинара 3 `InstrumentFactories.create` уже не падает с `NullPointerException` на неизвестном типе:
`Registry.get()` возвращает `Optional`, и `create()` сам бросает осмысленный
`IllegalArgumentException("Unknown instrument type: CRYPTO")` через `orElseThrow`. Но в `AddCommand` это исключение
никто не ловит — оно долетает до самого верха необработанным, и вместо понятного сообщения об ошибке пользователь
видит сырой Java-стектрейс на экране. Тип исключения уже правильный и информативный — но никем не услышанный.

**Поломка вторая — слишком тихая.**

```bash
market-pulse add STOCK SBER Сбербанк RUB
market-pulse add STOCK SBER Сбербанк RUB
```

Обе команды выполняются успешно, никакой ошибки. Но `InstrumentRepository.add` ничего не проверяет — в массиве теперь
два инструмента с тикером `SBER`, а `findByTicker` всегда находит только первый. Второй `add` не принёс никакой пользы
и никак об этом не сообщил — пользователь уверен, что добавление сработало, хотя толку от него никакого.

> **Вопрос группе:** какая из двух поломок хуже? Первая пугает, но честна — сразу видно, что что-то не так. Вторая
> молчит, и это опаснее: программа продолжает работать, но с неверными данными.

Сегодня наводим порядок в обеих ситуациях: даём громкой поломке услышанный голос, а тихой — голос вообще.

### Этап 2. Теория: коды возврата, исключения, иерархия Throwable (~15 мин)

**Коды возврата и специальные значения (sentinel values).** Классический до-исключений подход: функция возвращает не
результат, а признак ошибки — `-1`, `null`, глобальную переменную вроде `errno` в C. С этим стилем уже сталкивались:
`findByTicker` до семинара 3 возвращал `null`, если инструмент не найден, — тот же паттерн.

Плюсы: явный, предсказуемый, дешёвый. Минусы: проверку результата легко забыть — компилятор не заставит; при цепочке
вызовов проверки на каждом шаге захламляют «счастливый путь» кода; несколько разных причин ошибки трудно закодировать
одним специальным значением.

**Исключения.** Механизм JVM: `throw` прерывает нормальное выполнение и передаёт объект `Throwable` вверх по стеку
вызовов, пропуская все промежуточные фреймы, пока не найдётся подходящий `catch` — или пока исключение не долетит до
самого верха и не завершит поток. Ключевое отличие от кода возврата: исключение **нельзя случайно проигнорировать** —
если никто не поймал, программа явно упадёт, а не продолжит работать с мусором в данных (как во втором сценарии на
этапе 1).

**Иерархия `Throwable`:**

```
Throwable
 ├── Error                     — проблемы уровня JVM (OutOfMemoryError, StackOverflowError)
 │                                обычно не перехватываются приложением
 └── Exception
      ├── RuntimeException     — unchecked: NullPointerException, IllegalArgumentException,
      │                          IllegalStateException, NoSuchElementException
      └── остальные Exception  — checked: IOException и подобные
```

**Checked vs unchecked.** Checked-исключения (`Exception`, кроме `RuntimeException`) компилятор требует либо поймать,
либо объявить в `throws` сигнатуры метода — контракт проверяется статически. Unchecked (`RuntimeException` и его
подклассы) такого требования не несут — обычно это программные ошибки (неверный аргумент, нарушенный инвариант), от
которых вызывающий код в общем случае не ожидает восстановиться иначе, чем исправив сам вызов.

> **Замечание:** вокруг checked-исключений в Java-сообществе давняя дискуссия — многие крупные библиотеки и
> фреймворки сознательно избегают checked-исключений в своих API. Подробнее — в материале
> [«Исключения в Java»](./Исключения%20в%20Java.md).

**Когда заводить собственное исключение.** Оправдано, когда вызывающему коду важно программно отличить именно эту
ошибку от прочих (поймать по типу и отреагировать иначе) или когда нужно нести дополнительный контекст (какой именно
тикер задублирован). Если ошибка не требует особой реакции — как правило, достаточно стандартного
`IllegalArgumentException` или `IllegalStateException`, не нужно плодить типы ради самого факта наличия исключения.

**Стоимость исключений.** Коротко: создание исключения не бесплатно, в основном из-за заполнения трассировки стека
(`fillInStackTrace`). Для обычного, не горячего пути выполнения (например, обработка ошибки пользовательского ввода
раз за вызов команды) эта стоимость не имеет значения. Подробный замер — на продвинутом треке и в материале
[«Исключения в Java»](./Исключения%20в%20Java.md), раздел про стоимость исключений.

### Этап 3. Практика: собственные исключения и их обработка (~30 мин)

**3.1. Заводим `DuplicateTickerException`.**

```java
public class DuplicateTickerException extends RuntimeException {

    private final String ticker;

    public DuplicateTickerException(String ticker) {
        super("Инструмент с тикером уже существует: " + ticker);
        this.ticker = ticker;
    }

    public String getTicker() {
        return ticker;
    }
}
```

> **Обсуждение:** почему `RuntimeException`, а не checked? В CLI-приложении единственная разумная реакция на дубликат
> — сообщить пользователю и попросить исправить ввод, той же реакции ждём от `IllegalArgumentException` при
> неизвестном типе инструмента. Принудительный `throws DuplicateTickerException` в сигнатуре `add` и всех вызывающих
> его методов по цепочке добавил бы шума, не добавив ценности.

**3.2. `InstrumentRepository.add` проверяет дубликаты.**

```java
public void add(Instrument instrument) {
    if (findByTicker(instrument.getTicker()).isPresent()) {
        throw new DuplicateTickerException(instrument.getTicker());
    }
    instruments[size++] = instrument;
}
```

Тихая поломка из этапа 1 стала громкой — и теперь можно её обработать осознанно.

**3.3. Обработка в `AddCommand`.** Нужно поймать сразу два случая: `DuplicateTickerException` из шага 3.2 и
`IllegalArgumentException`, который `InstrumentFactories.create` уже бросает на неизвестном типе (этап 1) — сегодня
это исключение впервые кто-то ловит:

```java
@Override
public Integer call() {
    try {
        Instrument instrument = InstrumentFactories.create(type, ticker, name, currency);
        repository.add(instrument);
        System.out.println("Добавлено: " + instrument.getDescription());
        return 0;
    } catch (DuplicateTickerException e) {
        System.out.println("Ошибка: " + e.getMessage());
        return 1;
    } catch (IllegalArgumentException e) {
        System.out.println("Некорректные данные: " + e.getMessage());
        return 1;
    }
}
```

Оба `catch`-блока сейчас делают одно и то же — печатают сообщение и возвращают `1`. Если реакция действительно
одинаковая, уместнее multi-catch — один блок на несколько типов через `|`:

```java
try {
    Instrument instrument = InstrumentFactories.create(type, ticker, name, currency);
    repository.add(instrument);
    System.out.println("Добавлено: " + instrument.getDescription());
    return 0;
} catch (DuplicateTickerException | IllegalArgumentException e) {
    System.out.println("Не удалось добавить инструмент: " + e.getMessage());
    return 1;
}
```

> **Когда не сворачивать в multi-catch:** если разным типам ошибок нужна разная реакция (разные сообщения, разные
> коды завершения, разное логирование) — отдельные `catch`-блоки читаются понятнее, чем один блок с проверками
> `instanceof` внутри.

**3.4. Исключения с причиной (exception chaining).** На семинаре 7 у репозитория появится реальный источник данных
(файл, сеть), и там будут свои низкоуровневые ошибки — например, `IOException`. Оборачивать такую ошибку в доменное
исключение нужно, не теряя первоначальную причину:

```java
public class InstrumentLoadException extends RuntimeException {
    public InstrumentLoadException(String message, Throwable cause) {
        super(message, cause);
    }
}
```

```java
void example() {
    try {
        // низкоуровневая операция, способная бросить IOException
    } catch (IOException e) {
        throw new InstrumentLoadException("Не удалось загрузить инструмент", e);
    }
}
```

`getCause()` возвращает исходное исключение, а `printStackTrace()` печатает обе трассировки, вторую — с пометкой
`Caused by:`. Антипаттерн, которого стоит избегать: `catch (IOException e) { throw new RuntimeException("error"); }` —
без передачи `e` как `cause` исходная причина теряется безвозвратно, и при отладке видна только верхняя,
малоинформативная ошибка.

### Этап 4. try-with-resources (~15 мин)

Презентация по материалу [«Исключения в Java»](./Исключения%20в%20Java.md), раздел про `try-with-resources` и
`AutoCloseable`.

В проекте пока нет настоящих ресурсов — файлы и сеть появятся на семинаре 7, — но познакомиться стоит заранее.
`AutoCloseable` — интерфейс с единственным методом `close()`; `try-with-resources` гарантирует его вызов при выходе из
блока, даже если внутри было брошено исключение:

```java
public class InstrumentImportSession implements AutoCloseable {

    public InstrumentImportSession() {
        System.out.println("Сессия импорта открыта");
    }

    public void importLine(String line) {
        System.out.println("Импортируем: " + line);
    }

    @Override
    public void close() {
        System.out.println("Сессия импорта закрыта");
    }
}
```

```java
try (InstrumentImportSession session = new InstrumentImportSession()) {
    session.importLine("STOCK,SBER,Сбербанк,250");
} // close() вызывается автоматически — в том числе если importLine бросит исключение
```

Показываем, что это синтаксический сахар над `try/finally`, который пришлось бы писать руками:

```java
void example() {
    InstrumentImportSession session = new InstrumentImportSession();
    try {
        session.importLine("STOCK,SBER,Сбербанк,250");
    } finally {
        session.close();
    }
}
```

Разница не только в компактности: ручной `finally` легко написать с ошибкой (забыть его вовсе, закрыть не тот
ресурс), а `try-with-resources` синтаксически не позволяет забыть про `close()`.
