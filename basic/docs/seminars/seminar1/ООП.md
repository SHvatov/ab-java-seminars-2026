# Объектно-ориентированное программирование

---

## 1. Что такое ООП?

**ООП** — парадигма проектирования, в центре которой стоят **объекты**: сущности, объединяющие *состояние* (данные) и
*поведение* (методы, работающие с этими данными).

Три принципа, на которых сосредоточен этот материал:

- **Инкапсуляция** — сокрытие реализации от внешнего пользователя
- **Наследование** — переиспользование кода и построение иерархий
- **Полиморфизм** — работа с разными типами через единый интерфейс

**Ключевая идея:** эти принципы **не существуют по отдельности**. Они дополняют друг друга, и без любого из них теряется
смысл остальных.

---

## 2. Инкапсуляция - это не про геттеры и сеттеры

Самое частое заблуждение — что инкапсуляция это «сделать поля приватными и написать `get`/`set`». На самом деле суть
глубже: **инкапсуляция — это сокрытие деталей реализации за контрактом**.

Объект выставляет наружу:

- **что** он делает (сигнатуры методов — контракт);

и скрывает:

- **как** именно это реализовано (внутреннее устройство и состояние).

### Пример из Си — там, где нет классов

Инкапсуляция — не привилегия объектно-ориентированных языков. В качестве примера — системный вызов из стандартной
библиотеки Си:

```other
// fcntl.h — наружу торчит только контракт
int open(const char *pathname, int flags);
```

Видно **определение процедуры** — её сигнатуру. Это всё, что нужно, чтобы ей пользоваться. Реализация спрятана внутри
библиотеки и операционной системы, и в разных системах она **разная**:

```other
int open(const char *, int)      ← один контракт
                    │
     ┌──────────────┼──────────────┐
     ▼              ▼              ▼
   Linux          macOS         Windows
 ext4 / tmpfs   HFS+ / APFS       NTFS
```

В разных Unix-подобных системах под капотом совершенно разные реализации работы с файловой системой. Но для пользователя
это не имеет значения:

```other
int fd = open("file.txt", O_RDONLY);  // работает одинаково везде
```

> **Суть инкапсуляции:** наружу выставляется контракт, которым можно пользоваться, но детали реализации скрыты от
> конечного пользователя. Реализацию `open` можно переписать целиком — клиентский код этого не заметит.

### Почему это важнее, чем кажется

Хорошая инкапсуляция означает, что объект **сам отвечает за корректность своего состояния**:

```java
public class Stock {
    private final String ticker;  // скрыто

    public Stock(String ticker) {
        if (ticker == null || ticker.isBlank()) {
            throw new IllegalArgumentException("Ticker required");
        }
        this.ticker = ticker;     // инвариант защищён в момент создания
    }

    public String getTicker() {
        return ticker;            // только чтение, изменить снаружи нельзя
    }
}
```

Класс гарантирует, что `Stock` с пустым тикером просто не может существовать. Это и есть защита инвариантов — а не
формальные геттеры со сеттерами.

## 3. Наследование

### В чём суть

**Наследование** устанавливает отношение «является» (IS-A) и позволяет переиспользовать код. Базовый класс описывает
общее, подклассы — специфичное:

```other
              ┌─────────────┐
              │ Instrument  │   ticker, name, getDescription()
              └──────┬──────┘
          ┌──────────┼──────────┐
          ▼          ▼          ▼
      ┌───────┐  ┌──────┐   ┌─────┐
      │ Stock │  │ Bond │   │ Etf │
      └───────┘  └──────┘   └─────┘
       sector     coupon     index
```

**Основные возможности и области применения:**

- переиспользование полей и методов базового класса;
- переопределение (override) поведения в подклассах;
- построение иерархий типов, отражающих предметную область.

### Но наследование может всё сломать

Наследование создаёт **жёсткую связанность** между родителем и потомками. Чем сложнее иерархия, тем опаснее: любое
изменение в базовом классе **каскадно** расходится вниз по всем потомкам.

```other
          Instrument
        /     |      \           ← изменение в корне...
      ...    ...     ...
     /  \    /  \    /  \        ← ...протекает во все подклассы,
   ...  ... ... ... ... ...        конструкторы, equals/hashCode, toString
```

Добавление поля или изменение логики метода наверху требует проверки *всех* потомков. В глубоких иерархиях это
превращается в серьёзную проблему: всё начинает сыпаться от одного изменения.

### SOLID: Interface Segregation Principle (ISP)

Здесь же стоит отметить один из принципов SOLID — **Interface Segregation Principle**. Если подкласс вынужден
наследовать методы, которые ему не нужны (или для которых он не может дать осмысленную реализацию), — это сигнал, что
иерархия спроектирована неверно. Подробно SOLID разбирается на последующих семинарах, но эту связь полезно держать в
голове уже сейчас.

### Альтернатива — ассоциация и композиция

Не всегда нужно наследоваться. Часто проблемы наследования снимаются простым **объединением объектов**, а не
выстраиванием иерархии.

**Агрегация** — объект ссылается на другой, но тот живёт независимо:

```java
public class Quote {
    private final Instrument instrument;  // агрегация: инструмент сам по себе
    private final BigDecimal price;
}
// Один и тот же Stock может быть в десятках котировок.
// Удаление котировки не затрагивает инструмент.
```

**Композиция** — более сильная форма владения, «часть-целое»:

```java
public class Portfolio {
    public static class Position { /* ... */
    }     // живёт только внутри Portfolio

    private final List<Position> positions = new ArrayList<>();
}
// Удаление портфеля удаляет и его позиции.
```

> **Правило большого пальца:** предпочитать композицию наследованию. Наследоваться стоит только тогда, когда отношение
> IS-A действительно есть, а иерархия остаётся простой.

## 4. Полиморфизм

**Полиморфизм** — способность работать с объектами разных типов через общий интерфейс. Здесь рассматриваются два
основных вида.

### Полиморфизм подтипов (subtype, runtime)

Переопределение методов в подклассах. Один и тот же код вызывает правильную реализацию в зависимости от конкретного типа
объекта:

```java
void example() {
    List<Instrument> instruments = List.of(new Stock(), new Bond(), new Etf());
    for (Instrument instr : instruments) {
        System.out.println(instr.getDescription());  // вызовется метод конкретного подкласса
    }
}
```

Какой именно `getDescription()` выполнится, решается **в runtime** — JVM смотрит на фактический тип объекта.

### Перегрузка (overloading, compile-time)

Несколько методов с одним именем, но разными параметрами. Какой вызвать, решает **компилятор** по типам аргументов:

```java
int add(int a, int b) {
    return a + b;
}

double add(double a, double b) {
    return a + b;
}

String add(String a, String b) {
    return a + b;
}
```

### Шаблонный полиморфизм

Существует ещё **параметрический (шаблонный) полиморфизм** — это generics в Java. Он будет рассмотрен позднее, в
последующих семинарах.

## 5. Как принципы работают вместе

Самое важное. Три принципа — не независимый список, а **взаимосвязанная система**, где каждый держит остальные.

### Полиморфизм подтипов невозможен без наследования

Полиморфизм подтипов *буквально* построен на иерархии типов. Без наследования (или реализации общего интерфейса) нет
общего типа, через который можно единообразно обращаться к разным объектам:

```java
void example() {
    for (Instrument instr : instruments) {
        // работает только потому, что Stock, Bond, Etf — это Instrument
        System.out.println(instr.getDescription());
    }
}
```

Без наследования `instruments` распадается на несвязанные типы, и обойти их единым циклом становится нечем.

### Наследование без полиморфизма почти бессмысленно

Обратное тоже верно. Без полиморфизма наследование сводится к простому копированию полей и методов от родителя.
Переиспользование кода — это польза, но без возможности переопределять поведение и обращаться к потомкам через
родительский тип наследование теряет бо́льшую часть своей силы.

### Инкапсуляция — фундамент, без которого всё "течёт"

Инкапсуляция — краеугольный камень. Без сокрытия реализации внутреннее состояние объектов торчит наружу, любой может его
испортить, и тогда ни наследование, ни полиморфизм не спасают — иерархия строится поверх объектов, которым нельзя
доверять.

```other
   ИНКАПСУЛЯЦИЯ   →   скрыли реализацию, защитили инварианты
        │
        ▼
   НАСЛЕДОВАНИЕ   →   строим иерархии поверх надёжных объектов
        │
        ▼
   ПОЛИМОРФИЗМ    →   работаем со всей иерархией через единый тип
```

Без любого слоя конструкция рассыпается. Именно в этой взаимосвязи и заключается ООП, а не в трёх определениях по
отдельности.
