### Этап 2. Теория: ООП (~15 мин)

Презентация на 10–12 слайдов. Цель — не пересказать учебник, а показать **зачем** и **как принципы работают вместе**.

**Ключевые тезисы:**

- **Объект** — это сущность с состоянием и поведением. Объекты общаются через сообщения (вызовы методов).
- **Инкапсуляция** — не просто геттеры и сеттеры. Это про сокрытие деталей реализации и защиту инвариантов объекта.
  Класс сам отвечает за то, чтобы его состояние всегда было корректным.
- **Наследование** — механизм повторного использования и установления отношения «является». Имеет цену: сильная
  связность. Альтернативы — композиция и делегирование.
- **Полиморфизм** — возможность работать с разными типами через общий интерфейс. Сегодня рассматриваем два вида: *
  *полиморфизм подтипов** (переопределение методов) и **перегрузку** (overloading).

> **Важно:** показать, как принципы *дополняют* друг друга — инкапсуляция защищает состояние, наследование строит
> иерархии, полиморфизм позволяет писать обобщённый код.

### Этап 4. Совместная реализация кода (~40 мин)

Пишем код вместе, объясняя решения по ходу. Студенты повторяют у себя.

Этап разбит на четыре шага: сначала делаем наивную реализацию через enum и static-метод, затем рефакторим её в
полиморфизм. Потом смотрим, где наследование ломается, и разбираем агрегацию vs композицию.

#### 4.1. Вспомогательные типы

```java

public enum Currency {
```

#### 4.2. Базовый класс инструмента — инкапсуляция

Обсуждаем: почему поля `private`? Почему нет сеттеров? Ticker не должен меняться — это инвариант.

```java
// model/Instrument.java
public abstract class Instrument {

    private final String ticker;
    private final String name;
    private final Currency currency;

    public Instrument(String ticker, String name, Currency currency) {
        if (ticker == null || ticker.isBlank()) {
            throw new IllegalArgumentException("Ticker cannot be blank");
        }
        this.ticker = ticker;
        this.name = name;
        this.currency = currency;
    }

    public String getTicker() {
        return ticker;
    }

    public String getName() {
        return name;
    }

    public Currency getCurrency() {
        return currency;
    }

    public abstract String getDescription();

    @Override
    public String toString() {
        return ticker + " — " + name + " (" + getDescription() + ")";
    }
}
```

> **Акцент на инкапсуляцию:** инициализируем через конструктор, валидируем там же. Объект не может быть создан в
> некорректном состоянии.

#### 4.3. Наивная реализация — enum + static-метод

Пишем первую версию подклассов без переопределения `getDescription()`, а описание получаем через статический метод с
`switch` по типу инструмента. Это работает, но сразу видны проблемы.

```java
// model/InstrumentType.java — временно, потом удалим
public enum InstrumentType {
    STOCK,
    BOND,
    ETF
}
```

```java
// model/Stock.java (первая версия)
public class Stock extends Instrument {

    private final String sector;

    public Stock(String ticker, String name, Currency currency, String sector) {
        super(ticker, name, currency);
        this.sector = sector;
    }

    public String getSector() {
        return sector;
    }

    @Override
    public String getDescription() {
        // Пока оставим пустым — заполним через static-метод ниже
        return "";
    }
}
```

```java
// util/InstrumentFormatter.java — наивный подход
public class InstrumentFormatter {

    // Чтобы получить описание, нам нужно знать тип — используем instanceof
    public static String describe(Instrument instrument) {
        if (instrument instanceof Stock stock) {
            return "Акция, сектор: " + stock.getSector();
        } else if (instrument instanceof Bond bond) {
            return "Облигация, купон: " + bond.getCouponRate() + "%, погашение: " + bond.getMaturityYear();
        } else if (instrument instanceof Etf etf) {
            return "ETF, индекс: " + etf.getTrackingIndex();
        }
        return "Неизвестный инструмент";
    }
}
```

> **Обсуждение:** что не так с таким подходом? Добавим новый тип инструмента — нужно найти и обновить все такие `if`
> -цепочки по всей кодовой базе. Компилятор об этом не предупредит.

#### 4.4. Рефакторинг — полиморфизм вместо switch

Удаляем `InstrumentFormatter`, удаляем `InstrumentType`. Переносим логику описания в каждый подкласс — туда, где про
него всё известно.

```java
// model/Stock.java
public class Stock extends Instrument {

    private final String sector;

    public Stock(String ticker, String name, Currency currency, String sector) {
        super(ticker, name, currency);
        this.sector = sector;
    }

    public String getSector() {
        return sector;
    }

    @Override
    public String getDescription() {
        return "Акция, сектор: " + sector;
    }
}
```

```java
// model/Bond.java
public class Bond extends Instrument {

    private final double couponRate;
    private final int maturityYear;

    public Bond(String ticker, String name, Currency currency,
                double couponRate, int maturityYear) {
        super(ticker, name, currency);
        this.couponRate = couponRate;
        this.maturityYear = maturityYear;
    }

    public double getCouponRate() {
        return couponRate;
    }

    public int getMaturityYear() {
        return maturityYear;
    }

    @Override
    public String getDescription() {
        return "Облигация, купон: " + couponRate + "%, погашение: " + maturityYear;
    }
}
```

```java
// model/Etf.java
public class Etf extends Instrument {

    private final String trackingIndex;

    public Etf(String ticker, String name, Currency currency, String trackingIndex) {
        super(ticker, name, currency);
        this.trackingIndex = trackingIndex;
    }

    public String getTrackingIndex() {
        return trackingIndex;
    }

    @Override
    public String getDescription() {
        return "ETF, отслеживает индекс: " + trackingIndex;
    }
}
```

> **Акцент:** теперь добавление нового типа инструмента — это создание нового класса. Существующий код трогать не нужно.
> Компилятор проследит, чтобы `getDescription()` был реализован — `Instrument` абстрактный.

#### 4.5. Почему double не подходит для денег

Прежде чем двигаться дальше, показываем студентам один фрагмент:

```java
void example() {
    System.out.println(0.1 + 0.2); // 0.30000000000000004
}
```

`double` — тип с плавающей точкой, он не может точно представить большинство десятичных дробей. Для цен и финансовых
расчётов это недопустимо: ошибки накапливаются, округления расходятся. Правильный инструмент — `BigDecimal`: он хранит
число точно, как строку цифр с заданной точностью.

> **Акцент:** `BigDecimal` создаётся через `new BigDecimal("278.50")`, а не `new BigDecimal(278.50)` — конструктор от
`double` унаследует его неточность. Везде, где в нашей модели появится цена или доходность, используем `BigDecimal`.

#### 4.6. Когда наследование ломается — нарушение LSP через getDividends

Хотим посчитать годовую дивидендную доходность для любого инструмента из портфеля. Соблазн — вынести
`getDividends(BigDecimal currentPrice)` в базовый класс.

```java
// model/Instrument.java — плохая идея
public abstract class Instrument {
    // ...

    // Хотим считать дивиденды для всех инструментов сразу
    public abstract BigDecimal getDividends(BigDecimal currentPrice);
}
```

Смотрим, что происходит в подклассах:

```java
// model/Stock.java — всё честно
// dividendYield — процент годовой дивидендной доходности, например 6.5
private final BigDecimal dividendYield;

@Override
public BigDecimal getDividends(BigDecimal currentPrice) {
    // Годовая дивидендная доходность: цена × доходность / 100
    return currentPrice.multiply(dividendYield)
            .divide(BigDecimal.valueOf(100));
}
```

```java
// model/Bond.java — уже проблема
@Override
public BigDecimal getDividends(BigDecimal currentPrice) {
    // У облигации нет дивидендов. Что вернуть?
    return BigDecimal.ZERO;  // Вариант 1: тихо врём
    // throw new UnsupportedOperationException(); // Вариант 2: ломаем контракт
}
```

```java
// model/Etf.java — та же проблема
@Override
public BigDecimal getDividends(BigDecimal currentPrice) {
    // ETF обычно реинвестирует доход, а не выплачивает дивиденды
    return BigDecimal.ZERO; // Снова тихая ложь
}
```

> **Обсуждение:** оба варианта плохие. `BigDecimal.ZERO` — молчаливо искажает расчёты. `UnsupportedOperationException` —
> нарушает LSP: код, работающий с `List<Instrument>`, не может безопасно вызвать метод, не проверяя конкретный тип. Это
> сигнал, что `getDividends` не принадлежит базовому классу — он специфичен только для `Stock`. Правильное решение:
> убрать
> метод из `Instrument`, оставить только в `Stock`.

#### 4.7. Агрегация vs композиция

#### Шаг 1. Попытка через наследование — StockSnapshot

Хотим представить акцию с актуальной ценой на конкретный момент времени. Первый порыв — отнаследоваться от `Stock`,
добавить поле `price` и перегрузить `getDividends()` без аргумента, раз цена уже внутри:

```java
// model/StockSnapshot.java — попытка через наследование
public class StockSnapshot extends Stock {

    private final BigDecimal price;

    public StockSnapshot(String ticker, String name, Currency currency,
                         String sector, BigDecimal dividendYield, BigDecimal price) {
        super(ticker, name, currency, sector, dividendYield);
        this.price = price;
    }

    // Перегружаем — удобно, цена уже вшита
    public BigDecimal getDividends() {
        return getDividends(this.price);
    }
}
```

Выглядит аккуратно. Но тут компилятор напоминает: `getDividends(BigDecimal price)` из `Instrument` никуда не делся. Раз
`StockSnapshot` конкретный класс — его тоже нужно реализовать. И вот здесь начинается проблема:

```java
// Что писать сюда?
@Override
public BigDecimal getDividends(BigDecimal currentPrice) {
    // Вариант 1: игнорируем аргумент — зачем тогда он вообще есть?
    return getDividends();
    // Вариант 2: используем аргумент — тогда зачем нам this.price?
}
```

> **Обсуждение:** оба варианта плохие. В первом — метод молча игнорирует то, что ему передали; потребитель типа `Stock`
> передаст одну цену, получит результат по другой. Во втором — `this.price` бессмысленен. Поведение `getDividends`
> непредсказуемо в зависимости от того, через какой тип смотрит вызывающий код — это нарушение LSP. Плюс: один и тот же
`SBER` на двух биржах по разным ценам — это два разных объекта `StockSnapshot`? Что-то здесь не то.

#### Шаг 2. Переименовываем и меняем подход — Quote как агрегация

Проблема в том, что `StockSnapshot` — это не вид `Stock`. Это `Stock` плюс рыночная цена в момент времени.
Переименовываем класс в `Quote` и убираем наследование: `Quote` хранит ссылку на `Stock`, но не является им.

```java
// model/Quote.java — агрегация вместо наследования

import java.math.BigDecimal;

public class Quote {

    private final Instrument instrument;
    private final BigDecimal price;
    private final BigDecimal changePercent;

    public Quote(Instrument instrument, BigDecimal price, BigDecimal changePercent) {
        this.instrument = instrument;
        this.price = price;
        this.changePercent = changePercent;
    }

    public Instrument getInstrument() {
        return instrument;
    }

    public BigDecimal getPrice() {
        return price;
    }

    public BigDecimal getChangePercent() {
        return changePercent;
    }

    // Теперь getDividends() здесь честный: цена известна, делегируем в Stock
    public BigDecimal getDividends() {
        if (instrument instanceof Stock stock) {
            return stock.getDividends(this.price);
        }
        return BigDecimal.ZERO;
    }

    @Override
    public String toString() {
        String direction = changePercent.signum() >= 0 ? "▲" : "▼";
        return instrument.getTicker() + ": " + price + " " + instrument.getCurrency()
                + " " + direction + " " + changePercent.abs() + "%";
    }
}
```

> **Акцент:** один и тот же `sber` теперь может участвовать в любом количестве котировок — на разных биржах, в разные
> моменты времени — и оставаться единственным объектом. `getDividends()` в `Quote` корректен: цена известна, метод
> делегирует в `Stock` и возвращает `ZERO` для инструментов без дивидендов — без нарушения контрактов.

#### Шаг 3. Композиция для контраста — Portfolio и Position

Для контраста показываем другой тип отношения. `Portfolio` владеет `Position` — позиция не имеет смысла в отрыве от
портфеля и создаётся только внутри него:

```java
// model/Portfolio.java
public class Portfolio {

    // Position — внутренняя сущность, снаружи её не создать
    public static class Position {
        private final Instrument instrument;
        private final int quantity;

        private Position(Instrument instrument, int quantity) {
            this.instrument = instrument;
            this.quantity = quantity;
        }

        public Instrument getInstrument() {
            return instrument;
        }

        public int getQuantity() {
            return quantity;
        }
    }

    private final String name;
    private final List<Position> positions = new ArrayList<>();

    public Portfolio(String name) {
        this.name = name;
    }

    public void addPosition(Instrument instrument, int quantity) {
        positions.add(new Position(instrument, quantity));
    }

    public List<Position> getPositions() {
        return Collections.unmodifiableList(positions);
    }

    public String getName() {
        return name;
    }
}
```

> **Акцент:** разница между агрегацией и композицией — в семантике владения. Спросить студентов: что произойдёт с
> позициями, если портфель удалён? А с инструментом, если котировка удалена? Ответы разные — и это и есть разница.
