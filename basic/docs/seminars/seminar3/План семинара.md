### Этап 1. Мотивация: боль от Strategy на классах (~10 мин)

Открываем проект в состоянии, в котором он остался после семинара 2: `InstrumentFilter`, `TypeFilter`, `NoOpFilter` — три
файла ради одного условия фильтрации.

**Вопрос группе:** что если пользователю понадобится фильтр по валюте инструмента (`--currency`)? А если понадобится
фильтр «тип **и** валюта одновременно»?

Ответ по ходу обсуждения — ещё один класс `CurrencyFilter`, и либо ещё один класс `TypeAndCurrencyFilter`, либо ручная
комбинация условий внутри команды, что убивает весь смысл Strategy из семинара 2.

> **Акцент:** сама идея Strategy — правильная (алгоритм отбора инкапсулирован, взаимозаменяем), но механизм реализации
> — по классу на каждую стратегию — не масштабируется. Сегодня меняем механизм, не идею.

### Этап 2. Теория: функциональные интерфейсы и лямбда-выражения (~15 мин)

Презентация по материалу
[«Функциональные интерфейсы и лямбда-выражения»](./Функциональные%20интерфейсы%20и%20лямбда-выражения.md): SAM,
`@FunctionalInterface`, синтаксис лямбды, захват переменных и effectively final, функции высшего порядка,
`java.util.function`.

Смотрим на `InstrumentFilter` из семинара 2 другими глазами — один абстрактный метод, значит уже функциональный
интерфейс, просто не использовали это свойство:

```java
public interface InstrumentFilter {
    boolean matches(Instrument instrument);
}
```

`InstrumentFilter` структурно совпадает с `Predicate<Instrument>` — сигнатура `boolean test(Instrument)` вместо
`boolean matches(Instrument)`. На этапе 3 заменяем собственный интерфейс на стандартный.

---

### Этап 3. Практика: Strategy → лямбды, композиция, Comparator (~20 мин)

**3.1. Замена `InstrumentFilter` на `Predicate<Instrument>`.** Удаляем `InstrumentFilter`, `TypeFilter`, `NoOpFilter`
целиком — три файла больше не нужны.

```java
@Command(name = "list", description = "Список инструментов")
public class ListCommand implements Callable<Integer> {

    @Option(names = "--type", description = "Фильтр по типу инструмента")
    private String type;

    private final InstrumentRepository repository;

    public ListCommand(InstrumentRepository repository) {
        this.repository = repository;
    }

    @Override
    public Integer call() {
        Predicate<Instrument> filter = (type == null)
                ? instrument -> true
                : instrument -> instrument.getClass().getSimpleName().equalsIgnoreCase(type);

        for (Instrument instrument : repository) {
            if (filter.test(instrument)) {
                System.out.println(instrument.getDescription());
            }
        }
        return 0;
    }
}
```

> **Обсуждение:** паттерн Strategy никуда не делся — алгоритм отбора по-прежнему инкапсулирован и подменяем. Изменился
> только способ его выразить: вместо класса — значение-функция.

**3.2. Композиция предикатов.** Добавляем опцию `--currency` и комбинируем условия без единого `if`:

```java
@Option(names = "--currency", description = "Фильтр по валюте инструмента (RUB, USD, EUR)")
private Currency currency;

@Override
public Integer call() {
    Predicate<Instrument> filter = (type == null)
            ? instrument -> true
            : instrument -> instrument.getClass().getSimpleName().equalsIgnoreCase(type);

    if (currency != null) {
        filter = filter.and(instrument -> instrument.getCurrency() == currency);
    }

    for (Instrument instrument : repository) {
        if (filter.test(instrument)) {
            System.out.println(instrument.getDescription());
        }
    }
    return 0;
}
```

`Predicate` — функциональный интерфейс, но у него есть `default`-методы `and`, `or`, `negate`, которые сами возвращают
новый `Predicate`. Это и есть функция высшего порядка на практике: `and()` принимает предикат и возвращает предикат.

**3.3. Сортировка: `Comparator` через лямбду и метод-ссылку.** Добавляем `--sort-by-ticker`:

```java
@Option(names = "--sort-by-ticker", description = "Сортировать по тикеру")
private boolean sortByTicker;

@Override
public Integer call() {
    List<Instrument> matched = new ArrayList<>();
    for (Instrument instrument : repository) {
        if (filter.test(instrument)) {
            matched.add(instrument);
        }
    }

    if (sortByTicker) {
        matched.sort((a, b) -> a.getTicker().compareTo(b.getTicker()));
    }

    matched.forEach(instrument -> System.out.println(instrument.getDescription()));
    return 0;
}
```

`Comparator<T>` — тоже функциональный интерфейс (`int compare(T o1, T o2)`), поэтому лямбда подходит напрямую. То же
самое можно записать через `Comparator.comparing` и **метод-ссылку** — ссылку на уже существующий метод вместо новой
лямбды:

```java
matched.sort(Comparator.comparing(Instrument::getTicker));
```

`Instrument::getTicker` — синтаксический сахар над `instrument -> instrument.getTicker()`. Метод-ссылка уместна всегда,
когда тело лямбды — это просто вызов одного существующего метода.

### Этап 4. Теория: Generics как вид полиморфизма (~15 мин)

Презентация по материалу [«Generics»](./Generics.md): мотивация, собственный generic-класс, bounded type parameter
(`extends`), ключевое слово `super`.

На семинаре 1 разбирали полиморфизм подтипов (`Instrument` → `Stock`/`Bond`/`Etf`) и перегрузку. Generics — третий вид:
**параметрический полиморфизм**. Код пишется один раз и работает с любым типом, который подставят в качестве
параметра. `Iterator<Instrument>` и `Iterable<Instrument>` из семинара 2, `Predicate<Instrument>` и
`Comparator<Instrument>` с этого семинара — всё это generic-интерфейсы, которыми уже пользовались.

### Этап 5. Практика: обобщённый `Registry<K, V>` и bounded generics (~15 мин)

**5.1. Обобщаем `InstrumentFactories`.** В семинаре 2 реестр фабрик был жёстко завязан на `String` и `InstrumentFactory`:

```java
public final class InstrumentFactories {
    private static final Map<String, InstrumentFactory> REGISTRY = new HashMap<>();
    // ...
}
```

Сама идея — «регистрируем значение по ключу, потом достаём» — универсальна и не имеет отношения к инструментам.
Выносим её в обобщённый класс:

```java
public class Registry<K, V> {

    private final Map<K, V> items = new HashMap<>();

    public void register(K key, V value) {
        items.put(key, value);
    }

    public V get(K key) {
        return items.get(key);
    }
}
```

`InstrumentFactories` теперь — конкретное применение `Registry<String, InstrumentFactory>`, а не отдельная реализация
с нуля:

```java
public final class InstrumentFactories {
    private static final Registry<String, InstrumentFactory> REGISTRY = new Registry<>();

    private InstrumentFactories() {
    }

    public static void register(String type, InstrumentFactory factory) {
        REGISTRY.register(type.toUpperCase(), factory);
    }

    public static Instrument create(String type, String ticker, String name, Currency currency) {
        InstrumentFactory factory = REGISTRY.get(type.toUpperCase());
        if (factory == null) {
            throw new IllegalArgumentException("Unknown instrument type: " + type);
        }
        return factory.create(ticker, name, currency);
    }
}
```

> **Обсуждение:** `Registry<K, V>` можно переиспользовать где угодно — например, для реестра команд picocli или кеша
> котировок. Это и есть практическая польза generics: код пишется один раз для «любого K и V», а не копируется под
> каждую пару типов.

**5.2. Обобщённый метод с bounded type parameter.** Практическая задача поверх домена — первые N инструментов по
алфавиту тикера, метод должен работать с любым подтипом `Instrument` (и с `List<Stock>`, и с `List<Instrument>`):

```java
public final class InstrumentUtils {

    private InstrumentUtils() {
    }

    public static <T extends Instrument> List<T> topByTicker(List<T> items, int n) {
        List<T> sorted = new ArrayList<>(items);
        sorted.sort(Comparator.comparing(Instrument::getTicker));
        return sorted.subList(0, Math.min(n, sorted.size()));
    }
}
```

Подключаем метод к `ListCommand` — добавляем опцию `--top`, показывающую только первые N инструментов из уже
отфильтрованного списка:

```java
@Option(names = "--top", description = "Показать только первые N инструментов по алфавиту тикера")
private Integer top;

@Override
public Integer call() {
    List<Instrument> matched = new ArrayList<>();
    for (Instrument instrument : repository) {
        if (filter.test(instrument)) {
            matched.add(instrument);
        }
    }

    if (sortByTicker) {
        matched.sort(Comparator.comparing(Instrument::getTicker));
    }

    List<Instrument> result = (top != null)
            ? InstrumentUtils.topByTicker(matched, top)
            : matched;

    result.forEach(instrument -> System.out.println(instrument.getDescription()));
    return 0;
}
```

Компилятор выводит `T` из фактического типа списка `matched` (`List<Instrument>`), а `extends Instrument` внутри
`topByTicker` даёт доступ к `getTicker()` — вызов `market-pulse list --type STOCK --top 3` печатает три акции с самыми
ранними по алфавиту тикерами из отфильтрованных.

### Этап 6. Optional: прощаемся с `null` (~10 мин)

Презентация по материалу [«Optional»](./Optional.md): проблема, которую решает `Optional`, базовый API, где `Optional`
неуместен.

`InstrumentRepository.findByTicker` из семинара 2 возвращает `null`, если инструмент не найден, — классический источник
`NullPointerException` для тех, кто забудет проверить результат. Переводим на `Optional<Instrument>`:

```java
public Optional<Instrument> findByTicker(String ticker) {
    for (Instrument instrument : this) {
        if (instrument.getTicker().equalsIgnoreCase(ticker)) {
            return Optional.of(instrument);
        }
    }
    return Optional.empty();
}
```

Рефакторим `SearchCommand`:

```java
@Override
public Integer call() {
    return repository.findByTicker(ticker)
            .map(instrument -> {
                System.out.println(instrument.getDescription());
                return 0;
            })
            .orElseGet(() -> {
                System.out.println("Инструмент не найден: " + ticker);
                return 1;
            });
}
```

То же самое проделываем с `Registry.get()`:

```java
public Optional<V> get(K key) {
    return Optional.ofNullable(items.get(key));
}
```

`InstrumentFactories.create()` из этапа 5 обращался к `REGISTRY.get(...)` через проверку на `null` — после перехода
`Registry` на `Optional` этот код перестанет компилироваться, если его не обновить вместе с `Registry`:

```java
public static Instrument create(String type, String ticker, String name, Currency currency) {
    return REGISTRY.get(type.toUpperCase())
            .map(factory -> factory.create(ticker, name, currency))
            .orElseThrow(() -> new IllegalArgumentException("Unknown instrument type: " + type));
}
```

### Этап 7. Итоги (~5 мин)

Что изменилось в проекте за сегодня: три файла (`InstrumentFilter`, `TypeFilter`, `NoOpFilter`) исчезли и заменены
лямбдами; фильтрация умеет комбинироваться через `and`; сортировка появилась без единого нового класса;
`InstrumentFactories` стал частным случаем переиспользуемого `Registry<K, V>`; `findByTicker` и `Registry.get()` больше
не возвращают `null`.

Следующий семинар — корректная обработка ошибок и исключительных ситуаций: то, что сегодня в `AddCommand` тихо падает
на некорректном вводе валюты, получит явную обработку.
