## 1. До исключений: коды возврата и sentinel values

Ранние языки программирования не имели механизма исключений — единственным способом сообщить об ошибке было вернуть
из функции специальное значение, зарезервированное под «что-то пошло не так»: `-1` для функций, возвращающих
неотрицательное число, `NULL` для функций, возвращающих указатель, или отдельную глобальную переменную-код ошибки,
которую нужно проверить сразу после вызова.

Классический пример — `errno` в C: системные вызовы вроде `open()` возвращают `-1` при ошибке, а конкретную причину
нужно отдельно посмотреть в глобальной переменной `errno` сразу после вызова, потому что следующий же системный вызов
может её перезаписать.

```
int fd = open("data.txt", O_RDONLY);
if (fd == -1) {
    perror("open failed"); // errno нужно проверить прямо сейчас, до следующего вызова
    return -1;
}
```

У этого подхода есть реальные достоинства: он явный (весь поток управления виден прямо в коде, никаких скрытых
переходов), предсказуемый по стоимости (возврат значения — одна из самых дешёвых операций, которые вообще есть в
языке) и не требует никакой поддержки со стороны рантайма.

Но у него же есть фундаментальный недостаток: проверку результата ничто не гарантирует. Компилятор C не заставит
проверить возвращаемое значение `open()` — если разработчик забудет `if`, ошибка тихо проигнорируется, и программа
продолжит работать с невалидным `fd`, что рано или поздно проявится совершенно в другом, никак не связанном месте
кода. Чем длиннее цепочка вызовов, тем больше точек, где проверку можно забыть, и тем труднее локализовать
первопричину, когда что-то в итоге ломается.

Java, при всей объектной ориентированности, не свободна от того же паттерна там, где исключения исторически не
использовались: `String.indexOf` возвращает `-1`, если подстрока не найдена; `Map.get` возвращает `null`, если ключа
нет. Это тот же sentinel value, просто в объектно-ориентированной обёртке — и та же обязанность проверить результат,
лежащая целиком на вызывающем коде.

## 2. Как и зачем появились исключения

Механизм исключений — не изобретение Java. Ранние формы существовали ещё в LISP и PL/I 1960-х годов, но современный
вид (`try`/`catch`/`finally`, объекты-исключения, иерархия типов) сложился в языках вроде CLU (1970-е) и особенно Ada
и C++ в 1980-х, откуда пришёл почти напрямую в Java в середине 1990-х.

Идея решала именно проблему из раздела 1: сделать невозможным молчаливое игнорирование ошибки. `throw` не возвращает
значение по обычному пути — он прерывает выполнение и передаёт объект-ошибку вверх по стеку вызовов, минуя все
промежуточные фреймы, у которых нет `catch` нужного типа, пока не найдётся подходящий обработчик или пока исключение
не долетит до самого верха и не завершит поток с явным сообщением об ошибке. Забыть обработать исключение возможно,
но невозможно сделать это молча — если не поймать, программа не продолжит работать с испорченными данными, а явно
упадёт.

Второе важное новшество — отделение кода ошибки от бизнес-логики. В стиле с кодами возврата проверки ошибок
перемежаются с основной логикой на каждом шаге; исключения позволяют написать «счастливый путь» линейно, без веток на
каждый промежуточный вызов, и вынести обработку ошибок в отдельный блок:

```java
// стиль с кодами возврата — проверка после каждого шага
Instrument instrument = tryCreate(type, ticker, name, currency);
if (instrument == null) {
    return handleError();
}
boolean added = tryAdd(repository, instrument);
if (!added) {
    return handleError();
}

// стиль с исключениями — линейный happy path, ошибки в одном месте
try {
    Instrument instrument = create(type, ticker, name, currency);
    repository.add(instrument);
} catch (IllegalArgumentException | DuplicateTickerException e) {
    return handleError(e);
}
```

## 3. Иерархия Throwable

Все объекты, которые можно бросить через `throw` и поймать через `catch`, наследуются от `Throwable`:

```
Throwable
 ├── Error
 │    ├── OutOfMemoryError
 │    ├── StackOverflowError
 │    └── ...
 └── Exception
      ├── RuntimeException
      │    ├── NullPointerException
      │    ├── IllegalArgumentException
      │    ├── IllegalStateException
      │    ├── NoSuchElementException
      │    └── ...
      ├── IOException
      ├── SQLException
      └── ...
```

**`Error`** сигнализирует о проблемах уровня самой JVM, а не логики приложения — нехватка памяти, переполнение стека
вызовов. Ловить `Error` в обычном коде почти никогда не имеет смысла: если JVM исчерпала память,
`catch (OutOfMemoryError e)` вряд ли сможет что-то восстановить, а попытка продолжить работу после такой ошибки
обычно только маскирует более серьёзную проблему.

**`Exception`** — базовый класс для ошибок уровня приложения, тех, которые в принципе могут быть обработаны и с
которыми программа способна продолжить работу тем или иным образом.

**`RuntimeException`** — подмножество `Exception`, не требующее обязательной обработки компилятором (unchecked,
подробнее в разделе 4). Стандартная библиотека предоставляет готовые типы под частые случаи: `IllegalArgumentException`
(аргумент не соответствует контракту метода), `IllegalStateException` (объект не в том состоянии для вызова),
`NullPointerException` (обращение по `null`-ссылке), `NoSuchElementException` (запрошенный элемент отсутствует,
знакомо по `Iterator.next()`). Прежде чем заводить собственный тип, стоит проверить, не покрывает ли ситуацию уже
один из этих.

## 4. Checked vs unchecked: контракт и критика

Checked-исключения — любой подкласс `Exception`, кроме `RuntimeException` и его потомков. Компилятор требует, чтобы
метод либо поймал checked-исключение внутри себя, либо явно объявил его в `throws` — тем самым распространяя
обязанность обработки на своего вызывающего:

```java
public void readConfig(String path) throws IOException { // компилятор требует объявления
    Files.readString(Path.of(path));
}
```

Unchecked-исключения (`RuntimeException` и его подклассы) такого требования не несут — метод может их бросить, ничего
не объявляя, и вызывающий код не обязан ни ловить, ни знать об этом на этапе компиляции.

Замысел checked-исключений — заставить разработчика осознанно решить, что делать с ошибкой, которую в принципе можно
ожидать и с которой можно что-то сделать (файл может не открыться, сеть может быть недоступна). На практике этот
замысел вызвал одну из самых долгоживущих дискуссий в Java-сообществе.

**Аргументы за:** checked-исключения документируют в самой сигнатуре метода, какие ошибки от него ожидать, и не дают
о них забыть — в отличие от unchecked, о существовании которых можно узнать только из документации или исходного
кода.

**Аргументы против:** на практике checked-исключения плохо масштабируются по глубине вызовов. Если метод на пятом
уровне вложенности бросает checked-исключение, а обработать его осмысленно может только вызывающий код на первом
уровне, все промежуточные методы вынуждены либо ловить и немедленно перебрасывать, либо добавлять `throws` в свою
сигнатуру — просто чтобы транслировать чужую проблему дальше, не имея возможности с ней ничего сделать. Это и
приводит к антипаттерну `catch (Exception e) { throw new RuntimeException(e); }`, встречающемуся в реальном коде
повсеместно.

Ни C#, ни Kotlin, оба спроектированные позже Java с оглядкой на её опыт, не реализуют checked-исключения вовсе —
любое исключение в этих языках ведёт себя как unchecked в Java. Внутри самой Java-экосистемы многие широко
используемые библиотеки сознательно оборачивают checked-исключения нижних слоёв в unchecked на границе своего API:
Spring Data оборачивает checked `SQLException` в unchecked `DataAccessException`, Jackson использует unchecked
`JsonProcessingException`-наследники там, где это уместно. Это не отказ от идеи «обработать ошибку осознанно» — это
перенос решения о том, ловить или пробрасывать, с компилятора на разработчика библиотеки, знающего домен лучше, чем
универсальное правило «весь путь вызовов должен объявить throws».

## 5. Как проектировать собственные исключения

**Когда заводить свой тип.** Оправдано, если вызывающему коду важно программно отличить эту ошибку от прочих —
поймать именно её и отреагировать иначе, чем на все остальные, — или если нужно нести структурированный контекст
ошибки (какой именно тикер дублируется, какое именно поле не прошло валидацию), а не только текстовое сообщение.

**От чего наследоваться.** В подавляющем большинстве случаев — от `RuntimeException`, если нет специальной причины
требовать от вызывающего кода обязательной обработки (см. раздел 4). Наследование напрямую от `Exception` или
`Throwable` практически никогда не оправдано в прикладном коде.

**Неизменяемость и осмысленное сообщение.** Собственное исключение — обычный объект: если оно несёт данные (например,
тикер), эти данные стоит делать `final` и передавать через конструктор, как любой другой неизменяемый объект (см.
материал семинара 3 «ФП в Java», раздел про immutability). Сообщение (`getMessage()`) должно быть информативным для
человека, читающего логи, а не просто повторять имя класса исключения.

```java
public class DuplicateTickerException extends RuntimeException {

    private final String ticker;

    public DuplicateTickerException(String ticker) {
        super("Инструмент с тикером уже существует: " + ticker);
        this.ticker = ticker;
    }

    public String getTicker() {
        return ticker;
    }
}
```

**Не терять причину.** Если исключение оборачивает другое (например, доменное исключение оборачивает низкоуровневый
`IOException`), обязательно передавать исходное исключение как `cause` — через конструктор
`Throwable(String message, Throwable cause)`. Потерянная причина превращает отладку в гадание:
`catch (IOException e) { throw new RuntimeException("failed"); }` уничтожает всю информацию о том, что именно пошло
не так на самом деле.

## 6. try-with-resources и AutoCloseable

Ресурсы вроде файлов, сетевых соединений или потоков ввода-вывода нужно закрывать явно — JVM не делает этого
автоматически, в отличие от памяти, которую забирает сборщик мусора. До Java 7 закрытие ресурса корректно, в том
числе при исключении, писали через `try/finally`:

```java
FileInputStream stream = new FileInputStream("data.txt");
try {
    // работа с потоком
} finally {
    stream.close();
}
```

Проблема в том, что и `try`-блок, и сам `stream.close()` в `finally` способны бросить исключение — и если оба
бросают, второе по умолчанию скрывает первое, что теряет исходную причину проблемы. Написать эту конструкцию
правильно вручную (сохранить оба исключения) — не тривиально и почти никогда не делалось на практике.

`try-with-resources`, введённый в Java 7, автоматически вызывает `close()` любого объекта, реализующего
`AutoCloseable`, при выходе из блока — и корректно обрабатывает случай, когда исключения возникают и в теле, и при
закрытии: исключение из тела блока становится основным, а исключение из `close()` прикрепляется к нему как
*подавленное* (suppressed) и доступно через `getSuppressed()`, а не теряется.

```java
try (FileInputStream stream = new FileInputStream("data.txt")) {
    // работа с потоком
} // close() вызывается автоматически, даже если тело блока бросит исключение
```

Ресурсов может быть несколько — закрываются они в порядке, обратном объявлению:

```java
try (var in = new FileInputStream("data.txt");
     var out = new FileOutputStream("copy.txt")) {
    // сначала закроется out, затем in
}
```

## 7. Из чего складывается стоимость исключения

Создание и обработка исключения не бесплатны, и у этой стоимости есть конкретные, измеримые источники.

**Выделение объекта.** Каждое `new SomeException(...)` — обычное выделение объекта в куче, ничем принципиально не
отличающееся по этой части от любого другого `new`.

**`fillInStackTrace()`.** Это главная и самая заметная статья расходов. Конструктор `Throwable` по умолчанию вызывает
`fillInStackTrace()`, которая обходит весь текущий стек вызовов потока и материализует его в массив
`StackTraceElement` — по одному элементу на каждый активный вызов метода, вплоть до точки создания исключения. Чем
глубже стек в момент создания исключения, тем дороже эта операция.

**Разворачивание стека при обработке.** Когда исключение действительно долетает от места `throw` до места `catch`,
JVM должна пройти через промежуточные фреймы стека, выполняя логику `finally`-блоков и освобождая ресурсы
try-with-resources по пути. Эта часть тоже не бесплатна, хотя обычно заметно дешевле `fillInStackTrace()`.

`Throwable` предоставляет способ отключить самую дорогую часть — заполнение трассировки стека — через
четырёхаргументный защищённый конструктор:

```java
protected Throwable(String message, Throwable cause,
                     boolean enableSuppression, boolean writableStackTrace)
```

Передача `writableStackTrace = false` полностью пропускает вызов `fillInStackTrace()`. Цена — у такого исключения
`getStackTrace()` вернёт пустой массив, то есть при отладке не будет видно, откуда именно оно было брошено. Это
осознанный компромисс, уместный только там, где стоимость `fillInStackTrace()` доказанно значима (частое
использование в горячем пути) и где место возникновения не требуется для диагностики.

Практический замер разницы между обычным исключением, исключением без трассировки стека и кодом возврата — на
продвинутом треке и в материале «JMH и замер производительности».

## 8. Когда исключения — не тот инструмент

Исключения решают проблему «не дать ошибке потеряться незамеченной», но это не значит, что ими стоит сигнализировать
о чём угодно.

**Не для управления обычным потоком выполнения.** Ситуация, которая случается регулярно и является частью нормальной
логики (например, «элемент не найден при поиске» в коллекции, которую перебирают в цикле в поисках совпадения), — не
повод для исключения; для неё подходят `Optional` или обычная проверка условия. Исключение уместно для ситуации,
которая по смыслу является отклонением от нормального хода вещей, а не одним из ожидаемых исходов.

**Не для валидации, выполняемой на каждой итерации горячего цикла.** Здесь работает та же логика, что и в разделе 7:
если код исполняется миллионы раз в секунду, стоимость `fillInStackTrace()` на каждый «отказ» способна стать заметной
частью общего времени выполнения. Для таких случаев больше подходят коды возврата, `Optional`, или — если исключения
всё же нужны по архитектурным причинам — исключения с отключённой трассировкой стека.

**Не как замена документации.** Исключение с сообщением `"Error"` или `"Something went wrong"` формально решает
задачу «не дать ошибке потеряться», но не решает задачу «дать разработчику, читающему логи через полгода, понять, что
произошло». Осмысленное сообщение — не опциональная деталь, а часть контракта исключения не в меньшей степени, чем
его тип.
