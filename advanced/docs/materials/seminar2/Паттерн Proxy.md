## 1. Зачем нужен объект-заместитель

Proxy — структурный паттерн, предоставляющий объект-заместитель, который перехватывает обращения к другому объекту и
добавляет собственное поведение вокруг вызова (замер времени, логирование, кэширование, отложенная инициализация,
проверка прав доступа), не изменяя код целевого объекта.

Клиентский код взаимодействует с прокси так же, как с самим целевым объектом — оба реализуют один и тот же контракт, и
клиент не обязан знать, с чем именно он работает напрямую.

В Java эта идея реализуется тремя принципиально разными способами, различающимися по тому, на каком этапе создаётся
класс-заместитель: во время написания кода, во время выполнения на основе интерфейса или во время выполнения на основе
подкласса.

## 2. Самописный (статический) прокси

Класс-заместитель пишется вручную, реализует тот же интерфейс, что и целевой объект, хранит ссылку на него и делегирует
каждый вызов, добавляя собственную логику до или после делегирования:

```java
public interface Repository {
    void save(Object entity);

    Object findById(long id);
}

public class TimingRepositoryProxy implements Repository {
    private final Repository delegate;

    public TimingRepositoryProxy(Repository delegate) {
        this.delegate = delegate;
    }

    @Override
    public void save(Object entity) {
        long start = System.nanoTime();
        delegate.save(entity);
        System.out.printf("save() выполнен за %d нс%n", System.nanoTime() - start);
    }

    @Override
    public Object findById(long id) {
        long start = System.nanoTime();
        Object result = delegate.findById(id);
        System.out.printf("findById() выполнен за %d нс%n", System.nanoTime() - start);
        return result;
    }
}
```

**Преимущества.** Код прозрачен и отлаживается как обычный класс — никакой генерации байт-кода или рефлексии,
минимальные накладные расходы на вызов, отсутствие зависимости от сторонних библиотек.

**Ограничения.** Число классов-прокси растёт пропорционально числу проксируемых интерфейсов; каждый новый метод
интерфейса требует ручного добавления реализации в прокси, а сквозная логика (например, замер времени) дублируется в
каждом методе каждого прокси, что нарушает DRY при росте числа проксируемых типов.

## 3. JDK Dynamic Proxy

Пакет `java.lang.reflect` предоставляет механизм генерации класса-заместителя во время выполнения — без необходимости
писать класс прокси вручную. Генерация опирается на интерфейс: `Proxy.newProxyInstance` создаёт объект, реализующий
заданный набор интерфейсов, все вызовы методов которого перенаправляются в единственный обработчик `InvocationHandler`:

```java
Repository target = new InMemoryRepository();

Repository proxy = (Repository) Proxy.newProxyInstance(
        Repository.class.getClassLoader(),
        new Class<?>[]{Repository.class},
        (Object p, Method method, Object[] args) -> {
            long start = System.nanoTime();
            Object result = method.invoke(target, args);
            System.out.printf("%s() выполнен за %d нс%n", method.getName(), System.nanoTime() - start);
            return result;
        });
```

Один `InvocationHandler` покрывает все методы интерфейса — при добавлении нового метода в интерфейс прокси автоматически
начинает его перехватывать, без правки кода обработчика.

**Преимущества.** Не требует сторонних библиотек — механизм входит в стандартную библиотеку Java; один обработчик
покрывает произвольное число методов и интерфейсов; логика перехвата отделена от контракта в явном виде.

**Ограничения.** Работает только с интерфейсами — прокси не может быть создан для класса без интерфейса, а
сгенерированный класс не может дополнительно наследовать другой класс (он уже наследует `java.lang.reflect.Proxy`).
Каждый вызов метода проходит через `Method.invoke`, что дороже прямого вызова за счёт рефлексии — на большинстве задач
разница пренебрежимо мала, но заметна в горячих участках кода с интенсивными вызовами.

## 4. CGLIB

CGLIB (Code Generation Library) — сторонняя библиотека, генерирующая прокси иначе: не через реализацию интерфейса, а
через создание подкласса целевого класса во время выполнения. Это снимает главное ограничение JDK Dynamic Proxy —
интерфейс становится не обязателен:

```java
void example() {
    Enhancer enhancer = new Enhancer();
    enhancer.setSuperclass(InMemoryRepository.class);
    enhancer.setCallback((MethodInterceptor) (obj, method, args, methodProxy) -> {
        long start = System.nanoTime();
        Object result = methodProxy.invokeSuper(obj, args);
        System.out.printf(
                "%s() выполнен за %d нс%n", 
                method.getName(), System.nanoTime() - start);
        return result;
    });

    Repository proxy = (Repository) enhancer.create();
}
```

`Enhancer` генерирует новый класс байт-кодом непосредственно во время выполнения (в отличие от JDK Dynamic Proxy,
использующего для этого более узкий встроенный механизм), переопределяя методы целевого класса и вызывая исходную
реализацию через `methodProxy.invokeSuper`.

**Преимущества.** Не требует интерфейса — можно проксировать конкретный класс напрямую; переопределение методов
подклассом обычно быстрее вызова через рефлексию, поскольку сгенерированный байт-код вызывает метод напрямую, а не через
`Method.invoke`.

**Ограничения.** Генерация подкласса означает, что переопределить `final`-класс или `final`-метод невозможно —
переопределение запрещено языком независимо от способа генерации кода. Требует дополнительной зависимости (`cglib`, либо
получаемой транзитивно через другую библиотеку). Целевой класс должен иметь доступный конструктор, вызываемый при
создании прокси (в готовых фреймворках, например в Spring, для обхода этого ограничения дополнительно используется
библиотека Objenesis, позволяющая создавать экземпляр класса без вызова конструктора).

CGLIB генерирует байт-код через рефлексию поверх внутренностей JDK и давно не обновлялся под модульную систему Java.
На JDK 16+ строгая инкапсуляция внутренностей (JEP 396 — инкапсуляция по умолчанию, JEP 403 — сужение стандартного
API рефлексии) иногда приводит к `InaccessibleObjectException` при создании `Enhancer`. Обходной путь — тот же, что
и для других библиотек, полагающихся на глубокую рефлексию (например, JOL): явно открыть нужный пакет флагом JVM
`--add-opens java.base/java.lang=ALL-UNNAMED`.

## 5. Сравнение подходов

| **Критерий**                              | **Самописный прокси**                                                  | **JDK Dynamic Proxy**                                          | **CGLIB**                                              |
|-------------------------------------------|------------------------------------------------------------------------|----------------------------------------------------------------|--------------------------------------------------------|
| Требует интерфейс                         | Да                                                                     | Да                                                             | Нет                                                    |
| Работает с `final`-классами/методами      | — (ограничение не применимо)                                           | — (ограничение не применимо)                                   | Нет                                                    |
| Дополнительная зависимость                | Нет                                                                    | Нет (входит в JDK)                                             | Да                                                     |
| Поддержка новых методов без правки прокси | Нет — нужно дописывать вручную                                         | Да — автоматически                                             | Да — автоматически                                     |
| Относительная скорость вызова             | Без косвенности                                                        | Медленнее — вызов через рефлексию                              | Быстрее JDK Dynamic Proxy, но медленнее прямого вызова |
| Когда уместен                             | Небольшое, стабильное число методов; максимальная прозрачность отладки | Есть интерфейс; число проксируемых типов и методов может расти | Интерфейса нет или проксировать нужно конкретный класс |

## 6. Self-invocation: ограничение обоих динамических механизмов

И JDK Dynamic Proxy, и CGLIB работают одинаково с точки зрения того, что именно они перехватывают: вызов метода
**через ссылку на прокси**, полученную снаружи. Если внутри целевого объекта один метод вызывает другой через `this`
— напрямую, без обращения к прокси, — такой вызов проходит мимо `InvocationHandler`/`MethodInterceptor` целиком:

```java
public class InMemoryRepository implements Repository {
    @Override
    public Object findById(long id) {
        // ...
    }

    public Object findByIdLogged(long id) {
        return findById(id); // вызов через this — прокси этот вызов не видит
    }
}
```

Если снаружи вызвать `proxy.findByIdLogged(id)`, перехватится только сам вызов `findByIdLogged` — а вложенный вызов
`findById(id)` внутри него выполнится напрямую на реальном объекте, минуя весь механизм проксирования. Причина
структурная, а не случайный недосмотр: и `Proxy.newProxyInstance`, и `Enhancer` создают отдельный объект-обёртку,
существующий *снаружи* целевого объекта; `this` внутри методов самого целевого объекта продолжает указывать на него
же, а не на обёртку.

**Практическое следствие.** Это ровно та причина, по которой в Spring аннотации `@Transactional`, `@Cacheable` и
подобные не срабатывают при вызове аннотированного метода из другого метода того же бина (self-invocation) — Spring
AOP по умолчанию построен на тех же двух механизмах (JDK Dynamic Proxy или CGLIB), поэтому наследует то же
ограничение. Обходные пути в реальных проектах: вынести вызываемый метод в отдельный бин, внедрить в класс
самого себя через `ApplicationContext` (не рекомендуется), либо использовать `AspectJ` с compile-time или
load-time weaving, который не имеет этого ограничения, поскольку встраивает поведение прямо в байт-код класса, а не
оборачивает объект снаружи.

## 7. Проксирование в промышленных фреймворках

Оба динамических механизма — не учебная абстракция, а основа механизма Dependency Injection в Spring: если бин реализует
хотя бы один интерфейс, Spring по умолчанию создаёт JDK Dynamic Proxy; если интерфейса нет — CGLIB. Тем же способом
реализованы декларативные транзакции (`@Transactional`), кэширование (`@Cacheable`) и аспектно-ориентированное
программирование (Spring AOP в целом): аннотация на методе фактически превращает вызов в обращение к прокси,
добавляющему поведение вокруг оригинального метода — ту же задачу, что решают примеры этого материала, только для более
общего круга сквозных задач.
