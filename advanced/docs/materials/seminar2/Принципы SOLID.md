## 1. От ООП к принципам проектирования

Механизмы ООП допускают как удачное, так и неудачное применение; сам по себе язык не проверяет качество дизайна.
Принципы SOLID формулируют пять правил, следование которым делает код устойчивым к изменению и расширению:

- **S** — Single Responsibility Principle;
- **O** — Open/Closed Principle;
- **L** — Liskov Substitution Principle;
- **I** — Interface Segregation Principle;
- **D** — Dependency Inversion Principle.

## 2. Принцип подстановки Лисков (LSP)

Объекты подтипа должны быть заменяемы объектами базового типа без нарушения корректности программы. Метод базового
класса задаёт контракт (допустимые аргументы, возвращаемое значение, возможные исключения), и подкласс не вправе этот
контракт сужать.

Метод `getDividends(BigDecimal)`, объявленный в `Instrument`, не имеет осмысленной реализации для `Bond` — облигация
приносит купонный доход, а не дивиденды:

```java
// Нарушение LSP — оба варианта искажают контракт
public BigDecimal getDividends(BigDecimal shareCount) {
    return BigDecimal.ZERO;                                    // молчаливая подмена
}

public BigDecimal getDividends(BigDecimal shareCount) {
    throw new UnsupportedOperationException("Bonds do not pay dividends");  // отказ
}
```

Корректное решение — вынести операцию из базового класса в отдельную абстракцию, реализуемую только теми типами, для
которых она осмысленна:

```java
public interface DividendPaying {
    BigDecimal getDividends(BigDecimal shareCount);
}

public class Stock extends Instrument implements DividendPaying {
    @Override
    public BigDecimal getDividends(BigDecimal shareCount) {
        return dividendPerShare.multiply(shareCount);
    }
}
```

`Bond` интерфейс `DividendPaying` не реализует; код, работающий с дивидендами, оперирует типом `DividendPaying`, и
некорректный вызов становится невозможен на уровне компиляции.

## 3. Принцип разделения интерфейса (ISP)

Клиенты не должны зависеть от методов, которые они не используют. Единый интерфейс `Instrument` со всеми возможными
операциями (дивиденды, купонный доход, дата погашения, состав активов) заставил бы каждый подкласс реализовывать чужие
обязанности — то есть привёл бы к тому же нарушению LSP, что и в разделе 2. ISP и LSP взаимосвязаны: типичная причина
нарушения LSP — «толстый» интерфейс, а устранение — разложение на узкие роли:

```java
public interface DividendPaying {
    BigDecimal getDividends(BigDecimal shareCount);
}

public interface CouponBearing {
    BigDecimal getCouponYield();

    LocalDate getMaturityDate();
}

public interface Diversified {
    List<Instrument> getUnderlyingAssets();
}
```

Класс реализует только те интерфейсы, которые соответствуют его природе: `Stock implements DividendPaying`,
`Bond implements CouponBearing`, `Etf implements Diversified`.

> `List` — часть Java Collection Framework, подробно рассматривается на семинаре 5. `LocalDate` — класс из пакета
`java.time`, работа со временем в Java разбирается точечно, по мере необходимости.

## 4. Принцип единственной ответственности (SRP)

У класса должна быть одна причина для изменения. Если в `Portfolio` добавить наряду с расчётом стоимости ещё и вывод в
консоль:

```java
public class Portfolio {
    public BigDecimal totalValue(Map<String, BigDecimal> prices) { /* доменная логика */ }

    public void printReport() { /* форматирование и вывод */ }
}
```

`printReport` вводит вторую причину для изменения класса — переход на формат JSON или вывод в файл потребует править
`Portfolio` по причине, не связанной с учётом позиций. Форматирование выносится в отдельный компонент. Тот же вопрос
применим к CLI-командам: команда, которая одновременно парсит аргументы, обращается к данным и форматирует вывод,
нарушает SRP так же, как `Portfolio` с `printReport`.

> `Map` — часть Java Collection Framework, подробно рассматривается на семинаре 5.

## 5. Принцип открытости/закрытости (OCP)

Код должен быть открыт для расширения, но закрыт для изменения. Ранняя версия определения типа инструмента через
`instanceof`-цепочку требовала правки при каждом новом типе:

```java
void example() {
    if (type == InstrumentType.STOCK) return "Акция";
    else if (type == InstrumentType.BOND) return "Облигация";
    else if (type == InstrumentType.ETF) return "Фонд";
    else throw new IllegalStateException("Unknown type");
}
```

Переход к полиморфному `getDescription()`, выполненный на первом семинаре, устраняет проблему: добавление нового
подкласса `Instrument` не требует изменения ни одного существующего класса. Тот же принцип применим к созданию объектов
через фабрику и фильтрации инструментов через Strategy: добавление нового типа или нового правила отбора должно
сводиться к добавлению нового класса, а не к правке существующего.

## 6. Принцип инверсии зависимостей (DIP) — обзорно

Модули верхнего уровня не должны зависеть от модулей нижнего уровня — оба должны зависеть от абстракций. На практике это
означает, что бизнес-логика не должна напрямую зависеть от конкретной реализации внешнего клиента или библиотеки — она
должна зависеть от интерфейса, скрывающего детали.

Полноценное применение DIP становится наглядным при работе с фреймворками, управляющими внедрением зависимостей (
dependency injection) — тема будет подробно разобрана при знакомстве со Spring на одном из следующих семинаров.

## 7. Взаимосвязь принципов друг с другом и с ООП

Принципы SOLID не независимы друг от друга — каждый опирается на конкретный механизм ООП и одновременно создаёт условия,
при которых соблюдение соседних принципов становится естественным или, напротив, необходимым.

**Связь с механизмами ООП.** Каждый принцип регламентирует использование одного из столпов ООП:

| **Механизм ООП**        | **Принцип, который его регламентирует**                                                                                              |
|-------------------------|--------------------------------------------------------------------------------------------------------------------------------------|
| Наследование            | LSP — подтип обязан вести себя как заменитель базового типа                                                                          |
| Инкапсуляция            | SRP — единица инкапсуляции должна отвечать за одну область                                                                           |
| Полиморфизм             | OCP — расширение поведения через новые реализации, а не через изменение существующих                                                 |
| Абстракция (интерфейсы) | ISP — абстракция должна быть узкой, под конкретного потребителя; DIP — зависимость строится на абстракции, а не на конкретном классе |

**Связи между принципами.** ISP и LSP взаимно обусловлены: «толстый» интерфейс (нарушение ISP) почти всегда вынуждает
часть подклассов реализовывать чужие операции заглушками или исключениями (нарушение LSP) — это было продемонстрировано
в разделах 2–3 на примере `Instrument`. SRP можно рассматривать как ISP, применённый не к интерфейсу, а к классу: у
класса, как и у интерфейса, должна быть одна причина для изменения, единая ось ответственности.

OCP оказывается следствием, а не независимым правилом: расширяемость без модификации существующего кода становится
возможной только тогда, когда система уже полагается на LSP (новый подкласс действительно подставим на место базового),
на DIP (клиентский код зависит от абстракции, а не от конкретной реализации) и на ISP (абстракция достаточно узкая,
чтобы новая реализация не тянула за собой лишних обязательств). В этом смысле OCP — не столько отдельная техника,
сколько наблюдаемый эффект от корректного применения остальных четырёх принципов.

DIP связывает эту картину с полиморфизмом на уровне архитектуры: если модуль зависит от абстракции, а не от конкретного
класса, подмена реализации (в том числе ради соблюдения OCP) не требует изменения зависящего кода — ровно тот же
полиморфный механизм, что используется в LSP, применяется здесь на уровне модулей, а не отдельных объектов.

## 8. KISS и DRY

**KISS** (Keep It Simple, Stupid) — предпочтение простого решения сложному при прочих равных: паттерн проектирования
оправдан, когда решает реальную проблему, а не потому, что был недавно изучен.

**DRY** (Don't Repeat Yourself) — устранение дублирования *знания*, а не дублирования текста. Два похожих, но независимо
изменяющихся фрагмента кода не обязательно нарушают DRY; преждевременное обобщение ради устранения текстового сходства
может дать абстракцию хуже дублирования.

---

Принципы SOLID задают критерии качества дизайна, но не дают готовых конструкций для их достижения — такие конструкции
представляют собой паттерны проектирования (GoF).
