## 1. Введение в парадигму

**Объектно-ориентированное программирование (ООП)** — парадигма проектирования и разработки программного обеспечения, в основе которой лежит понятие *объекта*: сущности, объединяющей состояние (данные) и поведение (операции над этими данными).

В отличие от процедурного подхода, где данные и функции существуют раздельно, ООП связывает их воедино. Объект инкапсулирует собственное состояние и предоставляет внешнему коду ограниченный набор операций для взаимодействия с ним. Такой подход отражает естественный способ моделирования предметной области: программа рассматривается как совокупность взаимодействующих объектов, каждый из которых отвечает за определённую область ответственности.

Исторически концепция ООП восходит к языку Simula (1960-е годы) и получила развитие в языке Smalltalk. Автор термина «object-oriented», Алан Кэй, вкладывал в него смысл, отличный от того, что впоследствии стало доминирующим в языках вроде C++ и Java: в его понимании ключевыми были не классы и наследование, а обмен сообщениями между независимыми объектами. Это различие в трактовках остаётся предметом дискуссий и в настоящее время.

Основные преимущества, которые парадигма привносит в разработку:

- **Управляемость сложности.** Крупная система декомпозируется на логически обособленные единицы, каждая из которых может разрабатываться и анализироваться отдельно.
- **Переиспользование кода.** Общее поведение выносится в базовые абстракции и повторно применяется без дублирования.
- **Расширяемость.** Новая функциональность добавляется через введение новых типов, без модификации существующего кода.
- **Соответствие предметной области.** Структура программы отражает структуру моделируемой реальности, что упрощает понимание и сопровождение.

## 2. Принципы ООП и их взаимосвязь

Традиционно выделяются три основных принципа ООП: инкапсуляция, наследование и полиморфизм. Ряд источников добавляет к ним абстракцию как самостоятельный принцип.

### 2.1. Инкапсуляция

**Инкапсуляция** — сокрытие внутренней реализации объекта за определённым контрактом (публичным интерфейсом). Объект предоставляет внешнему коду информацию о том, *что* он делает, скрывая детали того, *как* это реализовано.

Распространённое заблуждение состоит в отождествлении инкапсуляции с формальным объявлением полей приватными и предоставлением методов доступа (`get`/`set`). В действительности суть принципа глубже: инкапсуляция обеспечивает **защиту инвариантов** объекта. Инвариант — это условие, которое должно оставаться истинным на протяжении всего времени существования объекта. Класс, спроектированный корректно, гарантирует, что его состояние не может быть приведено в некорректное состояние извне.

Существенно, что инкапсуляция не является привилегией объектно-ориентированных языков. Она реализуема и в процедурных языках. Показательным примером служит стандартная библиотека языка C. Системный вызов открытия файла объявлен в заголовочном файле следующим образом:

```other
// fcntl.h
int open(const char *pathname, int flags);
```

Наружу выставляется исключительно сигнатура процедуры — контракт. Реализация скрыта внутри библиотеки и операционной системы, причём в различных системах она различна: механизмы работы с файловой системой в Linux, macOS и Windows не совпадают. Тем не менее клиентский код обращается к единому интерфейсу и не зависит от конкретной реализации. Реализация может быть полностью переписана без каких-либо последствий для использующего её кода. В этом и заключается суть инкапсуляции.

### 2.2. Наследование

**Наследование** — механизм, устанавливающий между типами отношение «является» (IS-A) и позволяющий одному классу (потомку) перенимать состояние и поведение другого класса (родителя). Наследование служит двум основным целям: переиспользованию кода и построению иерархий типов, отражающих отношения предметной области.

Функциональные возможности, предоставляемые наследованием:

- переиспользование полей и методов базового класса без их повторного объявления;
- переопределение (override) унаследованного поведения в потомке;
- построение иерархий, в которых потомки специализируют более общий базовый тип.

Наследование, несмотря на свою полезность, сопряжено с существенными издержками, которые подробно рассматриваются в разделе 3. Основная из них — жёсткая связанность между родителем и потомками, вследствие которой изменения в базовом классе способны каскадно распространяться по всей иерархии.

### 2.3. Полиморфизм

**Полиморфизм** — способность единого интерфейса обслуживать объекты различных типов, при которой конкретное поведение определяется фактическим типом объекта. Различают несколько видов полиморфизма; в рамках настоящего материала рассматриваются два основных.

**Полиморфизм подтипов (subtype polymorphism)** реализуется через переопределение методов в подклассах. Код, оперирующий переменной базового типа, вызывает реализацию, соответствующую фактическому типу объекта. Выбор реализации осуществляется во время выполнения (runtime) и известен как позднее (динамическое) связывание.

**Специальный полиморфизм в форме перегрузки (overloading)** предполагает существование нескольких методов с одним именем, но различными сигнатурами. Выбор вызываемого метода производится компилятором на основе типов аргументов во время компиляции (compile-time). Данный вид известен как раннее (статическое) связывание.

Помимо перечисленных, существует **параметрический полиморфизм**, реализуемый в Java посредством обобщений (generics). Его рассмотрение выносится на последующие семинары.

### 2.4. Взаимосвязь принципов

Принципы ООП не являются независимым перечнем требований. Они образуют взаимосвязанную систему, в которой каждый принцип обеспечивает работоспособность остальных. Понимание этих связей существенно для корректного применения парадигмы.

**Полиморфизм подтипов невозможен без наследования.** Данный вид полиморфизма построен непосредственно на иерархии типов. При отсутствии наследования или реализации общего интерфейса отсутствует и общий тип, через который можно было бы единообразно обращаться к разнородным объектам. Совокупность объектов различных типов в этом случае распадается на несвязанные множества, и единообразная обработка становится невозможной.

**Наследование без полиморфизма в значительной мере утрачивает смысл.** При отсутствии полиморфизма наследование сводится к простому копированию полей и методов от родителя к потомку. Переиспользование кода при этом сохраняется, однако возможность переопределять поведение и обращаться к потомкам через базовый тип теряется. Именно эта возможность составляет основную ценность наследования; без неё оно превращается во второстепенный механизм.

**Инкапсуляция служит фундаментом для остальных принципов.** При отсутствии сокрытия реализации внутреннее состояние объектов становится доступным для произвольного изменения извне. В этих условиях ни наследование, ни полиморфизм не обеспечивают надёжности: иерархия типов выстраивается поверх объектов, целостность состояния которых не гарантирована. Инкапсуляция обеспечивает защищённость объектов, поверх которых затем возводятся иерархии и организуется полиморфное поведение.

Схематически взаимосвязь принципов может быть представлена следующим образом:

```other
Инкапсуляция  →  сокрытие реализации, защита инвариантов
     │
     ▼
Наследование  →  построение иерархий поверх защищённых объектов
     │
     ▼
Полиморфизм   →  единообразная работа с иерархией через общий тип
```

Устранение любого из уровней приводит к разрушению всей конструкции. Именно во взаимосвязи принципов, а не в трёх изолированных определениях, заключается существо парадигмы.

## 3. Проблемы и ограничения парадигмы

Объектно-ориентированный подход не является универсальным решением и обладает рядом известных ограничений. Осведомлённость о них необходима для взвешенного применения парадигмы.

### 3.1. Издержки наследования

Наследование создаёт **жёсткую связанность** (tight coupling) между базовым классом и его потомками. Потомок зависит от деталей реализации родителя, вследствие чего изменение базового класса способно нарушить работу всех производных классов. В глубоких и разветвлённых иерархиях этот эффект приобретает каскадный характер: добавление поля или изменение логики метода в корневом классе требует проверки и, возможно, модификации всех потомков.

Данная проблема послужила основанием для широко признанной рекомендации: **предпочитать композицию наследованию**. Вместо построения иерархии объекты объединяются посредством включения одного в другой. Различают две формы такого объединения:

- **Агрегация** — объект содержит ссылку на другой объект, существующий независимо. Время жизни включаемого объекта не связано со временем жизни включающего.
- **Композиция** — более сильная форма владения, при которой включаемый объект является неотъемлемой частью включающего и не существует отдельно от него.

Композиция и агрегация обеспечивают повторное использование поведения без жёсткой связанности, характерной для наследования, и в значительной части случаев являются предпочтительными.

### 3.2. Нарушение принципа подстановки

Некорректно спроектированная иерархия способна нарушать **принцип подстановки Барбары Лисков (LSP)**, согласно которому объекты базового типа должны допускать замену объектами производного типа без нарушения корректности программы. Если потомок не в состоянии предоставить осмысленную реализацию унаследованного метода — например, вынужден возвращать некорректное значение или возбуждать исключение, — это свидетельствует о неверном проектировании иерархии. Подобные ситуации подробно разбираются на практической части семинара.

### 3.3. Избыточная сложность для неподходящих задач

Не всякая задача обладает естественной объектной структурой. Для алгоритмических, вычислительных и утилитарных задач введение классов и иерархий способно усложнить решение, а не упростить его. Формальное требование поместить всякую логику в объект приводит к появлению искусственных сущностей, не имеющих аналога в предметной области, — классов, единственное назначение которых состоит в размещении процедуры. Известным проявлением этой тенденции служит распространение классов-обёрток вида `Manager`, `Processor` или `Helper`, лишённых собственного состояния и представляющих собой процедуры, замаскированные под объекты. В подобных случаях процедурное или функциональное решение оказывается более прямым и понятным.

### 3.4. Несоответствие ряду классов задач

Объектная модель эффективна там, где сущности предметной области обладают состоянием и поведением, однако для отдельных классов задач иные парадигмы оказываются более выразительными. Для обработки потоков данных, преобразований и вычислений, свободных от побочных эффектов, функциональный подход, оперирующий чистыми функциями и неизменяемыми структурами данных, нередко приводит к более лаконичному и надёжному коду. Современные языки, включая Java, сочетают несколько парадигм, что позволяет применять для каждой части системы наиболее подходящую модель. Приверженность исключительно объектной модели в ущерб прочим ограничивает выразительные возможности разработчика.

### 3.5. Отсутствие свойств «серебряной пули»

ООП не устраняет сложность автоматически самим фактом своего применения. Качество системы определяется качеством проектирования, а не выбором парадигмы. Неудачно спроектированная объектная модель — с чрезмерно глубокими иерархиями, размытыми областями ответственности и высокой связанностью — способна оказаться менее понятной и сопровождаемой, чем эквивалентное процедурное решение. Объектная ориентация предоставляет средства управления сложностью, но не гарантирует их корректного применения. Инструмент не заменяет инженерного суждения.

### 3.6. Мультипарадигменность современных языков

Перечисленные ограничения не свидетельствуют о несостоятельности парадигмы, а лишь очерчивают область её эффективного применения. Практическим ответом на них служит отказ от приверженности единственной парадигме. Современные языки программирования, как правило, являются мультипарадигменными: они сочетают объектную модель с элементами функционального, процедурного и иных подходов, позволяя выбирать для каждой части системы наиболее подходящее средство.

Java служит показательным примером такой эволюции. Изначально спроектированный как преимущественно объектно-ориентированный, язык последовательно вбирал средства других парадигм: лямбда-выражения и функциональные интерфейсы, потоковую обработку данных (Stream API), неизменяемые типы-записи (`record`), сопоставление с образцом (pattern matching). Эти средства не вытесняют объектную модель, а дополняют её, расширяя круг задач, решаемых естественным образом в рамках единого языка.

Из этого следует практический вывод: объектная ориентация является одним из инструментов, обладающим областью эффективного применения, а не универсальным методом, пригодным для любых задач. Владение несколькими парадигмами и способность выбирать подходящую под конкретную задачу составляют существенную часть инженерной квалификации. Выбор парадигмы определяется характером решаемой задачи, а не предпочтениями разработчика.

## 4. Принципы и шаблоны проектирования (обзорно)

Объектно-ориентированное проектирование опирается на совокупность принципов и типовых решений, выработанных практикой. В рамках настоящего материала они упоминаются обзорно; их систематическое рассмотрение выносится на последующие семинары.

**SOLID** — набор из пяти принципов проектирования, направленных на создание понятных, гибких и сопровождаемых систем: принцип единственной ответственности (SRP), принцип открытости/закрытости (OCP), принцип подстановки Лисков (LSP), принцип разделения интерфейсов (ISP) и принцип инверсии зависимостей (DIP). Следует отметить, что связь принципов SOLID исключительно с объектной ориентацией является предметом дискуссии: ряд авторов указывает на их более широкую применимость.

**KISS** (Keep It Simple, Stupid) и **DRY** (Don't Repeat Yourself) — общие принципы разработки, предписывающие соответственно минимизацию избыточной сложности и устранение дублирования.

**Шаблоны проектирования GoF** (Gang of Four) — каталог из двадцати трёх типовых решений повторяющихся задач проектирования, разделённых на порождающие, структурные и поведенческие. Шаблоны представляют собой формализованный опыт и общий словарь для описания архитектурных решений.

Все перечисленные группы принципов и шаблонов подробно рассматриваются на последующих семинарах модуля.

## 5. ООП в языке Java

Java является объектно-ориентированным языком, в котором объектная модель реализована на уровне синтаксиса и семантики. Ниже приводится краткий обзор соответствующих языковых средств.

### 5.1. Классы и объекты

Основной единицей является **класс** — описание типа, задающее состояние (поля) и поведение (методы). **Объект** представляет собой экземпляр класса, создаваемый оператором `new`. Каждый объект обладает собственным состоянием и размещается в области памяти, управляемой сборщиком мусора.

```java
public class Instrument {
    private final String ticker;   // поле (состояние)

    public Instrument(String ticker) {   // конструктор
        this.ticker = ticker;
    }

    public String getTicker() {    // метод (поведение)
        return ticker;
    }
}
```

### 5.2. Модификаторы доступа

Инкапсуляция обеспечивается модификаторами доступа, определяющими область видимости членов класса:

- `private` — доступ ограничен пределами класса;
- `protected` — доступ из класса, его потомков и того же пакета;
- `public` — доступ без ограничений;
- модификатор по умолчанию (package-private) — доступ в пределах пакета.

Корректная инкапсуляция предполагает объявление полей приватными и предоставление доступа к ним исключительно через методы, контролирующие соблюдение инвариантов.

### 5.3. Наследование и интерфейсы

Наследование классов реализуется ключевым словом `extends`. Java поддерживает **одиночное наследование классов**: класс может наследоваться не более чем от одного базового класса.

```java
public class Stock extends Instrument {
    private final String sector;
    // ...
}
```

Ключевое слово `abstract` позволяет объявить **абстрактный класс** — класс, не предназначенный для непосредственного создания экземпляров и способный содержать методы без реализации. Такие методы обязаны быть реализованы в неабстрактных потомках, что обеспечивается компилятором.

**Интерфейс** (`interface`) задаёт контракт без реализации (за исключением методов по умолчанию, введённых в Java 8). Класс может реализовывать произвольное число интерфейсов посредством ключевого слова `implements`, что компенсирует отсутствие множественного наследования классов и способствует соблюдению принципа разделения интерфейсов.

### 5.4. Полиморфизм

Полиморфизм подтипов реализуется через переопределение методов. Переопределяемый метод помечается аннотацией `@Override`, что позволяет компилятору проверить корректность переопределения:

```java
public abstract class Instrument {
    public abstract String getDescription();
}

public class Stock extends Instrument {
    @Override
    public String getDescription() {
        return "Акция, сектор: " + sector;
    }
}
```

При обращении к методу через переменную базового типа вызывается реализация, соответствующая фактическому типу объекта:

```java
List<Instrument> instruments = List.of(new Stock(...), new Bond(...));
for (Instrument instrument : instruments) {
    System.out.println(instrument.getDescription());  // вызов реализации подкласса
}
```

Специальный полиморфизм в форме перегрузки реализуется объявлением нескольких методов с одним именем и различными сигнатурами. Выбор метода производится компилятором на основе типов аргументов.

### 5.5. Параметрический полиморфизм

Java поддерживает **обобщения (generics)** — средство параметрического полиморфизма, позволяющее создавать классы и методы, параметризованные типами. Обобщения обеспечивают типобезопасность на этапе компиляции и устраняют необходимость явного приведения типов.

```java
List<Instrument> instruments = new ArrayList<>();  // список, параметризованный типом Instrument
```

Детальное рассмотрение обобщений, включая ограничения типовых параметров и механизм стирания типов, выносится на последующие семинары.
