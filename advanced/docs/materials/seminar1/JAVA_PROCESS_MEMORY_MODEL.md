## 1. Введение

Понимание того, как виртуальная машина Java размещает данные в памяти, относится к числу знаний, отличающих поверхностное владение языком от глубокого. Разработчик, не осведомлённый об устройстве памяти, воспринимает объекты как абстракции, лишённые физического представления, и оказывается неспособен объяснить расхождения в потреблении памяти, оценить накладные расходы структур данных или диагностировать проблемы производительности.

Настоящий материал рассматривает устройство памяти процесса JVM: разделение памяти на области, внутреннюю структуру объектов, размещение примитивов, ссылок и массивов, а также — обзорно — механизм автоматического освобождения памяти (сборку мусора). Все приводимые сведения относятся к виртуальной машине HotSpot — эталонной реализации JVM. Базовые принципы применимы и к другим реализациям, однако отдельные детали могут различаться.

За пределами материала остаётся модель памяти Java (Java Memory Model) в аспекте многопоточного взаимодействия — вопросы видимости изменений между потоками, отношения happens-before и семантики `volatile`. Эта тема, несмотря на схожесть названия, относится к иной проблемной области и рассматривается на семинарах, посвящённых многопоточности.

## 2. Области памяти процесса JVM

Виртуальная машина разделяет используемую память на несколько областей, различающихся назначением, временем жизни размещаемых данных и способом управления. На верхнем уровне выделяются две категории: Heap, где размещаются данные приложения, и Non-Heap, где хранятся служебные данные и код.

```other
┌───────────────────────────────────────────────────────┐
│                    Память процесса JVM                │
│                                                       │
│  ┌────────────────────────┐  ┌─────────────────────┐  │
│  │         Heap           │  │      Non-Heap        │  │
│  │                        │  │                      │  │
│  │  Young Generation      │  │  Metaspace           │  │
│  │  ├─ Eden               │  │  Code Cache          │  │
│  │  ├─ Survivor 0         │  │                      │  │
│  │  └─ Survivor 1         │  │                      │  │
│  │                        │  │                      │  │
│  │  Old Generation        │  │                      │  │
│  └────────────────────────┘  └─────────────────────┘  │
│                                                       │
│  ┌─────────────────────────────────────────────────┐  │
│  │  Stack (по одному на каждый поток)               │  │
│  └─────────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────────┘
```

### 2.1. Куча (Heap)

Куча — область, в которой размещаются все объекты и массивы, явно или неявно создаваемые приложением. Это единственная область, управляемая сборщиком мусора, и, как правило, наибольшая по объёму. Размер кучи задаётся при запуске параметрами `-Xms` (начальный размер) и `-Xmx` (максимальный размер).

Куча разделяется на поколения, что обусловлено особенностями работы сборщика мусора и подробно рассматривается в разделе 6. Младшее поколение (Young Generation) состоит из области Eden, в которой размещаются вновь создаваемые объекты, и двух областей Survivor, используемых при перемещении выживших объектов. Старшее поколение (Old Generation) содержит объекты, просуществовавшие достаточно долго, чтобы быть перемещёнными из младшего поколения.

### 2.2. Non-Heap

Область Non-Heap содержит служебные данные, необходимые для работы виртуальной машины. Ниже перечислены составляющие её области.

**Metaspace.** Хранит метаданные классов: структуру полей и методов, байт-код, информацию о наследовании, а также статические поля классов и пул интернированных строк. Располагается в нативной памяти операционной системы и растёт по мере загрузки новых классов. В отличие от предшествовавшей ему области PermGen (удалённой в Java 8), Metaspace не имеет жёсткого фиксированного размера по умолчанию.

**Code Cache.** Хранит машинный код, порождённый JIT-компилятором в результате компиляции часто исполняемого байт-кода. Наличие этой области объясняет, почему повторно исполняемые участки программы со временем выполняются быстрее: интерпретируемый байт-код замещается оптимизированным машинным кодом.

**Thread stacks.** Совокупность стеков всех потоков приложения. Рассматриваются отдельно в разделе 2.3.

**Symbol table и String table.** Таблицы символических имён (имён классов, методов, полей) и ссылок на интернированные строки.

**Внутренние структуры сборщика мусора.** Служебные данные, необходимые сборщику для учёта объектов и планирования сборок.

За исключением Metaspace и Code Cache, перечисленные области, как правило, не требуют внимания разработчика в повседневной работе.

**Off-heap память.** Отдельного упоминания заслуживает возможность размещения данных вне управляемой памяти JVM. Класс `java.nio.ByteBuffer` посредством метода `allocateDirect` позволяет выделить так называемый прямой буфер (direct buffer), физически расположенный в нативной памяти операционной системы, а не в куче. Такие буферы не подлежат сборке мусора обычным образом и не учитываются в объёме кучи, задаваемом параметром `-Xmx`; освобождение занятой ими памяти происходит опосредованно, при утилизации управляющего объекта-обёртки. Прямые буферы применяются главным образом для эффективного ввода-вывода, поскольку позволяют избежать копирования данных между управляемой и нативной памятью при взаимодействии с операционной системой. Тот факт, что в Java, языке с автоматическим управлением памятью, существует штатный механизм ручного выделения памяти мимо виртуальной машины, нередко оказывается неочевидным. Детальное рассмотрение off-heap памяти и работы с `ByteBuffer` выносится на семинары, посвящённые вводу-выводу.

При условии, что приложение не занимается динамической генерацией и выгрузкой классов, состояние Non-Heap в долгосрочной перспективе близко к статичному.

### 2.3. Стек (Stack)

Стек создаётся отдельно для каждого потока и хранит кадры вызовов методов (stack frames). Каждый кадр содержит локальные переменные метода, промежуточные результаты вычислений и адрес возврата. При вызове метода в стек помещается новый кадр; при завершении метода кадр удаляется.

Существенно, что в стеке размещаются значения примитивных локальных переменных и ссылки на объекты, но не сами объекты. Объект, на который указывает локальная ссылочная переменная, находится в куче. Данное разделение является ключевым для понимания того, где физически располагаются данные, и подробно рассматривается в разделе 4.

Размер стека потока ограничен и задаётся параметром `-Xss`. Исчерпание стека, как правило вследствие чрезмерно глубокой или бесконечной рекурсии, приводит к возбуждению исключения `StackOverflowError`.

## 3. Устройство объекта в памяти

Объект в куче не сводится к совокупности своих полей. Помимо полезных данных, каждый объект содержит служебную информацию — заголовок, — необходимую виртуальной машине для управления объектом. Понимание структуры объекта позволяет оценивать реальный объём памяти, потребляемый структурами данных.

Объект состоит из следующих частей:

```other
┌──────────────────────────────────────────────┐
│  Заголовок объекта (object header)           │
│  ├─ Mark Word         (8 байт)               │  хеш-код, состояние блокировки,
│  │                                           │  возраст для сборщика мусора
│  └─ Class Pointer     (4 или 8 байт)         │  ссылка на метаданные класса
├──────────────────────────────────────────────┤
│  Поля экземпляра (instance fields)           │  собственно данные объекта
├──────────────────────────────────────────────┤
│  Выравнивание (padding)                      │  дополнение до кратности 8 байт
└──────────────────────────────────────────────┘
```

**Mark Word** содержит служебные сведения, относящиеся к самому объекту: хеш-код (вычисленный при первом обращении), состояние блокировки, используемое механизмами синхронизации, а также возраст объекта, учитываемый сборщиком мусора. Размер Mark Word составляет 8 байт на 64-разрядных платформах.

**Class Pointer** — ссылка на метаданные класса, размещённые в Metaspace. По этой ссылке виртуальная машина определяет тип объекта и получает информацию о его структуре и методах. При включённом сжатии указателей (Compressed Oops, применяемом по умолчанию для куч объёмом до 32 ГБ) размер составляет 4 байта, в противном случае — 8 байт.

**Выравнивание** дополняет размер объекта до величины, кратной 8 байтам. Это требование обусловлено особенностями адресации и повышает эффективность доступа к памяти.

Как следствие приведённой структуры, минимальный объект, не содержащий полей, занимает в памяти 16 байт: 12 байт заголовка (8 + 4 при сжатых указателях) и 4 байта выравнивания. Данное обстоятельство имеет практическое значение: создание большого числа мелких объектов сопряжено с существенными накладными расходами, не связанными с полезными данными.

## 4. Примитивы, ссылки и массивы

Размещение данных в памяти зависит как от их типа, так и от контекста объявления. Один и тот же примитивный тип может располагаться в различных областях в зависимости от того, является ли он локальной переменной или полем объекта.

### 4.1. Примитивные типы

Примитивные типы (`int`, `long`, `double`, `boolean` и прочие) хранят значение непосредственно, без обёртки в объект. Их размещение определяется контекстом:

- **Локальная переменная примитивного типа** размещается в стеке, в кадре текущего метода.
- **Поле объекта примитивного типа** размещается в куче, внутри объекта, которому принадлежит.

Размеры примитивных типов фиксированы спецификацией: `byte` — 1 байт, `short` и `char` — 2 байта, `int` и `float` — 4 байта, `long` и `double` — 8 байт. Тип `boolean` формально не имеет определённого размера и обычно занимает 1 байт.

### 4.2. Ссылочные типы

Переменная ссылочного типа хранит не сам объект, а ссылку на него — по существу, адрес объекта в куче. Сам объект всегда находится в куче, независимо от того, где расположена ссылающаяся на него переменная:

- **Локальная ссылочная переменная** размещается в стеке и содержит ссылку на объект в куче.
- **Ссылочное поле объекта** размещается в куче, внутри объекта-владельца, и содержит ссылку на другой объект в куче.

Размер ссылки составляет 4 байта при включённом сжатии указателей и 8 байт в противном случае.

Данное разграничение объясняет поведение объектов-обёрток. Значение `int` занимает 4 байта, тогда как эквивалентный ему `Integer` является полноценным объектом: он включает заголовок и, вследствие этого, занимает 16 байт, а обращение к нему производится через ссылку. Массовое применение объектов-обёрток вместо примитивов приводит к многократному увеличению потребления памяти.

### 4.3. Массивы

Массив в Java является объектом и размещается в куче. В дополнение к стандартному заголовку объекта массив содержит поле длины (length) и последовательность элементов:

```other
┌──────────────────────────────────┐
│  Заголовок объекта               │
│  ├─ Mark Word                    │
│  └─ Class Pointer                │
├──────────────────────────────────┤
│  Длина массива (length, 4 байта) │
├──────────────────────────────────┤
│  Элемент [0]                     │
│  Элемент [1]                     │
│  ...                             │
├──────────────────────────────────┤
│  Выравнивание                    │
└──────────────────────────────────┘
```

Заголовок массива несколько превосходит заголовок обычного объекта за счёт поля длины. Содержимое массива зависит от типа его элементов: массив примитивов хранит значения непосредственно, тогда как массив ссылочного типа хранит ссылки на объекты, расположенные в куче отдельно.

## 5. Многомерные массивы: разбор задачи

Изложенных сведений достаточно для разбора вопроса, представляющего практический интерес и позволяющего проверить понимание материала.

> **Что занимает больше памяти: `int[10][1000]` или `int[1000][10]`?**

На первый взгляд оба массива содержат по 10 000 элементов типа `int` и потому должны занимать одинаковый объём. Однако это заключение ошибочно, и причина состоит в способе представления многомерных массивов в Java.

В Java отсутствует непрерывный многомерный массив. Конструкция `int[][]` представляет собой **массив ссылок на одномерные массивы**. Внешний массив содержит ссылки, каждая из которых указывает на отдельный внутренний массив, размещённый в куче независимо.

```other
int[10][1000]:                       int[1000][10]:

внешний массив:                      внешний массив:
[ ref, ref, ..., ref ]  (10 шт.)     [ ref, ref, ..., ref ]  (1000 шт.)
    │    │        │                      │    │        │
    ▼    ▼        ▼                      ▼    ▼        ▼
 [1000 int] × 10 массивов            [10 int] × 1000 массивов
```

Для оценки объёма следует учесть накладные расходы каждого массива-объекта: заголовок (16 байт с учётом поля длины) плюс данные, дополненные выравниванием.

**Массив `int[10][1000]`:**

- внешний массив: заголовок и 10 ссылок — приблизительно 64 байта;
- 10 внутренних массивов, каждый из которых содержит заголовок и 1000 элементов по 4 байта — приблизительно 4016 байт;
- суммарно: 64 + 10 × 4016 ≈ **40 224 байта**.

**Массив `int[1000][10]`:**

- внешний массив: заголовок и 1000 ссылок — приблизительно 4016 байт;
- 1000 внутренних массивов, каждый из которых содержит заголовок и 10 элементов по 4 байта — приблизительно 56 байт;
- суммарно: 4016 + 1000 × 56 ≈ **60 016 байт**.

Таким образом, `int[1000][10]` занимает существенно больше памяти — приблизительно в полтора раза. Причина заключается в накладных расходах: этот массив порождает 1001 объект-массив против 11 у `int[10][1000]`, и каждый дополнительный массив привносит собственный заголовок и выравнивание. Полезные данные в обоих случаях идентичны — различие целиком обусловлено служебными структурами.

Из этого следует практический вывод: при работе с большими многомерными массивами предпочтительна организация, при которой число внутренних массивов минимально, а их размер максимален. Данное наблюдение проверяется экспериментально на практической части семинара с применением инструментов VisualVM и JOL.

## 6. Сборка мусора (обзорно)

Java относится к языкам с автоматическим управлением памятью: освобождение памяти, занятой более не используемыми объектами, выполняется виртуальной машиной без явного участия разработчика. Этот процесс называется сборкой мусора (garbage collection). Настоящий раздел даёт лишь общее представление о механизме; детальное рассмотрение алгоритмов и конкретных сборщиков выносится на последующие семинары.

### 6.1. Гипотеза о поколениях

В основе большинства сборщиков мусора лежит **слабая гипотеза о поколениях**, согласно которой подавляющее большинство объектов существуют крайне непродолжительное время. Временные объекты — итераторы, локальные переменные, промежуточные результаты вычислений — становятся ненужными практически сразу после создания. Меньшая часть объектов живёт дольше, и лишь незначительное число существует на протяжении всего времени работы приложения.

Из этой закономерности следует практический принцип: усилия по освобождению памяти целесообразно в первую очередь направлять на недавно созданные объекты, поскольку именно среди них сосредоточена основная масса подлежащих удалению. Данное соображение обусловливает разделение кучи на младшее и старшее поколения.

### 6.2. Малая и полная сборка

Разделение на поколения определяет и разделение процессов сборки:

- **Малая сборка (minor GC)** затрагивает только младшее поколение. Выполняется часто и удаляет основную массу мёртвых объектов. Объекты, пережившие ряд малых сборок, перемещаются в старшее поколение.
- **Полная сборка (full GC)** может затрагивать оба поколения. Выполняется тогда, когда объём доступной памяти близок к исчерпанию, и обходится существенно дороже.

Мёртвым считается объект, до которого невозможно добраться по цепочке ссылок от так называемых **корней (roots)** — ссылок из стеков потоков, статических полей классов и иных заведомо достижимых источников. Существенно, что наличие ссылок на объект само по себе не гарантирует его жизни: группа взаимно ссылающихся объектов, недостижимая от корней, корректно распознаётся как мусор.

### 6.3. Критерии оценки сборщиков

Работа сборщика мусора не бесплатна: она оплачивается ресурсами процессора и приостановками выполнения программы. Приостановки, в течение которых работа приложения полностью остановлена, называются паузами **stop-the-world (STW)**. Эффективность сборщика оценивается по нескольким критериям:

- **максимальная задержка** — наибольшая продолжительность одной паузы STW;
- **пропускная способность** — отношение времени полезной работы к общему времени с учётом простоев на сборку;
- **потребляемые ресурсы** — объём процессорного времени и дополнительной памяти, расходуемых сборщиком.

Одновременное улучшение всех трёх показателей практически недостижимо: сокращение задержек учащает сборки и снижает пропускную способность, а сохранение пропускной способности требует более сложных алгоритмов, повышающих потребление ресурсов. Различные сборщики оптимизируют разные показатели.

### 6.4. Сборщики HotSpot

Виртуальная машина HotSpot предоставляет несколько сборщиков мусора, ориентированных на различные сценарии применения. Приводимый перечень носит справочный характер; систематическое рассмотрение выносится на последующие семинары.

- **Serial GC** — простейший последовательный сборщик для приложений с малым объёмом данных.
- **Parallel GC** — сборщик, ориентированный на максимальную пропускную способность за счёт распараллеливания.
- **Concurrent Mark Sweep (CMS)** — сборщик, снижающий задержки за счёт выполнения части работы параллельно с приложением (признан устаревшим и удалён в современных версиях).
- **Garbage-First (G1)** — сборщик, создававшийся для замены CMS, ориентированный на серверные приложения с большими объёмами данных; сборщик по умолчанию в современных версиях.
- **ZGC** и **Shenandoah GC** — сборщики, удерживающие паузы на субмиллисекундном уровне независимо от размера кучи.
- **Epsilon GC** — сборщик, не выполняющий освобождения памяти, предназначенный для специальных сценариев и тестирования.

## 7. Инструменты анализа памяти

Для практического исследования размещения данных в памяти применяются два инструмента, используемых на семинаре.

**VisualVM** — внешний инструмент мониторинга, подключаемый к процессу JVM. Позволяет наблюдать состояние областей памяти в реальном времени, отслеживать динамику использования кучи и активность сборщика мусора. Следует учитывать, что сам факт мониторинга оказывает влияние на исследуемое приложение, привнося дополнительную нагрузку; при точных измерениях это влияние необходимо принимать во внимание.

**JOL (Java Object Layout)** — библиотека, предоставляющая точные сведения о размещении объектов в памяти. В отличие от мониторинга, JOL анализирует структуру конкретных объектов: размер, расположение полей, состав заголовка, объём выравнивания. Метод `ClassLayout.parseInstance()` возвращает раскладку отдельного объекта (shallow size), а `GraphLayout.parseInstance()` — суммарный объём объекта вместе со всеми достижимыми по ссылкам объектами (deep size). Именно последний метод позволяет наглядно измерить разницу между рассмотренными в разделе 5 многомерными массивами.

Совместное применение обоих инструментов иллюстрирует два уровня анализа: VisualVM даёт представление о поведении памяти в динамике на уровне всего процесса, тогда как JOL предоставляет точные сведения о структуре отдельных объектов.
