## 1. Почему нельзя просто засечь время до и после

Интуитивно понятный способ измерить производительность куска кода — обернуть его вызовами `System.nanoTime()`:

```java
long start = System.nanoTime();
doSomething();
long elapsed = System.nanoTime() - start;
System.out.println(elapsed + " ns");
```

На уровне отдельного, редкого вызова это работает достаточно честно. Но для измерения производительности кода,
который выполняется миллионы раз в реальном приложении (как раз кандидат на оптимизацию), такой замер систематически
вводит в заблуждение — по нескольким независимым причинам.

**JIT-компиляция и разогрев.** JVM не компилирует байт-код в машинный код сразу. Код сперва исполняется
интерпретатором, затем, если метод вызывается достаточно часто, JIT-компилятор компилирует его сначала облегчённым
компилятором C1, а для по-настоящему горячих методов — более агрессивно оптимизирующим C2 (многоуровневая
компиляция, tiered compilation). Первые тысячи вызовов метода в рамках одного запуска JVM исполняются медленнее, чем
этот же метод после полной JIT-компиляции. Замер, сделанный сразу после старта JVM, измеряет по большей части
издержки разогрева, а не установившуюся производительность кода.

**Удаление мёртвого кода (dead code elimination).** Если результат вызова `doSomething()` нигде не используется,
оптимизирующий компилятор вправе заметить это и не выполнять вычисление вовсе — реальный производственный код так
себя, конечно, не ведёт (результат обычно используется), но код бенчмарка, написанный небрежно, именно так и
выглядит: вызов ради вызова, без использования результата.

**Свёртка констант (constant folding).** Если входные данные бенчмарка — литералы или `final`-поля, известные
компилятору на этапе компиляции, весь расчёт может быть заранее вычислен один раз и превращён в константу, вместо
того чтобы измерять исполнение цикла заново на каждой итерации.

**Сборка мусора и внешние источники шума.** GC-паузы, планировщик операционной системы, другие процессы на машине —
всё это добавляет шум к отдельным измерениям, который простое усреднение по паре запусков не гасит статистически
надёжно.

Все эти проблемы — не гипотетические; на них наступают регулярно, и именно поэтому команда, разрабатывающая саму
HotSpot JVM, написала специальный инструмент, а не оставила задачу самодельным замерам через `nanoTime()`.

## 2. Что такое JMH и зачем он существует

[JMH](https://openjdk.org/projects/code-tools/jmh/) (Java Microbenchmark Harness) — официальный инструмент для
написания микробенчмарков на JVM, разработанный в рамках проекта OpenJDK той же командой, что работает над HotSpot.
Это обстоятельство существенно: авторы JMH знают из первых рук, какие именно оптимизации компилятора способны
исказить наивный замер, и проектировали инструмент так, чтобы систематически их нейтрализовать — прогревом перед
измерением, изоляцией через отдельные процессы JVM, принудительным использованием результата вычисления и другими
механизмами, разобранными в разделе 4.

## 3. Подключение и структура бенчмарка

Maven-зависимости:

```xml
<dependency>
    <groupId>org.openjdk.jmh</groupId>
    <artifactId>jmh-core</artifactId>
    <version>1.37</version>
</dependency>
<dependency>
    <groupId>org.openjdk.jmh</groupId>
    <artifactId>jmh-generator-annprocess</artifactId>
    <version>1.37</version>
    <scope>provided</scope>
</dependency>
```

`jmh-generator-annprocess` — annotation processor, генерирующий на этапе компиляции вспомогательный код бенчмарка
(обвязку для прогрева, циклов измерения и прочего) из аннотированных классов — писать эту обвязку руками не нужно.

Минимальный бенчмарк:

```java
@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.NANOSECONDS)
@State(Scope.Thread)
@Warmup(iterations = 5, time = 1)
@Measurement(iterations = 5, time = 1)
@Fork(1)
public class SimpleBenchmark {

    @Benchmark
    public int addition() {
        return 2 + 2;
    }
}
```

**`@State(Scope.Thread)`** определяет область видимости состояния бенчмарка: `Thread` — отдельный экземпляр на каждый
поток измерения, `Benchmark` — общий на все потоки, `Group` — общий в пределах группы потоков. Поля, помеченные
`@State`, — то самое место, куда стоит выносить входные данные, чтобы избежать constant folding (раздел 4).

**`@BenchmarkMode`** задаёт, что именно измеряется: `Throughput` (операций в единицу времени — чем больше, тем
лучше), `AverageTime` (среднее время одной операции — чем меньше, тем лучше), `SampleTime` (распределение времени с
перцентилями), `SingleShotTime` (время одного холодного вызова, без прогрева — для сценариев вроде замера времени
запуска).

**`@Warmup`** и **`@Measurement`** задают число итераций прогрева (результаты которых не учитываются в итоговой
оценке) и число итераций собственно измерения, а также длительность каждой итерации.

**`@Fork`** задаёт, сколько раз JMH перезапустит бенчмарк в отдельном процессе JVM. Каждый форк — это полностью новая
JVM со своим состоянием JIT-компилятора; несколько форков защищают от ситуации, когда конкретному запуску «повезло»
или «не повезло» с тем, как именно легли решения компилятора об инлайнинге и оптимизациях в этом конкретном процессе.

## 4. Ловушки бенчмаркинга и как их избежать

**Мёртвый код.** Если результат вычисления внутри `@Benchmark`-метода нигде не используется, JMH сам предупредит об
этом при запуске (`CONSTANT FOLDED` / `DEAD CODE` warning), но правильный способ подстраховаться — либо возвращать
результат из метода бенчмарка (как в примере выше — JMH сам «использует» возвращаемое значение), либо, если операций
несколько, передавать результаты в `Blackhole`:

```java
@Benchmark
public void multipleOperations(Blackhole blackhole) {
    blackhole.consume(operationOne());
    blackhole.consume(operationTwo());
}
```

`Blackhole.consume(...)` — специальный метод, который JMH-рантайм гарантированно не даст оптимизатору выбросить как
«неиспользуемый», в отличие от обычного присваивания в локальную переменную, которую компилятор вправе счесть мёртвой.

**Свёртка констант.** Входные данные бенчмарка нужно объявлять как поля `@State`-класса, а не как локальные литералы
внутри метода — компилятор не может заранее вычислить значение поля, зависящего от рантайм-конфигурации, так же
уверенно, как значение локального литерала:

```java
@State(Scope.Thread)
public static class BenchmarkState {
    String type = "CRYPTO"; // не final, не литерал в теле метода — компилятору сложнее свернуть в константу
}

@Benchmark
public int throwWithStackTrace(BenchmarkState state) {
    try {
        validateOrThrow(state.type);
        return 0;
    } catch (IllegalArgumentException e) {
        return 1;
    }
}
```

**Недостаточный прогрев.** Слишком мало итераций `@Warmup` — и измерение фиксирует ещё не до конца оптимизированный
компилятором код, что особенно критично при сравнении «до» и «после» — если оба варианта прогреты недостаточно,
разница между ними может быть артефактом разного порядка JIT-компиляции, а не реальной разницей в производительности.

**Один форк.** `@Fork(1)` (или тем более отсутствие форка) не даёт статистической уверенности в стабильности
результата между независимыми запусками JVM. Для чернового исследования одного форка обычно достаточно; для выводов,
на которые будут опираться архитектурные решения, разумно использовать 3–5 форков.

## 5. Как читать результаты

Типичный вывод JMH в режиме `AverageTime`:

```
Benchmark                                    Mode  Cnt    Score    Error  Units
ExceptionOverheadBenchmark.returnCode        avgt   25    2.104 ±  0.031  ns/op
ExceptionOverheadBenchmark.throwWithoutTrace avgt   25   18.532 ±  0.842  ns/op
ExceptionOverheadBenchmark.throwWithTrace    avgt   25  412.671 ± 15.203  ns/op
```

**`Score`** — итоговая оценка (здесь — среднее время одной операции в наносекундах, поскольку режим `avgt` и единица
`ns/op`). **`Error`** — доверительный интервал вокруг оценки (по умолчанию 99,9%): если интервалы двух бенчмарков не
пересекаются, разница между ними статистически значима, а не шум замера. **`Cnt`** — сколько отдельных измерений
легло в основу оценки (итерации × форки). Числа в примере выше иллюстративны — не результат реального запуска, а
ожидаемая по порядку величины картина: код возврата дешевле исключения без трассировки стека на порядок, а исключение
с трассировкой — дороже последнего ещё на один-два порядка.

## 6. Пример: стоимость исключений

Полный код бенчмарка, сравнивающего код возврата, исключение с трассировкой стека и исключение без неё, — на
семинаре, этап «JMH: замеряем стоимость исключений». Там же — интерпретация результатов и вывод о том, когда
отключение трассировки стека (`writableStackTrace = false`) оправдано, а когда является преждевременной оптимизацией.
Общая теория стоимости исключений — в материале «Исключения в Java», раздел про стоимость исключений.

## 7. Микробенчмарк vs профилировщик

JMH отвечает на узкий вопрос: «какой из двух (или нескольких) конкретных фрагментов кода быстрее при прочих равных».
Это ценно, когда альтернативы уже сформулированы (throw с трассировкой или без, `ArrayList` или `LinkedList` для
конкретного паттерна доступа) и нужно решить между ними на основе цифр, а не интуиции.

Профилировщик (например, встроенный в JDK Java Flight Recorder или сторонний async-profiler) отвечает на другой
вопрос: «где в реальном, работающем под реальной нагрузкой приложении в принципе тратится время». Профилировщик не
требует заранее знать, что именно сравнивать, — он показывает, какие методы горячие, не выдвигая заранее гипотезы.
Разумный порядок работы: сначала профилировщиком найти, что вообще стоит оптимизировать, — и только затем, если
решение неочевидно, сузить вопрос до конкретной альтернативы и проверить её через JMH. Микробенчмарк, написанный до
профилирования, рискует тщательно измерить оптимизацию кода, который в реальном приложении и так не является узким
местом.
