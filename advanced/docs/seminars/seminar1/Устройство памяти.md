# Устройство памяти в JVM

---

## 1. Карта памяти процесса

```
┌───────────────────────────────────────────────────────┐
│                    Память процесса JVM                │
│                                                       │
│  ┌────────────────────────┐  ┌─────────────────────┐  │
│  │         Heap           │  │      Non-Heap       │  │
│  │  объекты и массивы     │  │  служебные данные,  │  │
│  │  управляется GC        │  │  код, метаданные    │  │
│  └────────────────────────┘  └─────────────────────┘  │
│                                                       │
│  ┌─────────────────────────────────────────────────┐  │
│  │  Stack — по одному на каждый поток              │  │
│  │  кадры вызовов, локальные переменные, ссылки    │  │
│  └─────────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────────┘
```

Три области: **Heap** — данные приложения, **Non-Heap** — служебные данные и код, **Stack** — контекст исполнения
потоков.

> Речь об устройстве памяти процесса. Java Memory Model (многопоточность, `volatile`, happens-before) — отдельная тема
> на будущих семинарах.

---

## 2. Heap — где живут объекты

- Все объекты и массивы приложения
- Единственная область под управлением GC, обычно наибольшая
- Размер: `-Xms` (начальный), `-Xmx` (максимальный)

```
Heap
├─ Young Generation     недавно созданные объекты
│  ├─ Eden               здесь возникают новые объекты
│  ├─ Survivor 0
│  └─ Survivor 1
└─ Old Generation       долгоживущие объекты
```

Большинство объектов живут недолго и удаляются в Young Generation. Пережившие несколько сборок — переходят в Old
Generation.

---

## 3. Сборка мусора

Классический алгоритм — три фазы:

```
Mark     →  Sweep      →  Compact
пометить    удалить        уплотнить
живые       мёртвые        оставшиеся
```

Мёртвый объект — недостижимый от корней (стеки потоков, статические поля). Группа взаимно ссылающихся, но недостижимых
объектов — тоже мусор.

Сборка расходует CPU и требует пауз **stop-the-world (STW)**. Баланс между длиной пауз, пропускной способностью и
ресурсами — всё сразу оптимизировать нельзя.

**Сборщики HotSpot:**

- **Serial GC** — однопоточный. Малые приложения, ограниченные ресурсы.
- **Parallel GC** — максимальная пропускная способность. Пакетная обработка.
- **G1** — по умолчанию. Компромисс пауз и throughput, большие кучи.
- **ZGC / Shenandoah** — субмиллисекундные паузы независимо от размера кучи.
- **Epsilon GC** — не освобождает память. Тестирование, спец. сценарии.

## 4. Non-Heap — служебные данные и код

- **Metaspace** — метаданные классов, байт-код, статические поля, пул строк. В нативной памяти, растёт при загрузке
  классов.
- **Code Cache** — машинный код от JIT. Поэтому «горячий» код со временем ускоряется.
- **Thread stacks, таблицы имён, структуры GC** — в повседневной работе внимания не требуют.

> **Off-heap:** `ByteBuffer.allocateDirect()` выделяет память мимо JVM — не под GC, не в счёт `-Xmx`. Для эффективного
> IO. Подробнее — на семинарах по IO.

---

## 5. Stack — контекст исполнения потоков

- Отдельный для каждого потока
- Кадры вызовов: локальные переменные, промежуточные результаты, адрес возврата
- Кадр добавляется при вызове, удаляется при выходе

```
В стеке лежат:    значения примитивов + ссылки на объекты
В стеке НЕ лежат: сами объекты — они всегда в Heap
```

Размер ограничен (`-Xss`). Глубокая рекурсия → `StackOverflowError`.

## 6. Из чего складывается вес объекта

```
┌────────────────────────────────────┐
│  Заголовок объекта                 │
│  ├─ Mark Word      (8 байт)        │  хеш-код, блокировки, возраст для GC
│  └─ Class Pointer  (4 или 8 байт)  │  ссылка на класс в Metaspace
├────────────────────────────────────┤
│  Поля объекта                      │  собственно данные
├────────────────────────────────────┤
│  Выравнивание (padding)            │  дополнение до кратности 8 байт
└────────────────────────────────────┘
```

- **Class Pointer** — 4 байта при Compressed Oops (куча до 32 ГБ), иначе 8
- Даже объект **без полей** = 16 байт (12 заголовок + 4 выравнивание)
- Множество мелких объектов = существенные накладные расходы: заголовок платится за каждый

---

## 7. Где что хранится

| Что                            | Где                                |
|--------------------------------|------------------------------------|
| `int x` — локальная переменная | Stack                              |
| `int x` — поле объекта         | Heap, внутри объекта               |
| `Integer x`                    | Heap (обёртка, 16 байт + ссылка)   |
| Ссылка на объект               | Stack или Heap (если поле)         |
| Сам объект                     | всегда Heap                        |
| `int[]`                        | Heap (заголовок + length + данные) |

Один и тот же `int` — в разных местах в зависимости от контекста. Обёртка `Integer` вместо примитива многократно
увеличивает потребление памяти.

## 8. Задача: два массива

```
Что занимает больше памяти?

        int[10][100]        vs        int[100][10]
```

Оба — по 1000 элементов `int`. Весят одинаково? Нет.

В Java нет непрерывных многомерных массивов. `int[][]` — это **массив ссылок на массивы**:

```
int[10][100]:                    int[100][10]:

[ref, ref, ..., ref]  (10)       [ref, ref, ..., ref]  (100)
  │    │        │                  │    │         │
  ▼    ▼        ▼                  ▼    ▼         ▼
[100 int] × 10                   [10 int] × 100
```

Каждый внутренний массив — отдельный объект с заголовком. Вопрос сводится к: **сколько объектов создаётся?**

---

## 9. Разбор

**`int[10][100]`** → 1 внешний + 10 внутренних = 11 объектов:

```
внешний:    заголовок + 10 ссылок          ≈   64 байта
внутренние: (заголовок + 100×4) × 10        ≈ 4160 байт
──────────────────────────────────────────────────────
итого                                       ≈ 4224 байта
```

**`int[100][10]`** → 1 внешний + 100 внутренних = 101 объект:

```
внешний:    заголовок + 100 ссылок          ≈  416 байт
внутренние: (заголовок + 10×4) × 100         ≈ 5600 байт
──────────────────────────────────────────────────────
итого                                       ≈ 6016 байт
```

`int[100][10]` тяжелее в ~1.5 раза. Данных поровну — разница в накладных расходах: 101 объект против 11, каждый лишний
массив = заголовок + выравнивание.

> **Вывод:** больше многомерный массив → меньше внутренних массивов, больше их размер. Проверим на практике — VisualVM,
> затем JOL.

---

## 10. Итоги

- **Heap** — объекты и массивы, под GC. **Non-Heap** — код и метаданные. **Stack** — по одному на поток.
- В стеке — примитивы и ссылки; **объекты всегда в Heap**.
- Вес объекта = заголовок (12 байт) + поля + выравнивание. Минимум — 16 байт.
- Многомерный массив = массив ссылок на массивы. Число объектов важнее, чем кажется.

Далее — практика: VisualVM и JOL.
