### Этап 2. Подключение picocli (~5 мин)

Проект переходит от точки входа `demo.Main` к полноценному CLI. В качестве библиотеки для разбора аргументов командной
строки используется [picocli](https://picocli.info/) — позволяет декларативно описывать команды и опции через аннотации,
не реализуя разбор аргументов вручную.

```xml

<dependency>
    <groupId>info.picocli</groupId>
    <artifactId>picocli</artifactId>
    <version>4.7.7</version>
</dependency>
```

Краткий обзор основных классов picocli, без разбора отдельного демонстрационного примера:

```java
/**
 * Основные типы picocli, используемые в проекте:
 *
 * @Command       — аннотация класса-команды (имя, описание, вложенные subcommands)
 * @Option         — аннотация именованного параметра (например, --type)
 * @Parameters     — аннотация позиционного параметра (например, тикер)
 * Callable<Integer> — интерфейс, реализуемый командой; call() возвращает код завершения
 * CommandLine     — точка запуска: разбирает args и вызывает call() нужной команды
 */
```

### Этап 3. Обсуждение: какие паттерны GoF применимы к CLI (~10 мин)

Формат — открытое обсуждение с группой, а не лекция. Вопросы для обсуждения:

- Какие паттерны проектирования уже знакомы студентам (из прошлого опыта, других курсов, самостоятельного изучения)?
- Как устроена обработка команды в CLI-приложении: что должно происходить, когда пользователь вводит `search AAPL`?
- Как устроено создание объектов `Stock`, `Bond`, `Etf`, когда параметры инструмента приходят не из кода, а из строки,
  введённой пользователем?
- Как отфильтровать список инструментов по типу так, чтобы добавление нового типа инструмента не требовало правки кода
  фильтрации?
- Как перебрать все инструменты в хранилище, не раскрывая наружу, как именно они там хранятся?
- Как добавить к любому методу репозитория побочное поведение (замер времени выполнения, логирование), не трогая код
  самого репозитория?

Ожидаемое направление ответов и разбор кандидатов:

| **Задача**                                                        | **Паттерн-кандидат** |
|-------------------------------------------------------------------|----------------------|
| Обработка пользовательской команды                                | Command              |
| Создание объекта инструмента по строковому типу                   | Factory Method       |
| Фильтрация списка инструментов без хрупкого сравнения по типу     | Strategy             |
| Обход инструментов в хранилище без раскрытия внутренней структуры | Iterator             |
| Побочное поведение вокруг вызовов метода без изменения его кода   | Proxy                |

Обсуждается, что picocli уже частично реализует Command на уровне библиотеки — каждая аннотированная классом `@Command`
команда, реализующая `Callable`, фактически является объектом-командой с методом `call()` в роли `execute()`. Это
наблюдение используется как переход к совместной реализации: паттерн не придумывается с нуля, а узнаётся в уже
используемом инструменте.

Iterator и Command фиксируются в списке кандидатов, но реализуются не сразу — оба паттерна разбираются на этапе 4,
вместе с Factory Method и Strategy, каждый в свою очередь. Proxy фиксируется, но подробно разбирается только на
этапе 5, после того как остальные паттерны уже реализованы и есть на чём его продемонстрировать.

### Этап 4. Совместная реализация: Factory Method, Strategy, Iterator, Command (~60–70 мин)

**Factory Method: создание инструмента по типу.** Пользователь вводит тип инструмента строкой (`STOCK`, `BOND`, `ETF`);
код, создающий объект, не должен разрастаться в `if/else` при добавлении нового типа:

```java
public interface InstrumentFactory {
    String getSupportedType();

    Instrument create(String ticker, String name, Currency currency);
}
```

**Наивный вариант — самрегистрация через статический блок**, как в базовом треке, по аналогии с историческим
`DriverManager` в JDBC:

```java
public final class InstrumentFactories {
    private static final Map<String, InstrumentFactory> REGISTRY = new HashMap<>();

    public static void register(String type, InstrumentFactory factory) {
        REGISTRY.put(type.toUpperCase(), factory);
    }

    public static Instrument create(String type, String ticker, String name, Currency currency) {
        InstrumentFactory factory = REGISTRY.get(type.toUpperCase());
        if (factory == null) {
            throw new IllegalArgumentException("Unknown instrument type: " + type);
        }
        return factory.create(ticker, name, currency);
    }
}

public class StockFactory implements InstrumentFactory {

    static {
        InstrumentFactories.register("STOCK", new StockFactory());
    }

    @Override
    public String getSupportedType() {
        return "STOCK";
    }

    @Override
    public Instrument create(String ticker, String name, Currency currency) {
        return new Stock(ticker, name, currency, "Unspecified", BigDecimal.ZERO);
    }
}
```

> **Обсуждение: это не настоящий OCP.** Реестр не содержит статического перечисления всех фабрик, и это выглядит как
> соблюдение OCP — но откроем `InstrumentFactories` внимательнее: статический блок класса выполняется только при
> загрузке `StockFactory` JVM, а без явного обращения к классу она может не загрузиться никогда. Значит,
> `InstrumentFactories` вынуждена сама форсировать загрузку каждой фабрики (`Class.forName(StockFactory.class...)` и
> так далее для каждого нового типа) — а это и есть правка существующего класса при добавлении `CryptoFactory`.
> Формально утверждение «ни `InstrumentFactories`, ни существующие фабрики не изменяются» не выдерживает проверки.

**Рефакторинг — `ServiceLoader`.** Корень проблемы выше — `InstrumentFactories` обязана знать обо всех своих фабриках
заранее, чтобы форсировать их загрузку. `java.util.ServiceLoader` — стандартный механизм JDK (реализация паттерна
*Service Provider Interface*, SPI), который снимает именно эту обязанность: код, которому нужны реализации
интерфейса (в терминах SPI — *service*), не перечисляет их в Java-коде вообще, а просто спрашивает у JVM «дай мне все
известные реализации `InstrumentFactory`» — а откуда JVM знает про эти реализации, решает не Java-код, а обычный
текстовый файл на classpath. Каждый класс-реализация (*provider*) должен иметь публичный конструктор без аргументов
— по нему `ServiceLoader` и создаёт экземпляр через рефлексию, без необходимости в статическом блоке или ручном
вызове `new`. Ровно по этой причине `DriverManager` в JDBC 4.0+ сам отказался от исторического
`Class.forName("...Driver")` в пользу `ServiceLoader` — обязанность форсировать загрузку каждого драйвера с
разработчика приложения была снята:

```java
public final class InstrumentFactories {
    private static final Map<String, InstrumentFactory> REGISTRY = loadRegistry();

    public static Instrument create(String type, String ticker, String name, Currency currency) {
        InstrumentFactory factory = REGISTRY.get(type.toUpperCase());
        if (factory == null) {
            throw new IllegalArgumentException("Unknown instrument type: " + type);
        }
        return factory.create(ticker, name, currency);
    }

    private static Map<String, InstrumentFactory> loadRegistry() {
        Map<String, InstrumentFactory> registry = new HashMap<>();
        for (InstrumentFactory factory : ServiceLoader.load(InstrumentFactory.class)) {
            registry.put(factory.getSupportedType().toUpperCase(), factory);
        }
        return registry;
    }
}

public class StockFactory implements InstrumentFactory {

    @Override
    public String getSupportedType() {
        return "STOCK";
    }

    @Override
    public Instrument create(String ticker, String name, Currency currency) {
        return new Stock(ticker, name, currency, "Unspecified", BigDecimal.ZERO);
    }
}
```

Статический блок и самрегистрация исчезают целиком — `ServiceLoader` сам находит все реализации `InstrumentFactory`
по файлу `META-INF/services/academy.backend.market_pulse.factory.InstrumentFactory`
([итоговый файл](../../../src/main/resources/META-INF/services/academy.backend.market_pulse.factory.InstrumentFactory)),
перечисляющему полные имена классов-фабрик, по одному на строку:

```other
academy.backend.market_pulse.factory.StockFactory
academy.backend.market_pulse.factory.BondFactory
academy.backend.market_pulse.factory.EtfFactory
```

Теперь добавление `CryptoFactory` — это новый класс плюс одна новая строка в текстовом файле ресурсов; ни
`InstrumentFactories`, ни `StockFactory`/`BondFactory`/`EtfFactory` не трогаются — вот это уже настоящий OCP.

**Strategy: фильтрация инструментов по типу.** Наивный способ реализовать фильтр `--type` в команде `list` — сравнение
строки с именем класса напрямую в коде команды. Такое условие разрастётся при появлении второго правила отбора. Strategy
инкапсулирует условие отбора в отдельный, взаимозаменяемый объект:

```java
public interface InstrumentFilter {
    boolean matches(Instrument instrument);
}

public class TypeFilter implements InstrumentFilter {
    private final String type;

    public TypeFilter(String type) {
        this.type = type;
    }

    @Override
    public boolean matches(Instrument instrument) {
        return instrument.getClass().getSimpleName().equalsIgnoreCase(type);
    }
}

public class NoOpFilter implements InstrumentFilter {
    @Override
    public boolean matches(Instrument instrument) {
        return true;
    }
}
```

Добавление нового способа отбора (например, `PriceAboveFilter`) сводится к реализации нового `InstrumentFilter` — код,
перебирающий инструменты, не изменяется.

**Iterator: обход `InstrumentRepository`.** В проекте появляется класс `InstrumentRepository` — обёртка над внутренним
массивом `Instrument[]` фиксированного размера, с методом добавления записей (`add(Instrument instrument)`) — без него
нечем будет наполнить репозиторий для проверки перебора.

Задача — перебрать все инструменты репозитория, не раскрывая наружу сам массив (то есть без метода вида
`getAll(): Instrument[]`). Сначала — собственный интерфейс перебора, до знакомства с `java.util.Iterator`, чтобы
показать, что `for-each` не магия, а конкретный механизм:

```java
public interface InstrumentIterator {
    boolean hasNext();

    Instrument next();
}
```

Проверка классическим циклом:

```java
void example() {
    InstrumentIterator iterator = repository.iterator();
    while (iterator.hasNext()) {
        Instrument instrument = iterator.next();
        System.out.println(instrument.getDescription());
    }
}
```

Далее — переход от собственного интерфейса к `java.util.Iterator<Instrument>` и `Iterable<Instrument>`. Одновременно
`InstrumentRepository` выделяется в интерфейс, а текущая реализация переименовывается в `InMemoryInstrumentRepository` —
это понадобится на этапе 5 для JDK Dynamic Proxy, которому для работы требуется интерфейс:

```java
public interface InstrumentRepository extends Iterable<Instrument> {
    void add(Instrument instrument);

    Instrument findByTicker(String ticker);
}

public class InMemoryInstrumentRepository implements InstrumentRepository {
    private Instrument[] instruments = new Instrument[100];
    private int size = 0;

    @Override
    public void add(Instrument instrument) {
        instruments[size++] = instrument;
    }

    @Override
    public Instrument findByTicker(String ticker) {
        for (Instrument instrument : this) {
            if (instrument.getTicker().equalsIgnoreCase(ticker)) {
                return instrument;
            }
        }
        return null;
    }

    @Override
    public Iterator<Instrument> iterator() {
        return new Iterator<>() {
            private int cursor = 0;

            @Override
            public boolean hasNext() {
                return cursor < size;
            }

            @Override
            public Instrument next() {
                if (!hasNext()) {
                    throw new NoSuchElementException();
                }
                return instruments[cursor++];
            }
        };
    }
}
```

Далее — явная демонстрация, что `for-each` является синтаксическим сахаром над `Iterator`, на паре эквивалентных
фрагментов:

```java
void example() {
// то, что пишет программист
    for (Instrument instrument : repository) {
        System.out.println(instrument.getDescription());
    }

// во что это разворачивается компилятором
    Iterator<Instrument> iterator = repository.iterator();
    while (iterator.hasNext()) {
        Instrument instrument = iterator.next();
        System.out.println(instrument.getDescription());
    }
}
```

Реализация `Iterable<Instrument>` — условие, необходимое и достаточное для использования `for-each`; без него компилятор
не примет объект в конструкции `for (... : repository)`.

**Command: CLI-команды поверх picocli.** Первые доменные команды приложения — `SearchCommand` и `AddCommand`, каждая
инкапсулирует один сценарий использования и делегирует получение данных доменному слою, не смешивая парсинг аргументов
с бизнес-логикой (связь с SRP):

```java

@Command(name = "search", description = "Поиск инструмента по тикеру")
public class SearchCommand implements Callable<Integer> {

    @Parameters(index = "0", description = "Тикер инструмента")
    private String ticker;

    private final InstrumentRepository repository;

    public SearchCommand(InstrumentRepository repository) {
        this.repository = repository;
    }

    @Override
    public Integer call() {
        Instrument instrument = repository.findByTicker(ticker);
        if (instrument == null) {
            System.out.println("Инструмент не найден: " + ticker);
            return 1;
        }
        System.out.println(instrument.getDescription());
        return 0;
    }
}
```

Команда `add`, использующая `InstrumentFactories`, показанную выше, — связывает Factory Method и Command: создание
объекта остаётся ответственностью фабрики, команда лишь разбирает аргументы и передаёт их дальше:

```java

@Command(name = "add", description = "Добавление инструмента в репозиторий")
public class AddCommand implements Callable<Integer> {

    @Parameters(index = "0", description = "Тип инструмента (STOCK, BOND, ETF)")
    private String type;

    @Parameters(index = "1", description = "Тикер инструмента")
    private String ticker;

    @Parameters(index = "2", description = "Название инструмента")
    private String name;

    @Parameters(index = "3", description = "Валюта инструмента (RUB, USD, EUR)")
    private Currency currency;

    private final InstrumentRepository repository;

    public AddCommand(InstrumentRepository repository) {
        this.repository = repository;
    }

    @Override
    public Integer call() {
        Instrument instrument = InstrumentFactories.create(type, ticker, name, currency);
        repository.add(instrument);
        System.out.println("Добавлено: " + instrument.getDescription());
        return 0;
    }
}
```

Команда `list`, использующая итератор, разобранный выше (уже в форме `for-each`, поскольку сахар разобран), и
`InstrumentFilter` из Strategy вместо прямого сравнения строк:

```java

@Command(name = "list", description = "Список инструментов")
public class ListCommand implements Callable<Integer> {

    @Option(names = "--type", description = "Фильтр по типу инструмента")
    private String type;

    private final InstrumentRepository repository;

    public ListCommand(InstrumentRepository repository) {
        this.repository = repository;
    }

    @Override
    public Integer call() {
        InstrumentFilter filter = (type == null) ? new NoOpFilter() : new TypeFilter(type);

        for (Instrument instrument : repository) {
            if (filter.matches(instrument)) {
                System.out.println(instrument.getDescription());
            }
        }
        return 0;
    }
}
```

Точка входа приложения создаёт репозиторий и регистрирует команды в корневом объекте picocli:

```java

@Command(name = "market-pulse")
public class MarketPulseCli implements Runnable {
    @Override
    public void run() {
        System.out.println("Используйте --help для списка команд");
    }
}

public class Main {
    public static void main(String[] args) {
        InstrumentRepository repository = new InMemoryInstrumentRepository();
        CommandLine cli = new CommandLine(new MarketPulseCli())
                .addSubcommand(new SearchCommand(repository))
                .addSubcommand(new AddCommand(repository))
                .addSubcommand(new ListCommand(repository));
        int exitCode = cli.execute(args);
        System.exit(exitCode);
    }
}
```

Команды регистрируются вручную через `addSubcommand`, а не через атрибут `subcommands` аннотации `@Command`: у
`SearchCommand`, `AddCommand` и `ListCommand` нет конструктора без аргументов, и picocli не смог бы создать их
самостоятельно по одному классу — каждой команде нужен уже готовый `repository`. Это же соображение пригодится на
этапе 5: и `InMemoryInstrumentRepository`, и его будущий Proxy-заместитель реализуют один и тот же интерфейс
`InstrumentRepository`, поэтому подмену в `Main` можно будет сделать без изменения кода команд.

Стоит отдельно отметить со студентами, что `TypeFilter` внутри по-прежнему сравнивает `getClass().getSimpleName()` —
Strategy убрал разрастание условий в `ListCommand`, но не саму хрупкость сравнения по имени класса. На продвинутом
треке это тем более уместно проговорить явно: более надёжная реализация (например, сверка через `InstrumentFactories`
или явный признак типа на `Instrument`) может быть предложена студентам как самостоятельное улучшение за рамками
занятия.

### Этап 5. Proxy: обсуждение и демонстрация одного из вариантов (~20–25 мин)

**Задача.** Представим ситуацию: приложение обращается к смежной системе — например, к T-Invest API за котировками —
и в последнее время такие обращения стали выполняться заметно дольше обычного. Чтобы разобраться в причине, нужно
замерять время выполнения каждого метода `InstrumentRepository` и логировать его вместе с параметрами вызова.

Самый прямолинейный вариант — просто прописать замер времени и логирование в начале и в конце каждого метода, который
делает такое обращение. Задачу это решает, но нарушает сразу два принципа: SRP (метод отвечает и за собственную логику,
и за диагностику) и DRY (пара «замерить время → залогировать» дублируется в каждом методе).

**Обсуждение возможных решений.** Естественный следующий шаг — вынести эту сквозную логику в отдельный
объект-заместитель, которому делегируется вызов. Три варианта, по нарастанию автоматизации — обсуждаются устно, без
кода:

- **Самописный (статический) прокси** — класс вручную реализует тот же интерфейс, что и целевой объект, и оборачивает
  каждый вызов. Дублирование никуда не девается: оно просто переезжает из целевого класса в прокси, а добавление
  нового метода в интерфейс требует не забыть дописать его и здесь.
- **JDK Dynamic Proxy** (`java.lang.reflect.Proxy`) — генерирует класс-заместитель во время выполнения; единственный
  `InvocationHandler` перехватывает вызовы любого метода, без ручного дублирования сигнатур. Ограничение — работает
  только с интерфейсами.
- **CGLIB** — генерирует подкласс целевого класса во время выполнения, интерфейс не обязателен. Ограничение — не может
  проксировать `final`-классы и `final`-методы, требует дополнительной зависимости.

**Показ материала.** Перед переходом к коду — материал [«Паттерн Proxy»](./Паттерн%20Proxy.md) целиком, включая
сравнительную таблицу трёх вариантов по критериям (необходимость интерфейса, работа с `final`, дополнительные
зависимости, накладные расходы на вызов).

**Демонстрация в коде.** Один из двух автоматизированных вариантов реализуется на проекте вживую — JDK Dynamic Proxy
или CGLIB, на выбор семинариста; самописный прокси кодом не демонстрируется, его роль — обосновать переход к
автоматизации. Ниже приведён код обоих вариантов, чтобы семинарист мог заранее выбрать и подготовить один из них.

*Если выбран JDK Dynamic Proxy* — `java.lang.reflect.Proxy` генерирует класс, реализующий заданные интерфейсы, во время
выполнения; вызовы перехватываются единственным `InvocationHandler`, без ручного дублирования сигнатур методов:

```java
InstrumentRepository target = new InMemoryInstrumentRepository();

InstrumentRepository proxy = (InstrumentRepository) Proxy.newProxyInstance(
        InstrumentRepository.class.getClassLoader(),
        new Class<?>[]{InstrumentRepository.class},
        (Object p, Method method, Object[] args) -> {
            long start = System.nanoTime();
            Object result = method.invoke(target, args);
            System.out.printf("%s() выполнен за %d нс%n", method.getName(), System.nanoTime() - start);
            return result;
        });
```

Обсуждается ограничение: сгенерированный класс наследует `java.lang.reflect.Proxy`, поэтому может реализовывать только
интерфейсы — прокси для класса без интерфейса этим способом не построить.

*Если выбран CGLIB* — работает подстановкой подкласса и не требует интерфейса; годится для случаев, когда прокси нужен
непосредственно для `InMemoryInstrumentRepository`, минуя интерфейс:

> **Для семинариста:** CGLIB генерирует байт-код через рефлексию поверх внутренностей JDK и давно не обновлялся под
> модульную систему Java. На JDK 21+ создание прокси через `Enhancer` иногда падает с `InaccessibleObjectException`
> из-за строгой инкапсуляции модулей (JEP 396/403). Если это случится на чьей-то машине — то же решение, что и для
> JOL на семинаре 1: добавить `--add-opens java.base/java.lang=ALL-UNNAMED` в VM options конфигурации запуска.

```xml

<dependency>
    <groupId>cglib</groupId>
    <artifactId>cglib</artifactId>
    <version>3.3.0</version>
</dependency>
```

```java
void example() {
    Enhancer enhancer = new Enhancer();
    enhancer.setSuperclass(InMemoryInstrumentRepository.class);
    enhancer.setCallback((MethodInterceptor) (obj, method, args, methodProxy) -> {
        long start = System.nanoTime();
        Object result = methodProxy.invokeSuper(obj, args);
        System.out.printf("%s() выполнен за %d нс%n", method.getName(), System.nanoTime() - start);
        return result;
    });

    InstrumentRepository repository = (InstrumentRepository) enhancer.create();
}
```

Обсуждается ограничение: CGLIB генерирует подкласс целевого класса, поэтому не может проксировать `final`-классы и
`final`-методы — переопределить их в сгенерированном подклассе невозможно.

> **Self-invocation — ограничение, общее для обоих автоматизированных вариантов.** Демонстрация выше замеряет
> `proxy.findByTicker(...)`, вызванный снаружи, — и это действительно перехватывается и логируется. Но внутри
> `InMemoryInstrumentRepository.findByTicker` перебор идёт через `for (Instrument instrument : this)`, что
> разворачивается в вызов `this.iterator()` — а `this` внутри метода ссылается на реальный объект, а не на прокси.
> Значит, этот внутренний вызов `iterator()` не попадёт ни в `InvocationHandler`, ни в `MethodInterceptor`:
> перехватываются только вызовы **извне**, через ссылку-прокси, а не вызовы `this.метод()` изнутри самого объекта.
> Это ровно та причина, по которой в Spring `@Transactional`/`@Cacheable` не срабатывает при вызове аннотированного
> метода из другого метода того же бина (self-invocation) — оба механизма проксирования оборачивают объект снаружи,
> а не перехватывают вызовы изнутри его собственного кода. Стоит явно проговорить это со студентами — инсайт,
> который отличает уверенное понимание AOP от поверхностного.
