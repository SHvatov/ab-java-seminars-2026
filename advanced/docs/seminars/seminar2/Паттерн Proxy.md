# Паттерн Proxy

---

## Задача

Приложение обращается к смежной системе (например, к `InstrumentRepository`), и нужно добавить сквозное поведение
вокруг каждого вызова — замер времени, логирование, — не изменяя код целевого класса. Прямое встраивание такой логики
в каждый метод нарушает сразу SRP (метод отвечает и за свою задачу, и за диагностику) и DRY (пара «замерить →
залогировать» дублируется в каждом методе).

**Proxy** (структурный) — объект-заместитель, реализующий тот же контракт, что и целевой объект, и перехватывающий
обращения к нему. Клиентский код не отличает прокси от настоящей реализации. В Java — три способа получить такой
заместитель, по нарастанию автоматизации.

---

## Три реализации

**Самописный (статический) прокси** — класс пишется вручную, реализует тот же интерфейс, что и целевой объект, и
оборачивает каждый вызов. Прозрачен и не требует библиотек, но дублирование никуда не девается — оно просто переезжает
в прокси: каждый новый метод интерфейса приходится дописывать руками.

**JDK Dynamic Proxy** (`java.lang.reflect.Proxy`) — генерирует класс-заместитель во время выполнения на основе
интерфейса; один `InvocationHandler` перехватывает вызовы любого метода. Не требует сторонних зависимостей, но работает
только с интерфейсами — класс без интерфейса им не проксировать.

**CGLIB** — генерирует подкласс целевого класса во время выполнения, интерфейс не обязателен. Снимает ограничение JDK
Dynamic Proxy, но не может проксировать `final`-классы и `final`-методы, и требует дополнительной зависимости.

---

## Сравнение подходов

| Критерий                       | Самописный прокси | JDK Dynamic Proxy     | CGLIB                          |
|--------------------------------|-------------------|-----------------------|--------------------------------|
| Требует интерфейс              | Да                | Да                    | Нет                            |
| Работает с `final`             | — (не применимо)  | — (не применимо)      | Нет                            |
| Дополнительная зависимость     | Нет               | Нет (входит в JDK)    | Да                             |
| Новые методы без правки прокси | Нет — вручную     | Да — автоматически    | Да — автоматически             |
| Скорость вызова                | Без косвенности   | Медленнее (рефлексия) | Быстрее JDK, медленнее прямого |

---

## В промышленных фреймворках

Оба динамических механизма — основа Dependency Injection в Spring: если бин реализует интерфейс, создаётся JDK Dynamic
Proxy, иначе — CGLIB. Тем же способом реализованы декларативные транзакции (`@Transactional`), кэширование
(`@Cacheable`) и Spring AOP в целом.
