### Этап 1. Мотивация: боль от Strategy на классах (~5 мин)

Проект в состоянии после семинара 2: `InstrumentFilter`, `TypeFilter`, `NoOpFilter`. Вопрос группе: что произойдёт с
этим набором классов при добавлении фильтра по валюте, а затем — комбинации «тип и валюта»? Быстро приходим к выводу:
количество классов растёт линейно с числом условий, а их комбинации — комбинаторно. Идея Strategy правильная, механизм
реализации — нет.

### Этап 2. Теория: функциональные интерфейсы и лямбда-выражения (~10 мин)

Презентация по материалу
[«Функциональные интерфейсы и лямбда-выражения»](./Функциональные%20интерфейсы%20и%20лямбда-выражения.md):
функциональный интерфейс, лямбда, сопоставление с анонимным классом `Iterator` из семинара 2, захват переменных,
`java.util.function`. Разбор захвата и байт-кода — мостик к этапу 4.

**Вопрос группе:** можно ли `Iterator` из `InMemoryInstrumentRepository` переписать лямбдой? Нет — `Iterator<T>`
объявляет два абстрактных метода (`hasNext()` и `next()`), это не функциональный интерфейс. Лямбда годится только там,
где абстрактный метод ровно один.

### Этап 3. Практика: Strategy → лямбды, композиция, Comparator (~15 мин)

Заменяем `InstrumentFilter`/`TypeFilter`/`NoOpFilter` на `Predicate<Instrument>`, удаляя все три класса:

```java
@Override
public Integer call() {
    Predicate<Instrument> filter = (type == null)
            ? instrument -> true
            : instrument -> instrument.getClass().getSimpleName().equalsIgnoreCase(type);

    if (currency != null) {
        filter = filter.and(instrument -> instrument.getCurrency() == currency);
    }

    List<Instrument> matched = new ArrayList<>();
    for (Instrument instrument : repository) {
        if (filter.test(instrument)) {
            matched.add(instrument);
        }
    }

    if (sortByTicker) {
        matched.sort(Comparator.comparing(Instrument::getTicker));
    }

    matched.forEach(instrument -> System.out.println(instrument.getDescription()));
    return 0;
}
```

`filter.and(...)` — `default`-метод `Predicate`, сам принимающий и возвращающий `Predicate`: композиция функций как
аргументов. `Comparator.comparing(Instrument::getTicker)` — метод-ссылка вместо `instrument -> instrument.getTicker()`,
применима, когда тело лямбды сводится к вызову одного существующего метода.

### Этап 4. Байткод: лямбда vs анонимный класс (~20 мин)

Компилируем текущую версию проекта и смотрим на результат через `javap` — разбор вывода и объяснение механизма
(`invokedynamic`, `LambdaMetafactory`, скрытые классы) даны в материале
[«Функциональные интерфейсы и лямбда-выражения»](./Функциональные%20интерфейсы%20и%20лямбда-выражения.md), здесь —
только команды и ключевой вывод:

```bash
javac -d out $(find src/main/java -name "*.java")
javap -c -p out/.../ListCommand.class
```

Для `InMemoryInstrumentRepository` в каталоге `out` появляется файл `InMemoryInstrumentRepository$1.class` —
анонимный класс существует физически, как байт-код на диске, до запуска программы. Для лямбды в `ListCommand.class`
отдельного класса нет — вместо этого инструкция `invokedynamic`, генерирующая реализацию `Predicate` во время
выполнения при первом обращении.

> **Точные номера констант и офсеты в вашей сборке будут отличаться — важна структура, а не конкретные цифры.**

*Подробный разбор `LambdaMetafactory`, скрытых классов и стоимости первого вызова — в сопроводительном материале
«Лямбда-выражения: устройство».*

### Этап 6. Практика: `Registry<K, V>`, bounded generics, `? super T` (~15 мин)

Обобщаем `InstrumentFactories`:

```java
public class Registry<K, V> {
    private final Map<K, V> items = new HashMap<>();

    public void register(K key, V value) {
        items.put(key, value);
    }

    public V get(K key) {
        return items.get(key);
    }
}
```

```java
public final class InstrumentFactories {
    private static final Registry<String, InstrumentFactory> REGISTRY = loadRegistry();

    public static Instrument create(String type, String ticker, String name, Currency currency) {
        InstrumentFactory factory = REGISTRY.get(type.toUpperCase());
        if (factory == null) {
            throw new IllegalArgumentException("Unknown instrument type: " + type);
        }
        return factory.create(ticker, name, currency);
    }

    private static Registry<String, InstrumentFactory> loadRegistry() {
        Registry<String, InstrumentFactory> registry = new Registry<>();
        for (InstrumentFactory factory : ServiceLoader.load(InstrumentFactory.class)) {
            registry.register(factory.getSupportedType().toUpperCase(), factory);
        }
        return registry;
    }
}
```

`Map` в `loadRegistry()` (семинар 2) заменяется на обобщённый `Registry<K, V>` — сам механизм обнаружения фабрик через
`ServiceLoader` не меняется, обобщается только структура хранения «ключ → значение».

Добавляем обобщённый метод `topByTicker` (этап 5) и универсальный `sortWith` с `Comparator<? super T>` —
демонстрируем на конкретном вызове с `List<Stock>` и компаратором `Comparator<Instrument>`, чтобы показать разницу
между `sortWith(list, Comparator<T>)`, которая такой вызов бы не приняла, и версией с `? super T`.

### Этап 7. Optional: прощаемся с `null` (~10 мин)

Презентация по материалу [«Optional»](./Optional.md): проблема, которую решает `Optional`, `orElse` vs `orElseGet`, где
`Optional` неуместен.

`findByTicker` и `Registry.get()` возвращают `null` при отсутствии значения — переводим оба на `Optional<T>`:

```java
public Optional<Instrument> findByTicker(String ticker) {
    for (Instrument instrument : this) {
        if (instrument.getTicker().equalsIgnoreCase(ticker)) {
            return Optional.of(instrument);
        }
    }
    return Optional.empty();
}
```

```java
@Override
public Integer call() {
    return repository.findByTicker(ticker)
            .map(instrument -> {
                System.out.println(instrument.getDescription());
                return 0;
            })
            .orElseGet(() -> {
                System.out.println("Инструмент не найден: " + ticker);
                return 1;
            });
}
```

`Registry.get()` меняется аналогично:

```java
public Optional<V> get(K key) {
    return Optional.ofNullable(items.get(key));
}
```

`InstrumentFactories.create()` из этапа 6 обращался к `REGISTRY.get(...)` через проверку на `null` — после перехода
`Registry` на `Optional` этот код перестанет компилироваться, если его не обновить вместе с `Registry`:

```java
public static Instrument create(String type, String ticker, String name, Currency currency) {
    return REGISTRY.get(type.toUpperCase())
            .map(factory -> factory.create(ticker, name, currency))
            .orElseThrow(() -> new IllegalArgumentException("Unknown instrument type: " + type));
}
```

### Этап 8. Vavr: Try и Either (~15 мин)

Презентация по материалу [«Vavr: Try и Either»](./Vavr%20Try%20и%20Either.md): подключение зависимости, `Try<T>` как
альтернатива `try/catch`, `Either<L, R>` для явного разделения «ошибка / успех» в сигнатуре метода.

> **Важно:** сегодня — только знакомство. `Try`/`Either` не внедряются повсеместно в проект — полноценная стратегия
> обработки ошибок и исключений станет темой семинара 4, и там же обсудим, где в проекте уместны исключения, а где —
> типы вроде `Either`.
