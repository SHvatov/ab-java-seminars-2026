### Этап 2. Теория: функциональные интерфейсы и лямбда-выражения (~10 мин)

Функциональный интерфейс — интерфейс с одним абстрактным методом (SAM). `InstrumentFilter` из семинара 2 уже им
является:

```java
public interface InstrumentFilter {
    boolean matches(Instrument instrument);
}
```

Лямбда — реализация функционального интерфейса без объявления класса:

```java
InstrumentFilter filter = instrument -> instrument.getClass().getSimpleName().equalsIgnoreCase("STOCK");
```

Сразу сопоставляем с тем, что аудитория уже писала на семинаре 2 — анонимный класс `Iterator` в
`InMemoryInstrumentRepository`:

```java
@Override
public Iterator<Instrument> iterator() {
    return new Iterator<>() {
        private int cursor = 0;

        @Override
        public boolean hasNext() {
            return cursor < size;
        }

        @Override
        public Instrument next() {
            if (!hasNext()) {
                throw new NoSuchElementException();
            }
            return instruments[cursor++];
        }
    };
}
```

**Вопрос группе:** можно ли `Iterator` переписать лямбдой? Нет — `Iterator<T>` объявляет два абстрактных метода
(`hasNext()` и `next()`), это не функциональный интерфейс. Лямбда годится только там, где абстрактный метод ровно один.
Разбор этого различия — мостик к этапу 4.

**Захват переменных.** Только *effectively final* — переменная, захваченная лямбдой, не может быть переприсвоена после
инициализации:

```java
void example() {
    String type = "STOCK";
    InstrumentFilter filter = instrument -> instrument.getClass().getSimpleName().equalsIgnoreCase(type);
    type = "BOND"; // ошибка компиляции
}
```

Причина — не техническое ограничение синтаксиса, а следствие того, как именно лямбда реализована на уровне JVM.
Подробности — на этапе 4.

**`java.util.function`:** `Predicate<T>` (`boolean test(T)`), `Function<T, R>` (`R apply(T)`), `Supplier<T>`
(`T get()`), `Consumer<T>` (`void accept(T)`) — структурно совпадают со своими доменными аналогами вроде
`InstrumentFilter`, поэтому в большинстве случаев не нужно объявлять собственный функциональный интерфейс.

### Этап 3. Практика: Strategy → лямбды, композиция, Comparator (~15 мин)

Заменяем `InstrumentFilter`/`TypeFilter`/`NoOpFilter` на `Predicate<Instrument>`, удаляя все три класса:

```java
@Override
public Integer call() {
    Predicate<Instrument> filter = (type == null)
            ? instrument -> true
            : instrument -> instrument.getClass().getSimpleName().equalsIgnoreCase(type);

    if (currency != null) {
        filter = filter.and(instrument -> instrument.getCurrency() == currency);
    }

    List<Instrument> matched = new ArrayList<>();
    for (Instrument instrument : repository) {
        if (filter.test(instrument)) {
            matched.add(instrument);
        }
    }

    if (sortByTicker) {
        matched.sort(Comparator.comparing(Instrument::getTicker));
    }

    matched.forEach(instrument -> System.out.println(instrument.getDescription()));
    return 0;
}
```

`filter.and(...)` — `default`-метод `Predicate`, сам принимающий и возвращающий `Predicate`: композиция функций как
аргументов. `Comparator.comparing(Instrument::getTicker)` — метод-ссылка вместо `instrument -> instrument.getTicker()`,
применима, когда тело лямбды сводится к вызову одного существующего метода.

### Этап 4. Байткод: лямбда vs анонимный класс (~20 мин)

Компилируем текущую версию проекта и смотрим на результат через `javap`:

```bash
javac -d out $(find src/main/java -name "*.java")
javap -c -p out/.../ListCommand.class
```

**Анонимный класс компилируется в отдельный `.class`-файл.** Для `InMemoryInstrumentRepository` из семинара 2 в
каталоге `out` появляется файл `InMemoryInstrumentRepository$1.class` — компилятор сгенерировал именованный вложенный
класс, реализующий `Iterator<Instrument>`, ещё на этапе компиляции. Этот класс существует физически, как байт-код на
диске, до запуска программы.

**У лямбды нет `.class`-файла на этапе компиляции.** В `ListCommand.class` вместо ссылки на отдельный класс встретится
инструкция `invokedynamic`:

```
0: invokedynamic #7,  0             // InvokeDynamic #0:test:(Ljava/lang/String;)Ljava/util/function/Predicate;
     ...
private static boolean lambda$call$0(java.lang.String, Instrument);
    Code:
       0: aload_1
       1: invokevirtual #12  // Method Instrument.getClass:()Ljava/lang/Class;
       4: invokevirtual #16  // Method java/lang/Class.getSimpleName:()Ljava/lang/String;
       7: aload_0
       8: invokevirtual #20  // Method java/lang/String.equalsIgnoreCase:(Ljava/lang/String;)Z
      11: ireturn
```

> **Точные номера констант и офсеты в вашей сборке будут отличаться — важна структура, а не конкретные цифры.**

Что здесь произошло: тело лямбды компилятор превратил в обычный приватный статический метод самого класса
`ListCommand` (`lambda$call$0`) — никакого нового класса на диске для этого не создано. А байт-код, где лямбда
*используется*, содержит `invokedynamic` — инструкцию, которая при **первом** выполнении вызывает bootstrap-метод
`java.lang.invoke.LambdaMetafactory.metafactory`. Этот метод генерирует реализацию `Predicate` **во время выполнения**,
используя скрытый класс (`hidden class`, механизм `MethodHandle`), и связывает `invokedynamic` с этой реализацией — при
повторных вызовах генерация уже не происходит, связывание кэшируется.

**Следствие для effectively final.** Захваченные переменные передаются в синтетический метод `lambda$call$0` как
обычные аргументы — по значению, в момент вызова. Если бы исходная переменная могла измениться после создания лямбды,
метод `lambda$call$0` в принципе не смог бы понять, какое значение подставлять: он получает копию на момент вызова
`invokedynamic`, а не связь с переменной. Ограничение "effectively final" — не искусственное синтаксическое правило, а
прямое следствие того, как это в принципе устроено.

*Подробный разбор `LambdaMetafactory` и сравнение с анонимными классами — в сопроводительном материале
«Лямбда-выражения: устройство».*

### Этап 5. Теория: Generics, type erasure, wildcard (~15 мин)

Generics — параметрический полиморфизм, третий вид после подтипового (семинар 1) и перегрузки. Уже использовали
неявно: `Iterator<Instrument>`, `Predicate<Instrument>`, `Comparator<Instrument>`.

**Type erasure.** В отличие от, например, C++ шаблонов, параметры типов в Java существуют только на этапе компиляции и
«стираются» в байт-коде: `List<String>` и `List<Integer>` в скомпилированном классе — это один и тот же `List`,
разница видна только компилятору при проверке типов. Отсюда прямое следствие — невозможно создать `new T()` или
`new T[]` внутри generic-класса, и невозможна перегрузка методов, отличающихся только параметром типа
(`void process(List<String>)` и `void process(List<Integer>)` в одном классе — ошибка компиляции, оба метода после
стирания типов имеют одинаковую сигнатуру `process(List)`).

**Bounded type parameter** — `<T extends Instrument>` ограничивает `T` подтипами `Instrument`, давая доступ к его
методам внутри generic-кода:

```java
public static <T extends Instrument> List<T> topByTicker(List<T> items, int n) {
    List<T> sorted = new ArrayList<>(items);
    sorted.sort(Comparator.comparing(Instrument::getTicker));
    return sorted.subList(0, Math.min(n, sorted.size()));
}
```

**Wildcard и `super` — правило PECS.** *Producer Extends, Consumer Super*: если структура данных только **отдаёт**
значения типа `T` наружу — используем `? extends T`; если только **принимает** значения `T` — используем `? super T`.

Практический пример — сортировка обобщённым `Comparator`, который «потребляет» элементы для сравнения:

```java
public static <T> void sortWith(List<T> items, Comparator<? super T> comparator) {
    items.sort(comparator);
}
```

`Comparator<? super T>` позволяет передать не только `Comparator<T>`, но и, например, `Comparator<Instrument>` для
вызова `sortWith(stocks, comparator)`, где `stocks` — это `List<Stock>`: компаратор, умеющий сравнивать более общий тип
`Instrument`, годится и для его подтипа `Stock`, потому что он «потребляет» `Stock` как частный случай `Instrument`.

*Подробный разбор type erasure, мостовых методов (bridge methods) и wildcard-типов — в сопроводительном материале
«Generics: устройство».*

### Этап 6. Практика: `Registry<K, V>`, bounded generics, `? super T` (~15 мин)

Обобщаем `InstrumentFactories`:

```java
public class Registry<K, V> {
    private final Map<K, V> items = new HashMap<>();

    public void register(K key, V value) {
        items.put(key, value);
    }

    public V get(K key) {
        return items.get(key);
    }
}
```

```java
public final class InstrumentFactories {
    private static final Registry<String, InstrumentFactory> REGISTRY = new Registry<>();

    public static void register(String type, InstrumentFactory factory) {
        REGISTRY.register(type.toUpperCase(), factory);
    }

    public static Instrument create(String type, String ticker, String name, Currency currency) {
        InstrumentFactory factory = REGISTRY.get(type.toUpperCase());
        if (factory == null) {
            throw new IllegalArgumentException("Unknown instrument type: " + type);
        }
        return factory.create(ticker, name, currency);
    }
}
```

Добавляем обобщённый метод `topByTicker` (этап 5) и универсальный `sortWith` с `Comparator<? super T>` —
демонстрируем на конкретном вызове с `List<Stock>` и компаратором `Comparator<Instrument>`, чтобы показать разницу
между `sortWith(list, Comparator<T>)`, которая такой вызов бы не приняла, и версией с `? super T`.

### Этап 7. Optional: прощаемся с `null` (~10 мин)

`findByTicker` и `Registry.get()` возвращают `null` при отсутствии значения — переводим оба на `Optional<T>`:

```java
public Optional<Instrument> findByTicker(String ticker) {
    for (Instrument instrument : this) {
        if (instrument.getTicker().equalsIgnoreCase(ticker)) {
            return Optional.of(instrument);
        }
    }
    return Optional.empty();
}
```

```java
@Override
public Integer call() {
    return repository.findByTicker(ticker)
            .map(instrument -> {
                System.out.println(instrument.getDescription());
                return 0;
            })
            .orElseGet(() -> {
                System.out.println("Инструмент не найден: " + ticker);
                return 1;
            });
}
```

`orElse(default)` вычисляет `default` всегда, даже когда значение присутствует — для дорогих вычислений или побочных
эффектов нужен именно `orElseGet(Supplier)`, вычисляющий значение лениво, только при пустом `Optional`. Это тот случай,
где знание про `Supplier<T>` с этапа 2 напрямую объясняет, почему у `Optional` есть два похожих метода, а не один.

### Этап 8. Vavr: Try и Either (~15 мин)

[Vavr](https://vavr.io/) — библиотека функциональных типов для Java: неизменяемые коллекции, `Option`, `Try`, `Either`
и другие конструкции, отсутствующие в стандартной библиотеке.

```xml
<dependency>
    <groupId>io.vavr</groupId>
    <artifactId>vavr</artifactId>
    <version>1.0.1</version>
</dependency>
```

**`Try<T>`** оборачивает вычисление, способное бросить исключение, в значение — `Success(T)` либо `Failure(Throwable)`,
без `try/catch` в месте использования:

```java
void example() {
    Try<Currency> currency = Try.of(() -> Currency.valueOf(currencyString));

    currency.onSuccess(value -> System.out.println("Валюта распознана: " + value))
            .onFailure(error -> System.out.println("Некорректная валюта: " + currencyString));
}
```

**`Either<L, R>`** представляет одно из двух значений — по соглашению `Left` для ошибки, `Right` для успеха. В отличие
от исключения, тип возможной ошибки виден прямо в сигнатуре метода:

```java
public Either<String, Instrument> parseInstrument(String type, String ticker, String name, String currencyStr) {
    try {
        Currency currency = Currency.valueOf(currencyStr.toUpperCase());
        return Either.right(InstrumentFactories.create(type, ticker, name, currency));
    } catch (IllegalArgumentException e) {
        return Either.left("Некорректный тип или валюта: " + e.getMessage());
    }
}
```

```java
parseInstrument(type, ticker, name, currencyStr)
        .peek(instrument -> repository.add(instrument))
        .peekLeft(error -> System.out.println("Ошибка: " + error));
```

> **Важно:** сегодня — только знакомство. `Try`/`Either` не внедряются повсеместно в проект — полноценная стратегия
> обработки ошибок и исключений станет темой семинара 4, и там же обсудим, где в проекте уместны исключения, а где —
> типы вроде `Either`.
