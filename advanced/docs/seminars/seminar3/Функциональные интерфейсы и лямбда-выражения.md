# Функциональные интерфейсы и лямбда-выражения

---

## Функциональный интерфейс

Интерфейс с одним абстрактным методом (SAM — *single abstract method*). `InstrumentFilter` из семинара 2 уже им
является:

```java
public interface InstrumentFilter {
    boolean matches(Instrument instrument);
}
```

Лямбда — реализация функционального интерфейса без объявления класса:

```java
InstrumentFilter filter = instrument -> instrument.getClass().getSimpleName().equalsIgnoreCase("STOCK");
```

Сопоставление с тем, что уже писали на семинаре 2, — анонимный класс `Iterator` в `InMemoryInstrumentRepository`:

```java
@Override
public Iterator<Instrument> iterator() {
    return new Iterator<>() {
        private int cursor = 0;

        @Override
        public boolean hasNext() {
            return cursor < size;
        }

        @Override
        public Instrument next() {
            if (!hasNext()) {
                throw new NoSuchElementException();
            }
            return instruments[cursor++];
        }
    };
}
```

`Iterator<T>` объявляет два абстрактных метода (`hasNext()` и `next()`) — это не функциональный интерфейс, лямбдой его
не переписать. Лямбда годится только там, где абстрактный метод ровно один.

**`java.util.function`:** `Predicate<T>` (`boolean test(T)`), `Function<T, R>` (`R apply(T)`), `Supplier<T>`
(`T get()`), `Consumer<T>` (`void accept(T)`) — структурно совпадают со своими доменными аналогами вроде
`InstrumentFilter`, поэтому в большинстве случаев не нужно объявлять собственный функциональный интерфейс.

## Захват переменных

Только *effectively final* — переменная, захваченная лямбдой, не может быть переприсвоена после инициализации:

```java
void example() {
    String type = "STOCK";
    InstrumentFilter filter = instrument -> instrument.getClass().getSimpleName().equalsIgnoreCase(type);
    type = "BOND"; // ошибка компиляции
}
```

Причина — не техническое ограничение синтаксиса, а следствие того, как именно лямбда реализована на уровне JVM (см.
ниже).

## Байткод: лямбда vs анонимный класс

**Анонимный класс компилируется в отдельный `.class`-файл.** Для `InMemoryInstrumentRepository` из семинара 2 в
каталоге сборки появляется файл `InMemoryInstrumentRepository$1.class` — компилятор сгенерировал именованный вложенный
класс, реализующий `Iterator<Instrument>`, ещё на этапе компиляции. Этот класс существует физически, как байт-код на
диске, до запуска программы.

**У лямбды нет `.class`-файла на этапе компиляции.** В байт-коде класса, где лямбда написана, вместо ссылки на
отдельный класс встретится инструкция `invokedynamic`:

```
0: invokedynamic #7,  0             // InvokeDynamic #0:test:(Ljava/lang/String;)Ljava/util/function/Predicate;
     ...
private static boolean lambda$call$0(java.lang.String, Instrument);
    Code:
       0: aload_1
       1: invokevirtual #12  // Method Instrument.getClass:()Ljava/lang/Class;
       4: invokevirtual #16  // Method java/lang/Class.getSimpleName:()Ljava/lang/String;
       7: aload_0
       8: invokevirtual #20  // Method java/lang/String.equalsIgnoreCase:(Ljava/lang/String;)Z
      11: ireturn
```

Тело лямбды компилятор превращает в обычный приватный статический метод самого класса (`lambda$call$0`) — никакого
нового класса на диске для этого не создаётся. Байт-код в месте, где лямбда *используется*, содержит `invokedynamic`
— инструкцию, которая при **первом** выполнении вызывает bootstrap-метод
`java.lang.invoke.LambdaMetafactory.metafactory`. Этот метод генерирует реализацию функционального интерфейса **во
время выполнения**, используя скрытый класс (`hidden class`, механизм `MethodHandle`), и связывает `invokedynamic` с
этой реализацией — при повторных вызовах генерация уже не происходит, связывание кэшируется.

**Следствие для effectively final.** Захваченные переменные передаются в синтетический метод `lambda$call$0` как
обычные аргументы — по значению, в момент вызова. Если бы исходная переменная могла измениться после создания лямбды,
метод `lambda$call$0` в принципе не смог бы понять, какое значение подставлять: он получает копию на момент вызова
`invokedynamic`, а не связь с переменной. Ограничение effectively final — не искусственное синтаксическое правило, а
прямое следствие того, как это устроено.
