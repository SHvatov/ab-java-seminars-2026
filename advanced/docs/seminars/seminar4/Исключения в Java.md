# Исключения в Java

---

## Коды возврата и sentinel values

Классический до-исключений подход: функция возвращает не результат, а признак ошибки — `-1`, `null`, глобальную
переменную вроде `errno` в C. Явный и дешёвый, но проверку результата ничто не гарантирует — компилятор не заставит,
и при цепочке вызовов легко забыть проверку на одном из шагов. `findByTicker`, возвращавший `null` до семинара 3, —
тот же паттерн, только в объектно-ориентированной обёртке.

## Исключения

`throw` прерывает нормальное выполнение и передаёт объект `Throwable` вверх по стеку вызовов, минуя все промежуточные
фреймы без подходящего `catch`, пока не найдётся обработчик — или пока исключение не долетит до самого верха и не
завершит поток. Ключевое отличие от кода возврата: исключение **нельзя случайно проигнорировать**.

---

## Иерархия Throwable

```
Throwable
 ├── Error                     — проблемы уровня JVM, обычно не перехватываются приложением
 └── Exception
      ├── RuntimeException     — unchecked
      └── остальные Exception  — checked: IOException и подобные
```

## Спор о checked-исключениях

Замысел — заставить разработчика осознанно обработать восстановимую ошибку. **Плюс:** checked-исключение
документирует прямо в сигнатуре метода, какие ошибки от него ожидать. **Минус:** плохо масштабируется по глубине
вызовов — если обработать ошибку осмысленно может только код на первом уровне, все промежуточные методы вынуждены
либо ловить и немедленно перебрасывать, либо просто транслировать `throws` дальше, не имея возможности с ошибкой
ничего сделать. Отсюда повсеместный антипаттерн `catch (Exception e) { throw new RuntimeException(e); }`.

**Ловушка Spring `@Transactional`.** По умолчанию транзакция откатывается только при unchecked-исключении. Если метод
бросает checked-исключение — даже объявленное в `throws` — транзакция по умолчанию **закоммитится**, если явно не
указать `@Transactional(rollbackFor = MyCheckedException.class)`. Ошибка, формально «правильно обработанная», тихо
проходит мимо механизма отката.

**Конфликт с функциональными интерфейсами.** Ни один интерфейс `java.util.function` не объявляет `throws`, поэтому
checked-исключение внутри лямбды, нацеленной на такой интерфейс, не компилируется:

```java
Function<String, byte[]> reader = path -> Files.readAllBytes(Path.of(path)); // ошибка компиляции: IOException checked
```

Единственный выход — обернуть в unchecked внутри лямбды:

```java
Function<String, byte[]> reader = path -> {
    try {
        return Files.readAllBytes(Path.of(path));
    } catch (IOException e) {
        throw new UncheckedIOException(e);
    }
};
```

Ни C#, ни Kotlin не реализуют checked-исключения вовсе; многие Java-библиотеки сознательно оборачивают checked
нижних слоёв в unchecked на границе своего API (Spring Data → `DataAccessException`, Jackson →
`JsonProcessingException`-наследники) — не отказ от идеи, а перенос решения с компилятора на разработчика библиотеки.

## Проектирование собственных исключений

Заводить свой тип оправдано, когда вызывающему коду важно программно отличить эту ошибку от прочих или нужен
структурированный контекст. В подавляющем большинстве случаев — наследоваться от `RuntimeException`.

```java
public class DuplicateTickerException extends RuntimeException {

    private final String ticker;

    public DuplicateTickerException(String ticker) {
        super("Инструмент с тикером уже существует: " + ticker);
        this.ticker = ticker;
    }

    public String getTicker() {
        return ticker;
    }
}
```

**Не терять причину.** Оборачивая исключение, обязательно передавать исходное как `cause` — `catch (IOException e) {
throw new RuntimeException("failed"); }` без передачи `e` уничтожает информацию о первопричине.

## try-with-resources и подавленные исключения

`try-with-resources` автоматически вызывает `close()` любого `AutoCloseable` при выходе из блока — в том числе при
исключении в теле. Если исключение бросают и тело, и `close()`, наружу вылетает исключение из тела (оно основное), а
исключение из `close()` прикрепляется к нему как *подавленное* и доступно через `getSuppressed()`, а не теряется, как
это было бы при ручном `try/finally`.

## Стоимость исключения

**Выделение объекта** — обычное, как любой `new`. **`fillInStackTrace()`** — главная статья расходов: обходит весь
текущий стек вызовов и материализует его в массив `StackTraceElement`; чем глубже стек, тем дороже. **Разворачивание
стека при обработке** — доставка исключения от `throw` до `catch`, дешевле `fillInStackTrace()`, но не бесплатно.

Отключить самую дорогую часть можно через четырёхаргументный конструктор:

```java
protected Throwable(String message, Throwable cause, boolean enableSuppression, boolean writableStackTrace)
```

`writableStackTrace = false` — осознанный компромисс, уместный только в горячих путях, где место возникновения
ошибки не нужно для диагностики. Практический замер — в материале
[«JMH и замер производительности»](./JMH%20и%20замер%20производительности.md) и на самом семинаре, этап 5.

---

## Когда исключения — не тот инструмент

Не для управления обычным потоком выполнения — для «элемент не найден» подходит `Optional`. Не для валидации на
каждой итерации горячего цикла — здесь работает та же логика, что и со стоимостью исключений. Не как замена
документации — сообщение вроде `"Error"` не помогает разработчику, читающему логи через полгода.
