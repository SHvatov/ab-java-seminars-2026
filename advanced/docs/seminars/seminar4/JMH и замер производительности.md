# JMH и замер производительности

---

## Почему нельзя просто засечь время

```java
long start = System.nanoTime();
doSomething();
long elapsed = System.nanoTime() - start;
```

Для кода, который выполняется миллионы раз, такой замер систематически вводит в заблуждение: **JIT-разогрев**
(первые вызовы метода исполняются интерпретатором и облегчённым компилятором, только «горячие» методы
JIT-компилируются агрессивно) искажает замер, сделанный сразу после старта; **удаление мёртвого кода** — если
результат вычисления нигде не используется, компилятор вправе не выполнять его вовсе; **свёртка констант** — если
входные данные известны на этапе компиляции, весь расчёт может быть заранее вычислен один раз; сборка мусора и другие
процессы добавляют шум, который простое усреднение не гасит.

---

## Что такое JMH

[JMH](https://openjdk.org/projects/code-tools/jmh/) (Java Microbenchmark Harness) — официальный инструмент для
микробенчмарков на JVM от команды OpenJDK/HotSpot, систематически нейтрализующий перечисленные искажения:
прогревом перед измерением, изоляцией через отдельные процессы JVM, принудительным использованием результата
вычисления.

```xml
<dependency>
    <groupId>org.openjdk.jmh</groupId>
    <artifactId>jmh-core</artifactId>
    <version>1.37</version>
</dependency>
<dependency>
    <groupId>org.openjdk.jmh</groupId>
    <artifactId>jmh-generator-annprocess</artifactId>
    <version>1.37</version>
    <scope>provided</scope>
</dependency>
```

`jmh-generator-annprocess` — annotation processor, генерирующий обвязку бенчмарка (прогрев, циклы измерения) из
аннотированных классов на этапе компиляции.

---

## Структура бенчмарка

```java
@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.NANOSECONDS)
@State(Scope.Thread)
@Warmup(iterations = 5, time = 1)
@Measurement(iterations = 5, time = 1)
@Fork(1)
public class SimpleBenchmark {

    @Benchmark
    public int addition() {
        return 2 + 2;
    }
}
```

`@State(Scope.Thread)` — область видимости состояния (`Thread`/`Benchmark`/`Group`); `@BenchmarkMode` — что
измеряется (`AverageTime`, `Throughput`, `SampleTime`, `SingleShotTime`); `@Warmup`/`@Measurement` — число и
длительность итераций прогрева и измерения; `@Fork` — сколько раз JMH перезапустит бенчмарк в отдельном процессе JVM,
защита от «повезло/не повезло» с решениями JIT в конкретном запуске.

## Ловушки и как их избежать

- **Мёртвый код** — возвращать результат из метода бенчмарка либо передавать в `Blackhole.consume(...)`, который
  гарантированно не даст оптимизатору выбросить значение как неиспользуемое.
- **Свёртка констант** — входные данные объявлять как поля `@State`-класса, а не как локальные литералы.
- **Недостаточный прогрев** — особенно критично при сравнении «до/после»: недопрогретые варианты дают разницу,
  которая может быть артефактом порядка JIT-компиляции, а не реальной разницей в производительности.
- **Один форк** — для чернового исследования достаточно, для выводов, на которых строятся архитектурные решения,
  разумны 3–5 форков.

## Как читать результаты

```
Benchmark                                    Mode  Cnt    Score    Error  Units
ExceptionOverheadBenchmark.returnCode        avgt   25    2.104 ±  0.031  ns/op
ExceptionOverheadBenchmark.throwWithoutTrace avgt   25   18.532 ±  0.842  ns/op
ExceptionOverheadBenchmark.throwWithTrace    avgt   25  412.671 ± 15.203  ns/op
```

`Score` — итоговая оценка; `Error` — доверительный интервал (99,9% по умолчанию): если интервалы двух бенчмарков не
пересекаются, разница статистически значима, а не шум замера; `Cnt` — число измерений (итерации × форки).

---

## Микробенчмарк vs профилировщик

JMH отвечает на узкий вопрос: какой из уже сформулированных вариантов быстрее. Профилировщик (Java Flight Recorder,
async-profiler) отвечает на другой: где в реальном приложении под нагрузкой вообще тратится время, не требуя заранее
знать, что сравнивать. Разумный порядок: сначала профилировщиком найти, что стоит оптимизировать, затем — если
решение неочевидно — сузить вопрос до конкретной альтернативы и проверить через JMH.
